# TETO 1.6 地基补全 — 最终实施计划

## Context

用户在规划阶段提出核心比喻，明确了 1.6 的目标架构：

> 旧系统：每个门都能进货，每个员工都能自己记账，每个部门都自己算报表，AI 也能偷偷改账本。
> 新系统：所有数据进来先过总检查站，所有修改数据也先过总检查站，AI 想改数据也要先过总检查站，所有统计报表都去统一算法中心算。

对应两层架构：
- **规则中心（治理层）**：统一写入网关。所有数据变更（用户输入、AI 修改、批量导入）都经过同一套规则校验
- **计算中心（计算层）**：统一读取入口。所有统计查询走同一个地方、同一套口径

## 现状

**已完成 — Record 域 (P1-P3)：**
- `record-service.ts`：8 个 Safely 函数
- `record-invariants.ts`：12 条纯逻辑规则

- `record-lifecycle-invariants.ts`：5 条生命周期规则
- `record-ai-service.ts` + AI 策略层：AI 写入通过规则中心

**已完成 — 基础设施：**

- `src/lib/computation/index.ts`：COMPUTATION 声明中心

- `src/lib/observability/`：Logger/Trace/ID Registry/Error Codes

## Batch 1：规则中心扩展 — 6 个域的 Domain Service

> 目标：Item/Goal/Phase/Tag/SubItem 拥有与 Record 相同的治理能力

## Batch 2：规则中心接入 — API 路由改调 Safely

> 目标：所有写入 API 通过域服务写入，实现"所有数据进来先过总检查站"

### 修改路由（16 个）

| 路由 | 改动 |
|------|------|
| `POST /api/v2/items` | → `createItemSafely` |
| `PATCH /api/v2/items/[id]` | → `updateItemSafely` |
| `DELETE /api/v2/items/[id]` | → `archiveItemSafely` |
| `POST /api/v2/goals` | → `createGoalSafely` |
| `PATCH /api/v2/goals/[id]` | → `updateGoalSafely` |
| `POST /api/v2/goals/[id]/confirm` | → `confirmGoalSafely` |
| `POST /api/v2/phases` | → `createPhaseSafely` |
| `PATCH /api/v2/phases/[id]` | → `updatePhaseSafely` |
| `DELETE /api/v2/phases/[id]` | → `deletePhaseSafely` |
| `POST /api/v2/tags` | → `createTagSafely` |
| `PATCH /api/v2/tags/[id]` | → `updateTagSafely` |
| `DELETE /api/v2/tags/[id]` | → `deleteTagSafely` |
| `POST /api/v2/sub-items` | → `createSubItemSafely` |
| `PATCH /api/v2/sub-items/[id]` | → `updateSubItemSafely` |
| `DELETE /api/v2/sub-items/[id]` | → `deleteSubItemSafely` |
| `POST /api/v2/sub-items/[id]/promote` | → `promoteSubItemSafely` |

### 改动模式

```typescript
// 改动前：
const item = await createItem(userId, body);
return apiSuccess(item, ctx.traceId, 201);

// 改动后：
const result = await createItemSafely({ userId, payload: body, supabase });
if (!result.ok) return apiDomainError(result.errors, ctx.traceId);
return apiSuccess(result.data, ctx.traceId, 201, result.warnings);
```

### 验证
- `npm run build` 通过
- 所有 POST/PATCH/DELETE 路由通过 Safely 写入
- 响应格式保持兼容（data + 可选 warnings）

## Batch 5：可观测性补全 — Span + Decision ID

> 目标：关键路径有 span 追踪，AI 调用有 decision_id，请求链路可追溯

### 修改文件

| 文件 | 改动 |
|------|------|
| `items/route.ts` POST | 包裹 startSpan/endSpan |
| `goals/route.ts` POST | 包裹 startSpan/endSpan |
| `insights/route.ts` GET | 包裹 startSpan/endSpan |
| `parse-semantic.ts` | 接入 genDecisionId('PARSE')，记录到日志 |
| `enhance-record.ts` | 接入 genDecisionId('ENHANCE')，记录到日志 |
| `record-service.ts` | createRecordSafely 内部加 span |
| 各新域服务 | genDecisionId 在 issues 合并节点插入；genToolCallId 在 DB 调用前插入 |

### genDecisionId 激活点（5 类）
1. 校验失败 → `genDecisionId('VALIDATION')`
2. 类型检查 → `genDecisionId('TYPE_CHECK')`
3. 状态转换 → `genDecisionId('STATE_TRANSITION')`
4. 资格判定 → `genDecisionId('ELIGIBILITY')`
5. AI 回退 → `genDecisionId('AI_FALLBACK')`

### genToolCallId 激活点（6 类）
- `genToolCallId('ITEM_CREATE')`、`genToolCallId('GOAL_CREATE')`、`genToolCallId('GOAL_ENGINE')`
- `genToolCallId('PHASE_CREATE')`、`genToolCallId('TAG_CREATE')`、`genToolCallId('SUBITEM_CREATE')`
- 已有的 `genToolCallId('RECORD_CREATE')` 保持不变

### 验证
- `npm run build` 通过
- genDecisionId ≥ 5 个调用点，genToolCallId ≥ 6 个调用点
- 3+ 路由有 span 追踪

```
Batch 1 (域服务) → Batch 2 (路由接入) → Batch 3 (计算中心) → Batch 4 (读写隔离)
                                                                  ↓

```
