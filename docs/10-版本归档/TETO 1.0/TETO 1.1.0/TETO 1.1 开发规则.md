7. Excel / 表格导入

## 六、本阶段核心开发原则

### 6.1 单主闭环原则

> **1.1 的所有开发必须优先服务唯一主闭环。**

唯一主闭环为：

> 任务定义 → 目标设置 → 每日记录 → 完成度计算 → 项目关联

如果某项需求不能直接增强这条路径，就不应抢占前期优先级。

### 6.2 单任务块推进原则

> **一次只推进一个明确任务块，不并行铺开所有 1.1 内容。**

具体要求：

- 不同时开做多个大页面
- 不同时推进多个复杂系统
- 一个任务块完成并验收后，再进入下一个任务块

### 6.3 最小可用闭环原则

> **先做可运行最小闭环，再补增强功能。**

TETO 1.1 的最小可用闭环固定为：

1. Tasks
2. Task Detail / Edit
3. Records

只要这三块跑通，就已经满足：

- 能建任务
- 能设目标
- 能记每日值
- 能查看完成情况

### 6.4 简单稳定优先原则

默认优先选择：

- 更简单
- 更稳定
- 更贴近当前版本
- 更少改动
- 更容易验证

不优先选择：

- 更通用
- 更复杂
- 更大而全
- 更偏未来预埋
- 更依赖抽象系统

### 6.5 不顺手扩展原则

> **即使进入了 1.1，也不允许因为“顺手”就无限扩大范围。**

例如：

- 做任务页时，不顺手做通用表单引擎
- 做项目结构时，不顺手做无限层级树系统
- 做完成度时，不顺手做高级 BI 报表中心
- 做导入时，不顺手做万能字段映射平台

## 七、本阶段开发顺序原则

### 7.1 固定优先级

本阶段固定开发优先级如下：

#### 第一优先级：主闭环基础

1. 数据结构落地
2. Tasks 页面
3. Task Detail / Edit 页面
4. Records 页面

#### 第二优先级：汇总与关联

5. Dashboard 页面
6. Projects 页面
7. Project Detail 页面

#### 第三优先级：增强能力

8. Import 页面
9. Layout Preferences / 区块排序

### 7.2 顺序约束

必须遵守以下顺序约束：

- 没有任务定义，不做记录页细化
- 没有目标结构，不做完成度口径扩张
- 没有记录闭环，不做复杂 Dashboard
- 没有项目挂接，不做项目汇总展示
- 没有基础导入模板，不做复杂导入增强

## 八、本阶段禁止事项

### 8.1 绝对禁止

以下事项在 TETO 1.1 阶段​**绝对禁止**：

1. **多人协作系统**

   - 不做多人账号体系
   - 不做团队空间
   - 不做复杂权限系统
2. **无限扩展型项目系统**

   - 不做无限层级项目树
   - 不做复杂项目依赖图
3. **复杂自动化系统**

   - 不做自动规则引擎
   - 不做自动派生任务系统
   - 不做复杂 workflow 编排
4. **AI 高级能力**

   - 不做 AI 自动解析任务配置
   - 不做 AI 自动规划器
   - 不做 AI 驱动任务生成器
5. **复杂报表中心**

   - 不做通用 BI 看板
   - 不做自定义图表搭建器
   - 不做自由分析器
6. **跨版本预埋**

   - 不为未来版本提前做抽象架构
   - 不为未来版本提前做通用引擎
   - 不把 1.2 的东西带进 1.1

### 8.2 默认不做

以下内容默认不在 1.1 第一轮展开：

- 移动端专用交互体系
- 离线同步系统
- 通知中心
- 审批流
- 复杂附件系统
- 公共模板市场
- 多维标签体系
- 跨项目自动汇总引擎

如确有必要，必须在后续阶段单独立项。

## 九、数据库规则

### 9.1 SQL 落地规则

凡涉及数据库结构新增或修改，必须：

1. 生成 SQL
2. 将 SQL 落到项目根目录 `sql/` 文件夹
3. 使用清晰、递增、可读命名
4. 不允许只在回复里给 SQL 而不落地文件
5. 尽量使用 `IF NOT EXISTS`
6. 不擅自引入 migration 系统，除非用户明确要求

### 9.2 数据库设计原则

TETO 1.1 的数据库设计必须遵守：

1. 优先服务真实页面与真实读写
2. 不为了抽象而抽象
3. 不为了未来无限扩展而先做复杂表
4. 能被页面真实验证
5. 能在 Supabase 中直观看到结果
6. 字段命名清晰、含义单一、职责明确

### 9.3 数据闭环原则

任何数据结构都必须能被以下方式验证：

- 页面能创建
- 页面能读取
- 页面能编辑
- 页面能回显
- 数据库中能看到真实记录
- 页面计算结果与数据库记录一致

## 十、页面与交互规则

### 10.1 页面职责单一原则

每个页面只承担明确职责：

- Tasks：管理任务定义
- Task Detail / Edit：编辑任务与目标
- Records：按日记录任务值
- Dashboard：总览与快捷入口
- Projects：管理项目结构
- Project Detail：查看项目维度信息
- Import：批量导入
- Layout Preferences：布局偏好

不允许把多个复杂工作流强行塞到同一页。

### 10.2 交互控制原则

前期交互优先：

- 可录入
- 可保存
- 可回显
- 可验证

前期不优先：

- 炫酷动画
- 复杂拖拽
- 高级联动配置
- 高自由布局系统

### 10.3 拖拽能力限制

1.1 允许的拖拽能力仅限：

- Dashboard 固定区块排序
- 可保存区块顺序偏好

1.1 不允许：

- 自由网格拖拽
- 组件级布局搭建器
- 跨页面布局编排器

## 十一、导入规则

### 11.1 导入定位

导入能力在 1.1 中属于增强模块，不是第一主闭环阻塞项。

### 11.2 导入范围限制

1.1 导入仅支持：

- 固定模板
- 明确字段
- 记录类数据导入

不支持：

- 万能任意表导入
- 自定义脚本导入
- 智能猜测型复杂映射系统

### 11.3 冲突处理原则

第一版导入默认：

- 冲突先提示
- 不自动覆盖
- 不做复杂合并策略

TETO 1.1 项目结构只支持：

- 主项目
- 一级子项目

项目在 1.1 中是：

> **任务的归属与汇总维度**

而不是一个独立的大型复杂项目管理平台。

## 十三、验收规则

### 13.1 完成标准

TETO 1.1 中，“完成”不以代码提交为准，而以**真实可验证**为准。

默认验收必须确认：

|检查项|要求|
| ----------| --------------------------------------|
|页面访问|页面能正常打开|
|数据创建|能真实创建数据|
|数据保存|新增/修改能真实保存|
|数据读取|已保存数据能读取|
|数据编辑|已有数据能修改|
|数据回显|页面能正确回显|
|Supabase|数据库中能看到对应记录|
|计算结果|完成度/统计结果与真实数据一致|
|闭环完整|功能不是“写了代码”，而是能完整操作|

### 13.2 验收方式

1. **本地验证**

   - 启动开发环境
   - 完整走一遍操作流程
   - 查看页面结果
   - 检查 Supabase 对应记录
2. **线上验证**

   - 部署后完整操作
   - 检查真实数据
   - 确认无报错
   - 确认展示结果一致

### 13.3 闭环验收要求

1.1 至少必须能真实完成以下动作链：

- 新建一个任务
- 为任务设置目标
- 在某一天录入任务值
- 在页面看到任务完成度
- 在数据库中看到对应数据
- 在项目维度看到该任务归属

若这条链跑不通，则 1.1 主闭环未完成。

## 十六、最终拍板结论

TETO 1.1 正式定义为：

> **以“任务定义、目标设置、每日记录、完成度、项目归属”为核心的新阶段开发。**

本阶段允许新增页面、路由、业务表与任务型闭环能力，  
但必须严格围绕 1.1 主闭环推进，禁止多人协作、复杂权限、无限层级项目、复杂自动化、AI 高级能力与跨版本预埋。

TETO 1.1 的唯一核心目标不是“做大”，而是：

> **先做出真实可用、真实可验证、结构稳定的任务闭环。**
