# 《TETO 1.3阶段开发清单与验收清单》

## 一、文档定位

本文件用于把前两份文档直接落成执行顺序。

本文件只解决三件事：

1. **第一阶段到底先做什么，后做什么**
2. **每一步做到什么程度才算完成**
3. **什么情况下不能进入下一步**

本文件不是产品构思文档，  
也不是技术方案长文。  
它就是第一阶段的实际推进清单。

---

## 二、第一阶段总目标

第一阶段只验证一个最小闭环：

> **用户可以围绕时间记录真实发生，补充状态，标记成果，关联目标，并在回看中看见结果。**

只要这个闭环没有真实跑通，  
第一阶段就不算完成。

---

## 三、第一阶段开发总原则

### 原则 1：只做闭环，不做扩张

先做成一个能用的最小系统，  
不同时展开太多支线。

### 原则 2：先主链路，后查看页

先把“记录 → 保存 → 回看”做通，  
再补全周边页面。

### 原则 3：先可验证，后优化

先确认能真实使用，  
再谈结构优化、体验优化、样式优化。

### 原则 4：一步一验，不跳步

每完成一个任务块，  
必须先验收，再进入下一块。

---

## 四、第一阶段开发顺序总表

第一阶段开发顺序固定为：

### 阶段 1

**底层数据结构与基础读写能力**

### 阶段 2

**今日页主链路**

### 阶段 3

**发生记录与详情回看**

### 阶段 4

**成果记录与成果页**

### 阶段 5

**目标系统与目标页**

这是固定顺序。  
不允许先做容器页，不允许先做复杂复盘，不允许先做 AI。

# 五、详细开发清单

---

## 阶段 1：底层数据结构与基础读写能力

### 目标

先把第一阶段最小数据模型落地，  
保证核心对象可以真实创建、保存、读取。

### 本阶段必须完成的对象

1. ​`time_block`
2. ​`event`
3. ​`outcome`
4. ​`goal`
5. ​`container`
6. ​`state_tag`

### 本阶段必须完成的能力

1. 能创建 time\_block
2. 能创建 event
3. event 能关联 time\_block
4. outcome 能关联 event
5. event 能关联 goal
6. event 能关联 container
7. event 能关联 state\_tag
8. 所有对象都能读取
9. 所有对象都能更新

### 本阶段明确不做

- 复杂联表优化
- 多对多高级关系
- 自动计算系统
- 权限系统
- AI 写入逻辑

### 本阶段验收标准

必须至少验证以下事实：

- 能手动创建一个时间段
- 能给这个时间段创建一条发生
- 能给这条发生补一个成果
- 能给这条发生补一个状态
- 能给这条发生挂一个目标
- 能读取并显示完整内容
- 刷新后数据不丢失

### 不通过条件

以下任一出现，本阶段不算完成：

- 数据能写不能读
- 数据能读不能改
- 关联关系保存后丢失
- 页面展示和实际数据不一致
- 只能演示假数据，不能跑真数据

---

## 阶段 2：今日页主链路

### 目标

把“今日页”做成系统的现实入口页。

### 本阶段必须完成的页面能力

1. 今日页可打开
2. 今日页可展示当天时间相关记录
3. 今日页可新增发生项
4. 新增发生时可填写最小字段：

   - 时间
   - 标题
   - 类型
5. 已创建发生项可在今日页列表中看见
6. 可从今日页进入发生详情或编辑

### 今日页第一版必须有的模块

1. 今日记录入口
2. 今日发生列表
3. 单条发生快速补全入口

### 今日页第一版不做

- 大型仪表盘
- 花哨统计卡片
- AI 建议区
- 多维任务看板
- 项目概览中枢

### 本阶段验收标准

必须至少能跑通：

> 打开今日页 → 新增一条今天的发生 → 保存成功 → 当页看见记录 → 刷新后仍存在

### 不通过条件

- 今日页只是静态壳子
- 不能直接记录发生
- 记录后列表不更新
- 必须跳到其他页才能完成核心记录

---

## 阶段 3：发生记录与详情回看

### 目标

把“发生”真正立起来，  
让系统先成立在现实层。

### 本阶段必须完成的能力

1. 发生项详情页或详情区
2. 可编辑发生描述
3. 可设置主动/被动
4. 可设置计划内/计划外
5. 可设置状态
6. 可查看关联时间信息
7. 可查看关联目标/容器信息
8. 可在发生页查看全部发生项

### 发生页第一版必须有的筛选

1. 按日期筛选
2. 按发生类型筛选

### 第一版可以暂缓的内容

- 高级组合筛选
- 排序器大全
- 批量编辑
- 拖拽式时间轴

### 本阶段验收标准

必须至少能跑通：

> 在今日页创建发生 → 进入发生详情 → 补充主动/被动、计划内/外、状态 → 保存 → 返回发生页可看到更新结果

### 不通过条件

- 发生只有标题，没有完整信息结构
- 状态不能保存
- 发生页无法查看真实历史数据
- 详情编辑后不能回显

---

## 阶段 4：成果记录与成果页

### 目标

让系统不只停在“发生了什么”，  
而是开始回答“形成了什么”。

### 本阶段必须完成的能力

1. 可为发生项补成果
2. outcome 至少支持以下类型：

   - 完成
   - 推进
   - 产出
   - 恢复
   - 维持
   - 修复
   - 消耗
   - 偏离
   - 无明显成果
3. 可填写成果摘要
4. 可填写目标接近度：

   - 更接近
   - 无明显变化
   - 更偏离
5. 成果页可查看全部成果项
6. 成果页可看到成果来源发生

### 本阶段不做

- 多成果拆分
- 成果权重评分
- 自动成果识别
- AI 自动判断接近度

### 本阶段验收标准

必须至少能跑通：

> 选择一条发生 → 补一条成果 → 设置成果类型与目标接近度 → 保存 → 在成果页可看见该成果 → 点开可看到来源发生

### 不通过条件

- 成果保存后无法回看
- 成果页只是假数据
- 成果无法关联到真实发生
- 目标接近度无法显示

---

## 阶段 5：目标系统与目标页

### 目标

让系统具备方向层，  
不是只停留在时间和流水。

### 本阶段必须完成的能力

1. 创建目标
2. 编辑目标
3. 查看目标列表
4. 在发生项中关联目标
5. 目标页可查看：

   - 目标基本信息
   - 关联发生
   - 关联成果

### 目标页第一版必须回答的问题

1. 这个目标最近有哪些相关发生
2. 这个目标最近形成了哪些成果
3. 这些成果整体是更接近、无明显变化还是更偏离

### 本阶段不做

- 目标树
- 子目标系统
- 目标自动拆解
- 复杂路线图

### 本阶段验收标准

必须至少能跑通：

> 创建一个目标 → 在一条发生上挂这个目标 → 给该发生补成果 → 打开目标页 → 能看到该发生和成果已关联进去

### 不通过条件

- 目标只能创建，不能承接关联
- 目标页看不到真实发生
- 目标和成果之间没有链路

---

## 阶段 6：复盘页基础版

### 目标

让用户开始看见“时间去了哪里、发生了什么、形成了什么”。

### 本阶段必须完成的能力

1. 可按日期范围查看记录
2. 可看到发生总量
3. 可看到成果总量
4. 可看到成果类型分布
5. 可看到目标关联基础情况
6. 可看到时间范围内的基础列表回看

### 第一版复盘页允许非常朴素

甚至可以先是：

- 汇总数字
- 简单分组
- 列表回看

不要求先上图表大屏。

### 本阶段不做

- 高级 BI
- 复杂多维钻取
- AI 周报
- 自动洞察结论

### 本阶段验收标准

必须至少能跑通：

> 选择一个日期范围 → 能看到该范围内的发生与成果汇总 → 能看见不同成果类型的基础分布 → 能看见关联目标的基础情况

### 不通过条件

- 复盘页只显示静态数字
- 数据和真实记录对不上
- 只能看总数，不能回看对应内容

---

## 阶段 7：容器页基础版

### 目标

保留任务、项目、领域、主题等组织能力，  
但明确它们已经降级，不再统治底层。

### 本阶段必须完成的能力

1. 创建容器
2. 编辑容器
3. 查看容器列表
4. 在发生项中挂容器
5. 容器页能看见：

   - 关联发生
   - 关联成果

### 第一版容器类型

1. 任务
2. 项目
3. 领域
4. 主题

### 本阶段不做

- 容器层级嵌套
- 看板系统
- 甘特图
- 项目高级管理功能

### 本阶段验收标准

必须至少能跑通：

> 创建一个容器 → 给某条发生挂这个容器 → 给该发生补成果 → 打开容器页 → 能看到关联发生与成果

### 不通过条件

- 容器存在但无法承接真实数据
- 容器抢占首页
- 容器逻辑倒逼记录逻辑

---

## 阶段 8：全链路联调验收

### 目标

确认第一阶段不是“页面都做了”，  
而是真的成为一个可用闭环。

### 必须跑通的完整链路

至少手动跑通以下场景：

#### 场景 A：主动推进类

> 创建今天 10:00-11:00 时间段 → 记录“撰写方案” → 标记主动 / 计划内 → 状态“专注” → 成果“推进” → 目标接近度“更接近” → 关联目标“完成新骨架设计” → 在今日页、发生页、成果页、目标页、复盘页都能看见

#### 场景 B：被动应急类

> 创建今天 14:00-15:00 时间段 → 记录“处理临时问题” → 标记被动 / 计划外 → 状态“疲惫” → 成果“维持”或“修复” → 目标接近度“无明显变化” → 在相关页面都可回看

#### 场景 C：恢复类

> 创建今天 22:00-22:40 时间段 → 记录“散步恢复” → 标记主动 / 计划内或计划外均可 → 状态“恢复中” → 成果“恢复” → 目标接近度“更接近”或“无明显变化” → 在成果页和复盘页能被正确理解为真实成果，而不是空白时间

### 本阶段验收标准

只有当这三类场景都跑通时，  
第一阶段才算基本成立。

### 不通过条件

- 只能支持任务型记录
- 恢复型记录无法表达
- 被动事件无法表达
- 只有发生，没有成果
- 只有成果，没有来源
- 只有目标，没有关联链路
- 复盘看不出真实差异

# 七、第一阶段完成标准

第一阶段完成，不以“页面数量”判断，  
也不以“代码量”判断。

第一阶段只有一个完成标准：

> **真实最小闭环已经可用。**

也就是：

1. 用户能记录现实
2. 用户能看见发生
3. 用户能标记成果
4. 用户能关联目标
5. 用户能回看结果
6. 系统能表达主动、被动、恢复、推进等不同现实类型

只要这 6 条没有同时成立，  
就不能说第一阶段完成。

---

# 八、第一阶段明确禁止事项

在第一阶段完成前，以下事项一律禁止抢优先级：

1. UI 精修
2. 动画优化
3. 大屏图表
4. AI 自动总结
5. AI 智能建议
6. 项目高级管理
7. 多用户系统
8. 企业权限
9. 自动化工作流
10. 复杂评分模型
11. 复杂目标树
12. 平台级预埋设计

原则很简单：

> **闭环没通，任何增强都不该先做。**
