# TETO AI 开发协作规则 ai-dev-rules

## 1. 文档定位
本文件用于约束 TETO 1.0 项目的 AI 协作开发行为，作为我与 Trae / AI 助手协作时的固定执行规则。

目标不是发散设计，而是确保开发顺序稳定、范围清晰、输出可验证。

---

## 2. 项目阶段
当前只开发 TETO 1.0。

---

## 3. TETO 1.0 固定范围
当前只做以下内容：

- Daily Record
- Diary / Review
- Projects
- Dashboard
- Stats
- 基础认证
- Supabase 数据存储

当前明确不做：

- 第二大脑
- 财务模块
- AI 自动日记解析
- 语音自动结构化
- 多人协作
- 企业版
- 复杂角色权限
- 高级动态配置系统

---

## 4. 开发顺序
严格按以下顺序推进，不要跳步：

1. 基础设施
2. Supabase 与认证
3. Daily Record
4. Diary / Review
5. Projects
6. Dashboard
7. Stats

---

## 5. 开发执行规则
1. 一次只推进一个明确任务块
2. 不要同时开发多个无关模块
3. 不要擅自扩展功能
4. 不要在未被要求时擅自重构
5. 不要顺手修改无关页面、组件或数据结构
6. 如果当前任务没有明确要求，不要自行引入复杂抽象层

---

## 6. 数据库与 SQL 规则
凡是涉及数据库结构新增或修改，必须同时满足：

1. 生成 SQL
2. 把 SQL 保存到项目根目录 `sql/` 文件夹
3. 使用清晰、递增、可读的命名
4. 不允许只在回复里给 SQL 而不实际落地文件
5. 尽量使用 `IF NOT EXISTS`，保证可重复执行
6. 不要擅自引入 migration 系统，除非我明确要求

### 6.1 SQL 文件命名示例
- `sql/001_init_profiles.sql`
- `sql/002_daily_records.sql`
- `sql/003_diary_reviews.sql`
- `sql/004_projects_and_logs.sql`

---

## 7. 输出要求
每次完成任务后，必须清晰告诉我：

1. 修改了哪些文件
2. 新增了哪些文件
3. 如果涉及数据库，新增了哪些 SQL 文件
4. 我还需要补哪些环境变量或配置
5. 我应该如何本地验证
6. 当前任务是否已经达到可验收状态
7. 下一步最适合做什么

---

## 8. 回答方式要求
1. 先给直接结论，再讲方法
2. 优先告诉我当前最应该做的步骤
3. 不要同时给多个平行方案让我自己选
4. 不要长篇空泛解释
5. 结论必须可立刻执行
6. 如果我问开发顺序，必须给出唯一优先动作

---

## 9. 默认原则
如果需求不明确，默认选择：

- 更简单
- 更稳定
- 更贴近 1.0
- 更少改动

而不是：

- 更大
- 更炫
- 更抽象
- 更平台化

---

## 10. 禁止事项
不要擅自：

- 扩表
- 加新模块
- 加无关页面
- 加复杂动态配置
- 加复杂权限系统
- 加 migration 框架
- 加花哨 UI
- 做与当前任务无关的重构
- 偏离 TETO 1.0 主闭环

---

## 11. 验收原则
是否完成，不以“代码写了”为准，而以“真实可验证”为准。

必须至少满足以下验证标准中的相关项：

- 页面能打开
- 功能能真实操作
- 数据能真实保存
- Supabase 中能看到记录
- 数据能读取
- 数据能编辑
- 数据能正确回显

---

## 12. 当前协作原则
我当前是在搭建 TETO 1.0 的稳定开发闭环。

因此 AI 助手必须优先帮助我：
1. 锁定当前唯一下一步
2. 限制需求扩散
3. 保持开发顺序稳定
4. 保证数据库修改可落地、可执行、可验证
5. 保证每一步都能验收，而不是只停留在“代码看起来写了”

---

## 13. 使用方式
每次给 Trae / AI 助手发开发任务前，可附加一句：

`请严格遵守 docs/ai-dev-rules.md 规则执行，不要超出当前任务范围。`

如果任务涉及数据库，再附加一句：

`如果涉及数据库结构修改，必须同时把 SQL 落到项目根目录 sql/ 文件夹。`
