# 1. /stats V2 页面结构方案

### 第一层：全局核心摘要

- 全局时间范围选择器 ：周 / 月 / 年
- 核心指标卡片 （4个）：

  - 本周状态（目标/完成/差值）
  - 本月状态（目标/完成/差值）
  - 累计状态（目标/完成/差值）
  - 落后任务数

### 第二层：任务统计主表

- 可折叠 ：默认展开
- 核心列 ：

  - 任务名
  - 本周：完成 / 目标 / 进度
  - 本月：完成 / 目标 / 进度
  - 累计：完成 / 目标 / 进度
  - 趋势（上升/下降/稳定）
  - 操作（查看详情）

### 第三层：趋势分析

- 可折叠 ：默认展开
- 固定维度 ：

  - 近7天趋势
  - 近30天趋势

### 第四层：项目维度汇总

- 可折叠 ：默认收起
- 核心信息 ：

  - 项目名
  - 活跃任务数
  - 完成率
  - 风险状态

### 第五层：任务详情抽屉

- 点击任务主表中的“查看详情”打开
- 结构化信息 ：基础信息、当前结果、趋势与记录、判断与建议

## 2. 全局时间范围最终规则

### 全局时间范围影响模块：

- 全局核心摘要（本周/本月/累计状态）
- 任务统计主表（所有时间相关列）
- 趋势分析（近7天/近30天趋势）

### 全局时间范围不影响模块：

- 项目维度汇总（固定显示所有项目）
- 任务详情抽屉（固定显示本周/本月/累计 + 近7天/近30天）

### 任务详情规则：

- 固定显示：本周 / 本月 / 累计依据
- 固定显示：近7天 / 近30天趋势
- 不随全局时间范围变化而改变结构
- 如需体现当前筛选，仅在顶部显示“当前筛选：XX”提示

## 5. 数据口径统一表

### 1. 任务管理中的目标配置字段

- 支持：日目标 / 周目标 / 月目标
- /stats 读取优先级：月目标 > 周目标 > 日目标

### 2. /stats 的各类目标值计算

- 本周目标 ：日目标 × 7 或 周目标
- 本月目标 ：日目标 × 30 或 周目标 × 4.29 或 月目标
- 累计目标 ：同本月目标（当前月份）
- 近7天目标 ：同本周目标
- 近30天目标 ：同本月目标
- 预测值相关目标 ：使用本月目标

### 3. /stats 的各类完成值计算

- 本周完成 ：从 task_daily_records 表，筛选本周日期范围，累加 value_number
- 本月完成 ：从 task_daily_records 表，筛选本月日期范围，累加 value_number
- 累计完成 ：从 task_daily_records 表，筛选本月日期范围，累加 value_number
- 近7天完成 ：从 task_daily_records 表，筛选近7天日期范围，累加 value_number
- 近30天完成 ：从 task_daily_records 表，筛选近30天日期范围，累加 value_number

### 4. 差值定义

- 差值 = 目标 - 完成
- 正值表示未达标，负值表示超额完成

### 5. 进度定义

- 进度 = (完成 / 目标) × 100%
- 允许超过 100%
- 格式：保留一位小数，如 105.2%

### 6. 小数显示规则

- 必须取整 ：任务数、项目数、天数
- 允许一位小数 ：进度百分比
- 允许两位小数 ：速度、差值
- 绝不允许裸小数 ：所有数值必须格式化

### 7. 无目标值任务处理

- 显示“无目标”
- 不参与汇总分母
- 在汇总计算中排除，避免误导

## 6. 删除 / 保留 / 下沉 / 延后清单

## 7. 如果进入实现，唯一优先动作是什么

唯一优先动作 ：统一数据口径，确保任务管理中的目标配置与今日记录中的实际记录在 /stats 页面的展示结果一致，解决数据不一致问题。

具体来说：

1. 修改 /api/stats/route.ts ，确保所有统计数据都基于统一的数据源和计算规则
2. 修正 calculateStats 函数，确保目标值和完成值的计算逻辑正确
3. 确保任务详情中的数据与主表一致，避免时间范围冲突
4. 实现小数格式化，确保所有数值显示符合规范
