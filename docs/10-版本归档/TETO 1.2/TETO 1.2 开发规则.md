# TETO 1.2 开发规则

## 一、文档定位

本文件是 TETO 1.2 阶段的版本规则文档，用于定义：

1. 本阶段开发目标
2. 本阶段范围边界
3. 本阶段允许事项与禁止事项
4. 本阶段执行顺序原则
5. 本阶段验收标准
6. 本阶段与其他文档的优先级关系

- 代替《TETO AI 协作总则（总控版）》
- 代替《TETO 1.2 开发清单》
- 代替页面设计稿
- 代替数据结构设计稿

- 代替具体实现代码

## 三、阶段定义

### 3.1 当前阶段定义

```txt
- 当前主阶段：TETO 1.2
- 当前任务类型：结构整理与页面归位
- 前置阶段：TETO 1.1（已结项）
- 后续阶段：TETO 1.3（当前禁止进入）
```

### 3.2 阶段目标

TETO 1.2 的阶段目标定义为：

> 以“结构整理、页面归位、功能去重、表达校准”为核心，
> 把 1.1 已经落地的能力收敛成清晰、稳定、可持续使用的结构。

### 3.3 阶段本质

TETO 1.2 是：

- 从 1.1 的“功能已做出但结构混乱”，转向“结构清晰、职责稳定”
- 从多个重复入口，转向单一明确入口
- 从功能名词膨胀，转向按真实实现粒度表达
- 从页面并存与职责打架，转向统一归位

### 3.4 阶段来源

TETO 1.1 完成报告已经明确指出：

- 界面与功能组织存在混乱
- 存在功能重复
- 文档数量变多且互相冲突
- 表述曾过度夸张
- 后续应优先处理结构整理与表达校准

因此，1.2 不是扩张版，而是收敛版。

## 四、阶段范围边界

### 4.1 本阶段必须覆盖的核心范围

本阶段必须围绕以下事项展开：

1. 一级页面结构整理
2. 页面职责重新归位
3. 重复页面删除或合并
4. 今日记录入口统一
5. 首页 / 工作台 / 仪表盘重复问题清理
6. 任务管理、项目管理、统计分析职责边界固定
7. 菜单命名、页面命名、路由命名统一
8. 文档表达校准
9. 真实可用性修补
10. 必要的小范围兼容修正

### 4.2 本阶段优先服务的主闭环

TETO 1.2 的主闭环固定为：

> 进入系统 → 看今日概览 → 做今日记录 → 管任务 → 看项目 → 查统计

只要某项开发内容不能直接提升这条使用路径，就不是 1.2 第一优先级。

### 4.3 本阶段允许新增的内容类型

本阶段允许：

- 调整页面结构
- 删除重复页面
- 新增必要的小型摘要区
- 统一入口与默认首页
- 调整菜单与路由
- 调整页面内模块归属
- 做少量数据库兼容修补
- 为结构整理所必须的前后端修改

本阶段不以新增大功能为主。

## 六、本阶段核心开发原则

### 6.1 收敛优先原则

> 1.2 的第一原则不是“继续做大”，而是“先收敛”。

只要一个问题可以通过合并、删除、归位解决，就不优先用新增页面或新增系统解决。

### 6.2 去重优先原则

> 只要两个页面或两个入口承担的是同类职责，就优先考虑去重。

不允许在 1.2 中继续保留多个“首页性质”的页面并存。

### 6.3 单职责原则

> 每个一级页面只承担一个清晰职责。

不允许：

- 一个职责拆成多个重复页面
- 多个复杂职责混塞进一个页面
- 页面名字与页面实际用途长期不一致

### 6.4 最小改动优先原则

默认优先选择：

- 更简单
- 更稳定
- 更少改动
- 更容易验证
- 更贴近当前真实实现

不优先选择：

- 更抽象
- 更通用
- 更偏未来版本
- 更依赖大面积重构
- 更像平台化预埋

### 6.5 真实实现优先原则

1.2 中的文档、命名、页面归类，都必须服从真实实现。

不允许再把：

- 任务管理中的组成能力
- 统计展示中的一部分内容
- 页面上的一个摘要区

夸写成独立大系统。

### 6.6 单任务块推进原则

> 一次只推进一个明确任务块。

例如：

- 先定菜单结构
- 再删重复页面
- 再补今日概览区
- 再统一路由文案

不并行铺满所有改动。

## 七、本阶段固定结构规则

### 7.1 一级结构固定

TETO 1.2 一级结构固定为：

1. 今日记录
2. 任务管理
3. 项目管理
4. 统计分析

### 7.2 默认首页固定

默认首页固定为：

> 今日记录

### 7.3 今日记录顶部规则

今日记录顶部允许存在：

> 今日概览区

但这个概览区的定位是：

- 摘要区
- 快速判断区
- 进入系统后的第一眼信息区

它不是：

- 独立工作台
- 独立仪表盘
- 独立首页替代页

### 7.4 不再保留的一级页面

以下页面在 1.2 中不再独立保留：

- 首页
- 工作台
- 仪表盘
- 每日记录（旧页）
- 日记复盘（独立入口）

## 八、本阶段开发顺序原则

### 8.1 固定优先级

本阶段固定开发顺序如下：

#### 第一优先级：结构定稿

1. 定一级结构
2. 定默认首页
3. 定页面职责
4. 定保留页 / 删除页 / 并入页

#### 第二优先级：去重与归位

5. 删除旧“每日记录”
6. 取消“日记复盘”独立入口
7. 删除“首页 / 工作台 / 仪表盘”独立存在
8. 把可保留摘要归位到今日记录或统计分析

#### 第三优先级：页面整理

9. 整理今日记录
10. 整理任务管理
11. 整理项目管理
12. 整理统计分析

#### 第四优先级：表达统一

13. 统一菜单文案
14. 统一按钮文案
15. 统一路由命名
16. 统一文档表述

#### 第五优先级：小范围修补

17. 修跳转问题
18. 修空状态和错误状态
19. 修读写回显问题
20. 做必要兼容调整

### 8.2 顺序约束

必须遵守以下顺序约束：

- 没定一级结构，不做页面美化
- 没定页面职责，不做局部细枝末节优化
- 没删重复页面，不做新的摘要页
- 没整理入口，不扩展统计展示
- 没完成结构归位，不进入未来版本规划

## 九、本阶段页面职责规则

### 9.1 今日记录

今日记录只负责：

- 今日任务录入
- 今日备注 / 日记 / 简短总结
- 状态填写
- 补录
- 顶部概览摘要

今日记录不负责：

- 任务定义配置
- 项目结构配置
- 深度历史统计主看板
- 独立工作台功能扩张

### 9.2 任务管理

任务管理只负责：

- 任务列表
- 任务创建
- 任务编辑
- 目标值配置
- 单位、类型、参与统计等属性
- 与任务有关的拖拽、导入、关联等组成能力

任务管理不负责：

- 今日记录主入口
- 独立全局首页
- 深度历史统计中心

### 9.3 项目管理

项目管理只负责：

- 项目列表
- 项目详情
- 项目状态
- 项目推进
- 项目下任务查看

项目管理不负责：

- 任务规则配置
- 今日主记录入口
- 统计中心

### 9.4 统计分析

统计分析只负责：

- 历史记录查看
- 周 / 月 / 年统计
- 趋势查看
- 汇总结果查看
- 项目维度汇总结果

统计分析不负责：

- 今日主录入
- 任务定义配置
- 项目结构配置

## 十、数据库与 SQL 规则

### 10.1 数据库变更限制

1.2 原则上不是数据库大扩张阶段。

因此：

- 可以做小范围兼容变更
- 可以做必要字段补充
- 可以做为结构整理服务的小范围调整

但不允许：

- 为新大模块重建整套数据结构
- 为未来版本预埋复杂模型
- 借结构整理之名开启数据库大重构

### 10.2 SQL 落地规则

凡涉及数据库结构新增或修改，必须：

1. 生成 SQL
2. 将 SQL 落到项目根目录 `sql/`
3. 使用清晰、递增、可读命名
4. 明确提醒用户去 Supabase SQL Editor 手动执行
5. 用户未反馈执行成功前，不算完成
6. 执行后必须做真实验证

### 10.3 数据闭环原则

任何因 1.2 调整而影响的数据逻辑，都必须能验证：

- 页面能打开
- 数据能保存
- 数据能读取
- 数据能编辑
- 数据能回显
- Supabase 中能看到对应结果
- 页面展示与数据库真实一致

## 十一、本阶段禁止事项

### 11.1 绝对禁止

以下事项在 TETO 1.2 阶段绝对禁止：

1. 多人协作系统
2. 企业级权限系统
3. 第二大脑
4. 财务模块
5. AI 自动解析 / 自动规划大系统
6. 通用低代码平台
7. 自由布局编辑器
8. 通用 BI 平台
9. 移动端 App
10. 万能导入器
11. 与结构整理无关的大规模重构
12. 把 1.3 规划提前带入 1.2

### 11.2 默认不做

以下内容默认不在 1.2 第一轮展开：

- 炫酷交互动效
- 高级图表中心
- 复杂通知系统
- 跨模块自动化编排
- 大规模样式重做
- 高自由度个性化配置系统

### 11.3 禁止的页面结构行为

1. 不保留多个首页性质页面
2. 不允许同一摘要在多个一级页重复铺开
3. 不允许用新页面掩盖旧页面不删除
4. 不允许页面名字和实际职责长期错位

## 十二、命名与表达规则

### 12.1 命名统一原则

菜单名、页面名、路由名、文档名应尽量一致，避免：

- 页面叫 A，路由像 B，文档写成 C
- 同一个东西反复换名
- 一个名字在不同地方指不同含义

### 12.2 表达降噪原则

1.2 中应尽量减少夸张表达。
2. 能写成“任务管理中的功能”，就不要写成“独立系统”
3. 能写成“统计分析中的视图”，就不要写成“独立平台”
4. 能写成“今日记录顶部概览区”，就不要写成“工作台系统”

### 12.3 文档与实现对齐原则

文档必须描述真实已落地内容，不得长期领先实现或偏离实现。

## 十三、验收规则

### 13.1 完成标准

TETO 1.2 中，“完成”不以代码提交为准，而以真实可验证为准。

默认验收必须确认：

| 检查项   | 要求                                                           |
| ---------- | ---------------------------------------------------------------- |
| 页面访问 | 页面能正常打开                                                 |
| 结构清晰 | 一级结构明确、无重复总入口                                     |
| 默认首页 | 默认首页正确进入今日记录                                       |
| 页面职责 | 每个一级页职责明确                                             |
| 页面去重 | 重复页面已删除或并入                                           |
| 数据保存 | 数据能真实保存                                                 |
| 数据读取 | 数据能真实读取                                                 |
| 数据编辑 | 已有数据能修改                                                 |
| 数据回显 | 页面能正确回显                                                 |
| Supabase | 数据库中能看到对应记录                                         |
| 展示一致 | 页面展示与数据库一致                                           |
| 使用路径 | 用户进入后能顺利完成“看概览→做记录→管任务→看项目→查统计” |

### 13.2 验收方式

1. 本地验证

   - 启动开发环境
   - 完整走一遍页面入口
   - 检查读写回显
   - 检查 Supabase 数据
2. 线上验证

   - 部署后完整操作
   - 验证入口正确
   - 验证主要数据链路
   - 验证无明显结构混乱

### 13.3 结构验收要求

1.2 至少必须真实满足以下判断：

- 用户不会再问“首页和工作台有什么区别”
- 用户不会再遇到多个总览页并存
- 用户知道进入系统后先去哪里
- 用户知道任务、项目、统计分别去哪
- 页面名字与页面职责一致

若这些仍然做不到，则 1.2 结构整理未完成。

## 十六、最终拍板结论

TETO 1.2 正式定义为：

> 以“结构整理、页面归位、功能去重、表达校准”为核心的收敛阶段。

本阶段允许：

- 调整结构
- 删重合并
- 固定入口
- 统一命名
- 校准表达
- 做必要的小范围修补

本阶段不允许：

- 扩大版本范围
- 新增大模块
- 提前进入未来版本
- 借整理之名做无关大重构
