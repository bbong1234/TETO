# TETO 1.0.1 维护规则

## 一、文档定位

本文件是 TETO 1.0.1 阶段的**版本规则文档**，用于定义：

1. 本阶段范围边界
2. 本阶段禁止事项
3. 本阶段执行原则
4. 本阶段验收标准

本文件**不负责**：

- 定义具体维护任务（由《TETO 1.0.1 维护清单》负责）
- 定义 AI 协作总则（由《TETO AI 协作总则（总控版）》负责）
- 定义产品功能设计（由产品文档负责）

## 三、阶段定义

### 3.1 当前阶段

```
- 当前主阶段：TETO 1.0.1
- 当前任务类型：发布后维护
- 前置阶段：TETO 1.0.0（已完成）
- 后续阶段：TETO 1.1（当前禁止进入）
```

### 3.2 阶段目标

> **TETO 1.0 发布后的小修补和可用性增强**

具体包括：

1. 真实使用后暴露的问题修补
2. 反馈闭环补齐（趋势展示、预测展示）
3. 展示体验优化（空白区域、信息密度）
4. 基础交互增强（折叠能力）

### 3.3 阶段边界

**本阶段只处理：**

- 现有模块的小修补
- 现有数据的展示增强
- 现有交互的体验优化
- 影响真实使用的 bug 修复

**本阶段不处理：**

- 新模块建设
- 大规模重构
- 复杂配置系统
- 1.1 方向扩展

## 四、范围边界

### 4.1 允许的模块

本阶段允许在以下模块内进行维护：

| 模块 | 允许的操作 | 禁止的操作 |
|------|-----------|-----------|
| Stats | 趋势展示增强 | 新增统计维度、复杂图表系统 |
| Projects | 预测展示、进度反馈 | 目标值系统、DIY 任务系统 |
| Daily Record | 展示优化、空白处理 | 表格视图重构、新字段扩展 |
| Diary Review | 展示优化、空白处理 | 新分析维度、AI 解析 |
| Dashboard | 信息展示优化 | 新模块入口、复杂配置 |
| 认证系统 | Bug 修复 | 权限系统、多人协作 |

### 4.2 允许的数据库操作

| 操作类型 | 允许 | 说明 |
|----------|------|------|
| 新增字段 | ✅ 仅限展示计算所需 | 如 predicted_remaining_days |
| 新增表 | ❌ | 本阶段不新增业务表 |
| 修改表结构 | ✅ 仅限小修补 | 不做大规模重构 |
| 新增索引 | ✅ | 性能优化允许 |

### 4.3 允许的前端操作

| 操作类型 | 允许 | 说明 |
|----------|------|------|
| UI 微调 | ✅ | 间距、颜色、布局微调 |
| 新增组件 | ✅ 仅限展示用途 | 如趋势图表、预测卡片 |
| 新增页面 | ❌ | 本阶段不新增页面 |
| 新增路由 | ❌ | 本阶段不新增路由 |
| 折叠功能 | ✅ | 基础折叠能力 |
| 拖拽布局 | ❌ | 属于 1.1 或更后版本 |

## 五、执行原则

### 5.1 单任务原则

> **一次只推进一个明确任务块**

- 不并行开发多个大块
- 完成一个验收一个
- 不跳步

### 5.2 最小改动原则

> **优先选择更简单、更稳定、更少改动的方案**

- 能不改数据库就不改
- 能用前端计算就不用后端存储
- 能用现有组件就不新建
- 能小修补就不重构

### 5.3 真实验证原则

> **完成标准以"真实可验证"为准，不以"代码写了"为准**

验收必须满足：

- 页面能打开
- 数据能真实保存
- 数据能读取
- 数据能编辑
- 页面能正确回显
- Supabase 中能看到对应记录
- 趋势/预测展示能真实变化

### 5.4 不顺手扩展原则

> **不因"顺手"而扩展到无关模块**

- 修 Stats 时不顺手改 Dashboard
- 修 Projects 时不顺手做目标值系统
- 修 Daily Record 时不顺手做表格视图

## 六、禁止事项

### 6.1 绝对禁止

以下事项在本阶段**绝对禁止**：

1. **新增模块**
   - 不做第二大脑
   - 不做财务模块
   - 不做新业务模块

2. **大规模重构**
   - 不重构数据库架构
   - 不重构认证系统
   - 不重构页面路由结构

3. **复杂系统**
   - 不做 DIY 任务系统
   - 不做目标值系统
   - 不做自由拖拽布局
   - 不做复杂权限系统

4. **提前进入 1.1**
   - 不提前开发 1.1 功能
   - 不为 1.1 预埋复杂系统
   - 不按 1.1 规划调整当前结构

### 6.2 需确认事项

以下事项需要用户明确确认后才能执行：

1. 数据库新增字段
2. 新增 SQL 文件
3. 修改现有组件结构
4. 调整页面布局

## 七、数据库规则

### 7.1 SQL 落地规则

凡涉及数据库结构新增或修改，必须：

1. 生成 SQL
2. 将 SQL 落到项目根目录 `sql/` 文件夹
3. 使用清晰、递增、可读命名
4. 不允许只在回复里给 SQL 而不落地文件
5. 尽量使用 `IF NOT EXISTS`
6. 不擅自引入 migration 系统

### 7.2 命名规范

```
sql/
  001_init_profiles.sql
  002_daily_records.sql
  003_diary_reviews.sql
  004_projects_and_logs.sql
  005_xxx_enhancement.sql  ← 本阶段新增
```

## 八、验收规则

### 8.1 验收清单

每项维护任务完成后，必须确认：

| 检查项 | 要求 |
|--------|------|
| 页面访问 | 页面能正常打开 |
| 数据保存 | 新增/修改数据能真实保存 |
| 数据读取 | 数据能正确读取 |
| 数据回显 | 页面能正确回显已保存数据 |
| Supabase | 数据库中能看到对应记录 |
| 功能完整 | 功能闭环完整，不只是"代码写了" |

### 8.2 验收方式

1. **本地验证**
   - `npm run dev` 启动开发服务器
   - 完整操作流程
   - 检查 Supabase 数据

2. **线上验证**
   - 部署后完整操作流程
   - 检查真实数据
   - 确认无报错

## 九、切换规则

### 9.1 本文件生效条件

本文件在以下条件下生效：

- 《TETO AI 协作总则（总控版）》中"当前生效开发阶段"声明为 TETO 1.0.1
- 当前生效规则文档声明为本文件

### 9.2 本文件失效条件

本文件在以下条件下失效：

- 用户明确声明切换到新阶段
- 《TETO AI 协作总则（总控版）》中"当前生效开发阶段"被修改

### 9.3 切换后处理

切换到新阶段后：

1. 本文件保留为历史参考
2. 不再作为执行依据
3. 新阶段规则文档生效
