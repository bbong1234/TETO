## 六、1.4 的核心原则

### 1）先清理旧污染，再新增新结构

1.4 不是在旧残留上继续堆功能。  
先把 1.3 残留旧代码清掉，再新增 1.4 能力。

### 2）新增能力必须挂在新骨架上

所有新增内容，都必须服从：

- 记录是第一入口
- 事项是长期主题组织工具
- 洞察是理解层

不允许把旧版任务 / 项目逻辑重新抬回底层第一结构。

### 3）事项必须真正变强，不再只是简单容器

1.4 必须让事项真正具备长期承接能力。  
不能继续停留在“标题 + 描述 + 关联记录”的弱状态。

### 4）先做网页端，不做移动端优化

当前阶段只把网页端主链路做实。  
移动端优化不进入 1.4 当前主任务。

### 5）新增目标，但不先做复杂目标系统

1.4 要新增独立目标功能，  
但先做基础可用版，不做复杂评分模型、复杂路线图、复杂自动判断。

### 6）历史必须进入同一主结构，不另起孤岛

历史事项、历史记录、历史阶段，都应进入当前事项体系和回看体系，  
不能做成一个独立历史仓库。

### 7）完成标准仍然是“真实可验证”

不以“代码写了”为完成，  
而以页面能打开、数据能真实保存、读取、编辑、回显、与数据库一致为完成。

## 九、1.4 固定开发顺序

## P0：先清理 1.3 残留旧代码

### 1. 清理旧页面残留

做什么：

- 删除已不属于当前主结构的旧版页面残留
- 清掉还留在代码库里、但已不再服务记录 / 事项 / 洞察主骨架的页面文件

如何做：

- 先对照现有主导航和真实启用页面
- 逐个排查旧任务页、旧项目页、旧统计页及其变体残留
- 只清理“确认无用”的旧文件，不顺手扩改新结构

最终效果：

- 代码库中的页面层先变干净
- 后续新增 1.4 能力时不再被旧页面逻辑干扰

### 2. 清理旧 API 残留

做什么：

- 删除已失效或已不再属于当前骨架的旧 API 残留

如何做：

- 对照当前实际调用链
- 清理旧任务 / 项目 / 统计相关接口残留
- 确保当前记录 / 事项 / 洞察 API 不受影响

最终效果：

- API 层更干净
- 后续新增 1.4 功能时不会误接旧接口

### 3. 清理旧 DB 操作层与旧类型残留

做什么：

- 删除旧 DB 操作层残留和旧类型文件残留

如何做：

- 对照当前已生效的类型入口和 DB 操作目录
- 清理已失效的旧表操作文件、旧逻辑入口、旧类型文件
- 保证当前生效结构唯一、清晰

最终效果：

- 代码结构更稳定
- 后面新增功能时不容易再次混入旧模型

来源：`TETO 1.3 完成报告.md` 第 5.8、5.9、8.3 节

## P1：新增事项的核心承接能力

### 4. 新增事项内部承接能力

做什么：

- 让事项从“简单容器”升级为“真正可长期使用的主题空间”

如何做：

- 在事项内部加入更完整的内容承接区
- 不再只展示标题、描述、状态、关联记录
- 围绕“一个长期主题如何持续被使用、被组织、被回看”来增强事项页

最终效果：

- 事项页不再是薄弱详情页
- 用户进入事项后，会把它当成长期主题空间来使用
- 事项真正开始承接 1.4 主结构

### 5. 新增事项的桌面图标 / 文件夹式组织能力

做什么：

- 给事项增加更直观的主题入口形式
- 让事项更像桌面图标、文件夹、主题入口，而不是普通文字列表项

如何做：

- 在事项列表和进入方式上增加更清晰的入口感
- 让事项更容易被归类、识别、进入、回看
- 重点做“组织感”和“入口感”，不是先做花哨视觉

最终效果：

- 用户更容易把事项当成真实长期主题入口
- 系统整体更像长期主题桌面，而不是纯列表工具

## P2：新增事项—阶段闭环

### 6. 新增事项下的阶段能力

做什么：

- 把阶段正式新增到事项下面

如何做：

- 支持在事项下创建阶段
- 支持查看阶段
- 支持编辑阶段
- 支持删除阶段
- 让阶段稳定挂在事项下，不做独立漂浮结构

最终效果：

- 系统开始具备“时间段现实”的表达能力
- 用户在事项里不只能看到零散记录，还能看到某个长期主题经历过哪些阶段

### 7. 理顺记录、事项、阶段的实际使用关系

做什么：

- 防止新增阶段后，系统再次变乱

如何做：

- 保持记录继续作为第一入口
- 保持事项负责长期主题承接
- 保持阶段负责表达某段时间的持续现实
- 在页面和操作流中，把三者分工明确拉开

最终效果：

- 用户清楚什么时候记记录，什么时候进事项，什么时候建立阶段
- 结构稳定，不会把阶段做成大号记录，也不会把事项做成大号任务

- 避免 1.4 做完后再次留下新的结构垃圾

## 十二、1.4 的阶段验收口径

1.4 验收不看“概念讲得多完整”，  
也不看“页面看起来多高级”，  
只看以下四件事是否真的成立：

### 1）旧污染是否先被清掉

如果旧页面、旧 API、旧 DB 操作层、旧类型残留还在持续污染结构，  
则 1.4 不能算稳。

### 2）事项是否真正能承接长期主题

用户进入事项后，  
是否真的能把它当成某个长期主题空间使用，  
而不是只看一个弱详情页。

### 3）历史、阶段、目标是否真正挂进主结构

不是“有这些词”，  
而是它们真的进入记录 / 事项 / 洞察主骨架，  
并能形成连续回看。

### 4）洞察是否真正开始理解长期现实

不是只显示数量，  
而是开始帮助用户理解：

- 哪些事项重要
- 哪些事项经历了阶段变化
- 哪些事项承接了历史到当前的连续性
- 哪些内容与目标方向有关

只要这四条真实成立，  
1.4 才算成立。
