# TETO 发展规划路线图 Spec（v3）

## 0. 文档定位

本文档基于代码库全面审计，结合多轮核心设计问题的讨论，形成 TETO 长期发展规划。

与 v2 版本的关键变化：
- **补充完整维度体系**：从 3 个维度扩展到 7 个维度
- **重新定义知识沉淀**：Note 支持记录级附着 + 间隔复习机制
- **澄清"第二大脑"概念**：TETO 是现实映射型数字分身，不是知识图谱型第二大脑
- **修正排名功能定位**：个人排名归入洞察模块增强，多人排名归入 3.0
- **新增离线/同步需求评估**
- **新增未覆盖风险识别**

# 第一部分：核心设计问题定论

## 1. 事项 vs 维度：二元分类体系

### 1.1 问题

情绪、财务、睡眠和英语、运动、考试是同一类东西吗？应该作为记录的属性还是独立的事项？

### 1.2 结论

**它们不是同一类东西。当前系统缺少一个根本性的分类区分。**

| | 事项 (Item) | 维度 (Dimension) |
|---|---|---|
| 本质 | 你**主动去做**的事 | 你做任何事时的**状态/背景** |
| 示例 | 英语、运动、考试、项目A | 情绪、精力、财务、时间 |
| 特征 | 有开始有结束，有推进有停滞 | 横切所有事项，持续存在 |
| 提问方式 | "你在做什么？" | "你做这件事时状态如何？" |
| 数据位置 | record.item_id | record.mood / energy / cost / duration |

### 1.3 完整维度体系

v2 只识别了情绪、财务、睡眠 3 个维度。基于对三层九组模型（L1-L3 全部字段）的系统审查，完整的维度体系应包含 **7 个维度**：

| # | 维度 | 数据字段 | 分类 | 说明 |
|---|------|---------|------|------|
| D1 | **情绪** | mood, energy, body_state | 纯维度 | 你做任何事时的内心状态和身体感受 |
| D2 | **财务** | cost, money_direction, money_currency | 纯维度 | 你做任何事时可能伴随的资金流动 |
| D3 | **时间** | duration_minutes, occurred_at, occurred_at_end, time_precision | 纯维度 | 你做任何事都消耗时间，这是最基础的公共度量轴 |
| D4 | **效率/产出** | status(运转状态), outcome_type, outcome_direction | 纯维度 | 你做任何事的结果如何——是推进了还是消耗了 |
| D5 | **社交** | people, relation_roles | 纯维度 | 你做任何事可能涉及其他人，社交环境影响表现 |
| D6 | **空间/环境** | location, place_type | 纯维度 | 你做任何事都在某个物理/虚拟空间中，环境影响状态 |
| D7 | **睡眠** | body_state(累/困/没精神), duration_minutes(睡眠时长) | 混合型 | 可作维度（身体状态影响一切），也可作事项（主动追踪睡眠质量） |

**为什么 D4 效率/产出是独立维度，而不是事项的属性？**

因为"事情做得怎么样"是横切所有事项的：同一件事（如英语）今天可能高效推进，明天可能低效消耗。outcome_type 和 outcome_direction 不是事项的固有属性，而是每次记录时的事后评价。它们的聚合（"英语事项上 72% 的记录是正向产出"）才是事项级别的属性。

**为什么 D5 社交和 D6 空间是独立维度？**

- 社交：你和同事开会 vs 你独处学习，同样的时长产出完全不同。社交环境是一个影响所有事项表现的正交因素。
- 空间：在家工作 vs 在公司工作 vs 在通勤路上，对效率和情绪的影响截然不同。place_type 已有完整的枚举值。

**维度-事项交叉分析的完整矩阵**：

```
          D1情绪  D2财务  D3时间  D4效率  D5社交  D6空间
英语      😊72%   ¥200   15h    ⚡正向   独处    🏠家
运动      😊89%   ¥0     6h     ⚡正向   独处    🌳户外
项目A     😐45%   ¥0     25h    ⚡中性   同事    🏢公司
社交活动  😊80%   ¥300   4h     ⚡正向   朋友    🍻店铺
```

### 1.4 对系统架构的影响

当前架构已经天然支持这个二元体系——只是没有显式地命名和利用它：

```
记录 (Record)
 ├── 事项归属：item_id → "这件事属于哪个主题"
 └── 维度属性：mood, energy, body_state, cost, duration_minutes → "这件事的状态如何"
```

**不需要改数据模型**。Record 已经同时拥有事项归属和维度属性。需要补的是：

1. **维度分析视图**：按维度（而非事项）聚合记录的分析面板
   - 情绪维度面板：聚合所有记录的 mood/energy/body_state
   - 财务维度面板：聚合所有记录的 cost/money_direction
   - 时间维度面板：聚合所有记录的 duration_minutes（部分已有）
   - 效率维度面板：聚合所有记录的 outcome_type/outcome_direction/status
   - 社交维度面板：聚合所有记录的 people/relation_roles
   - 空间维度面板：聚合所有记录的 location/place_type

2. **维度-事项交叉分析**：一个维度在一个事项上的表现
   - "你在英语事项上的情绪分布"
   - "你在运动事项上的时间投入"
   - "你在所有事项上的精力变化"
   - "你独处 vs 有人陪伴时的效率差异"
   - "你在不同地点的产出分布"

3. **允许维度成为事项**：用户可以把"睡眠管理"或"预算控制"作为事项来追踪，但系统不应默认这样做。维度和事项的转换是用户的选择，不是系统的强制。

### 1.5 设计原则

> **事项回答"你在做什么"，维度回答"你做得怎么样"。两者正交，不互相替代。**

## 2. 复盘 vs 知识沉淀：两种不同的能力

### 2.1 问题

"今天学了二元一次方程，写自己的解法总结或心得体会"——这属于复盘还是第二大脑？如何与现有系统关联？

### 2.2 结论

**这是知识沉淀，不是复盘。当前系统需要区分三种不同性质的内容：**

| | 复盘 (Review) | 知识沉淀 (Knowledge Note) | 想法 (Thought) |
|---|---|---|---|
| 关注点 | **你自己**的行为模式 | **主题**的理解和掌握 | **即时**的灵感和念头 |
| 示例 | "最近英语停滞是因为换了方法不适应" | "二元一次方程消元法的步骤和易错点" | "也许可以试试用Anki背单词" |
| 与事项的关系 | 回看一个或多个事项的整体表现 | 附着在某个事项/子项上的深度理解 | 可以属于事项，也可以是自由的 |
| 数据来源 | 系统已有记录+洞察的再分析 | 用户的主动思考，需要独立输入 | 用户的即时想法，轻量记录 |
| 已有支撑 | 洞察模块的部分能力 | **完全缺失** | 想法类型记录 |

### 2.3 知识沉淀的架构设计

知识沉淀需要在现有系统中增加一种新的实体类型：**Note（笔记）**

```
Note
 ├── id
 ├── user_id
 ├── item_id (可选) → 附着在哪个事项上
 ├── sub_item_id (可选) → 附着在哪个子项上
 ├── record_id (可选) → 附着在哪个记录上（记录级笔记）
 ├── title → 笔记标题（如"二元一次方程消元法"）
 ├── content → 富文本内容
 ├── note_type → 'knowledge' | 'review' | 'insight' | 'reflection'
 ├── source_record_ids (可选) → 从哪些记录衍生出来
 ├── tags
 ├──
 ├── ── 间隔复习字段（可选，仅 knowledge 类型） ──
 ├── review_enabled → boolean → 是否启用间隔复习
 ├── review_interval_days → number → 当前复习间隔（天）
 ├── review_next_date → string → 下次复习日期
 ├── review_count → number → 已复习次数
 ├── review_ease_factor → number → 难度因子（SM-2 算法）
 ├── last_reviewed_at → string → 上次复习时间
 ├──
 ├── created_at
 └── updated_at
```

关键设计决策：

1. **Note 总是有上下文的**：它不是孤立的笔记，而是附着在某个事项/子项/记录上的理解沉淀。这与"第二大脑"式自由笔记系统本质不同。

2. **Note 支持记录级附着**：用户可以在任何一条记录上直接添加笔记，形成"这条记录 + 我对这条记录的理解"的绑定关系。例如：记录"今天学了二元一次方程" → 笔记"消元法的步骤和易错点"。

3. **Note 可以从记录触发**：用户看到一条记录，可以点击"添加笔记"来写深度理解。这条笔记自动关联到该记录和所属事项。

4. **Note 不取代记录**：记录是"发生了什么"的事实捕捉，Note 是"我理解了什么"的知识沉淀。两者共存于事项下。

5. **Note 可搜索可回看**：在事项页中，Note 和 Record 并列展示，但用不同样式区分。Record 是时间线视图，Note 是知识库视图。

6. **Note 支持间隔复习**（仅 knowledge 类型）：用户可以选择对某条笔记启用间隔复习。系统基于改良的 SM-2 算法自动计算下次复习时间，并在到达时提醒用户回顾。

#### 2.3.1 间隔复习机制

**设计原则**：

- 间隔复习是**可选**功能，不强制。用户可以选择对哪些笔记启用。
- 复习的触发是"提示你回看"，不是"强制你做题"。点击复习后看到的是笔记原文，用户自己判断是否记得。
- 复习反馈只有三档：**记得 / 模糊 / 忘了**，对应不同的间隔调整策略。

**算法（简化版 SM-2）**：

```
初始间隔：
  第1次复习：1天后
  第2次复习：3天后
  第3次复习：7天后
  第4次复习：14天后
  第5次复习：30天后
  之后按 ease_factor 递增

反馈调整：
  "记得" → interval × ease_factor（默认 2.2）
  "模糊" → interval × 1.3，ease_factor 不变
  "忘了" → interval 重置为 1 天，ease_factor 降低

复习队列：
  每日检查 review_next_date <= 今天 的笔记
  在洞察页或首页展示"今日待复习笔记"入口
```

**与其他系统的区别**：

- Anki/超级记忆：以卡片为核心，做题式复习，正反面翻转
- TETO 的间隔复习：以笔记为核心，回看式复习，自然语言内容。不做翻转，不做做题，只是"提醒你回看一下这条笔记"

**复习与复盘的关系**：

- 间隔复习 = 微观层面，针对单条知识笔记的遗忘对抗
- 周期复盘 = 宏观层面，针对整体行为模式的结构化回看
- 两者互补但不重叠：复习帮你记住知识，复盘帮你看清行为

### 2.4 复盘的架构设计

复盘不需要新实体类型，而是基于已有数据的结构化生成：

```
Review（不是一个表，而是基于已有数据的计算结果）
 ├── 期间范围 → 本周/本月/自定义
 ├── 事项维度的变化 → 活跃→停滞、目标进度变化
 ├── 维度维度的变化 → 情绪趋势、支出变化、时间分配变化
 ├── 关联分析 → 维度变化是否与事项变化相关
 └── 结构化输出 → 事实总结 + 趋势 + 异常标记
```

复盘结果可以保存为一种特殊的 Note（类型为 `review`），这样可以回看历史复盘，但不引入新的存储结构。

### 2.5 "第二大脑"概念澄清

**用户的直觉是对的：TETO 本身就是一种第二大脑。** 但它与 Obsidian 类工具的"第二大脑"有本质区别。

#### 2.5.1 两种"第二大脑"

| | TETO（现实映射型数字分身） | Obsidian（知识图谱型第二大脑） |
|---|---|---|
| **起点** | 发生了什么（事件/事实） | 想到了什么（想法/概念） |
| **核心数据** | 记录（事实流） | 笔记（知识节点） |
| **组织方式** | 事项→记录→洞察（树状） | 笔记↔笔记（网状双向链接） |
| **分析能力** | 统计、对比、趋势、预测 | 关系发现、图谱遍历 |
| **核心问题** | "我最近的生活是怎样的？" | "这个概念和那个概念有什么关系？" |
| **时间性** | 强时间性（每条记录有精确时间） | 弱时间性（笔记是"知识"，不是"事件"） |
| **用户画像** | 想看清自己现实状况的人 | 想构建知识体系的人 |

#### 2.5.2 为什么 TETO 是"数字分身"而非"知识图谱"

TETO 的核心价值不在于帮你组织知识，而在于**帮你映射现实**。它是一面镜子：

- 你输入的是每天真实发生的事 → 镜子照出你的现实
- 系统整理成事项和洞察 → 镜子帮你看到你平时看不到的结构
- 你基于洞察调整行动 → 你改变了，镜中的映像也随之变化

Obsidian 的核心价值在于**帮你连接思想**。它是一个图谱：

- 你输入的是想法和概念 → 图谱增加节点
- 系统通过链接发现关联 → 图谱帮你看到概念间的关系
- 你基于关联产生新理解 → 知识网络扩展

**两者不是替代关系，而是互补关系。** 一个人可能同时需要：
- TETO：帮我记住每天发生了什么、看清行为模式、追踪目标进度
- Obsidian：帮我整理学到的知识、建立概念间的联系

#### 2.5.3 TETO 知识沉淀的边界

TETO 的知识沉淀（Note）是**事项级**的，不是**全局级**的：

- ✅ "我在学英语时积累的单词记忆法" → 附着在"英语"事项下，TETO 管得了
- ✅ "今天学了这个算法，我的理解是..." → 附着在"算法学习"事项下，TETO 管得了
- ❌ "贝叶斯定理和频率学派的哲学分歧" → 这是一个独立的知识话题，不属于某个事项，TETO 管不了
- ❌ "经济学原理的思维导图" → 这是一个知识体系，不是现实映射，TETO 管不了

**如果用户需要管理独立于事项的知识网络，应使用 Obsidian 等工具。**

#### 2.5.4 Spec v2 中"第二大脑"措辞的修正

v2 中说"TETO 不做第二大脑"，这个表述不够准确。更准确的表述是：

> **TETO 不做知识图谱型第二大脑。TETO 是现实映射型数字分身。它帮你看清"你最近的现实是怎样的"，而不是帮你看清"你的知识网络是什么样的"。**

## 3. 排名与 3.0 协同设计

### 3.1 排名功能的重新定位

**用户的核心观点**：个人排名已在洞察模块中逐步实现（ItemActivityPanel 的 time_ranking、GoalProgressPanel 的进度排名），不需要作为独立功能。2.0 应该实现的是**多人排名**（竞赛排行榜），让多个用户在共享空间内比较进度和贡献。

**修正后的排名设计**：

#### 3.1.1 个人排名 → 洞察模块增强（1.7）

个人排名本质上是洞察模块的排序和对比能力，应自然融入现有洞察面板：

- 事项投入排名 → ItemActivityPanel 增强（已有 time_ranking，增加更多排序维度）
- 目标完成率排名 → GoalProgressPanel 增强（增加排序和对比视图）
- 维度表现排名 → 新增维度面板的排序功能
- 自我对比排名 → PeriodComparisonPanel 增强

这些不需要独立的"排名页面"，而是洞察面板内的排序/对比交互增强。

#### 3.1.2 多人排名 → 3.0 多用户协同功能

多人排名需要在多用户架构基础上实现：

- **竞赛排行榜**：共享空间内的成员排名（目标完成率、投入时长、记录频率等）
- **维度排名**：谁在某事项上投入最多、谁的目标完成率最高
- **贡献度排名**：团队内每个成员的贡献量化

多人排名的伦理约束：
- 排名是**激励工具**，不是**考核工具**
- 排名数据来源受四级可见度模型约束
- 成员可以退出排名展示
- 排名维度由组织管理员配置，不是系统强制

### 3.2 3.0 的核心设计原则

> **协同，不是监控；透明，不是侵犯；贡献可见，但隐私可控。**

3.0 不是把个人数据开放给别人看。它的核心是**共享事项下的协同现实系统**。

#### 3.2.1 核心问题

3.0 应该能回答：

- 我们共同的目标是什么？
- 每个人负责什么？
- 每个人做了哪些相关行动？
- 哪些行动有证据？
- 哪些事情卡住了？
- 团队整体是否在靠近目标？
- 团队/家庭资源、时间、责任是否分配合理？

#### 3.2.2 数据共享的四级模型

```
Level 0：私密 (Private)
  → 只有自己可见
  → 默认状态

Level 1：统计可见 (Stats-visible)
  → 别人只能看到聚合统计
  → 如："本周英语投入 5 小时"、"目标完成率 60%"
  → 看不到具体记录内容

Level 2：摘要可见 (Summary-visible)
  → 别人可以看到记录摘要
  → 如："今天背了30个单词"
  → 看不到情绪、财务等维度属性

Level 3：全文可见 (Full-visible)
  → 别人可以看到完整记录
  → 需要用户主动设置
  → 适用于团队工作记录等场景
```

**默认是 Level 0，用户必须主动选择提升可见度。**

#### 3.2.3 团队版 vs 家庭版

**团队版能看到**：
- 成员任务推进情况
- 成员贡献记录
- 事项进展
- 目标完成率
- 阻塞点
- 协作记录
- 复盘结论

**家庭版能看到**：
- 家庭共同事项
- 家庭支出
- 家务分担
- 孩子成长记录
- 家庭目标
- 共同计划
- 情绪和负担分布

**两者的共享逻辑不同**：
- 团队版：以共享事项为核心，围绕事项共享行动和进度
- 家庭版：以共享生活空间为核心，围绕生活场景共享状态和责任

#### 3.2.4 权限控制

```typescript
// 共享事项的权限模型
interface SharedItem {
  item_id: string;
  organization_id: string;

  // 成员在该事项上的角色
  members: {
    user_id: string;
    role: 'owner' | 'collaborator' | 'observer';
    visibility_level: 0 | 1 | 2 | 3;  // 对其他成员的可见度
  }[];

  // 事项级别的共享设置
  default_visibility: 0 | 1 | 2 | 3;

  // 记录级别的精细控制
  record_visibility_override: {
    record_id: string;
    visibility: 0 | 1 | 2 | 3;
  }[];
}
```

#### 3.2.5 3.0 的架构策略

不做全面重构，而是**在现有单用户架构上加一层共享层**：

```
现有层（不动）：
  records, items, goals, phases, tags → user_id 隔离

新增层（共享）：
  organizations → 团队/家庭
  organization_members → 成员关系和角色
  shared_items → 哪些事项被共享到哪个组织
  shared_records → 哪些记录被共享（及可见度级别）
  organization_goals → 组织级别的共享目标
```

这种"外挂式"架构的优势：
- 现有单用户功能完全不受影响
- 共享是可选的、渐进的
- 可以独立开发和测试
- 回退风险低

## 5. 各版本详细定义

### 5.1 TETO 1.5：稳定核心四模块 + 规则中心 + 计算中心

**目标**：确保记录·事项·洞察·目标四大模块的基础链路可信可走通，同时建立系统级规则中心和计算中心

**前置条件**：1.4 已完成

**范围**：按现有 1.5 执行清单 8 个块执行（不再赘述），另新增以下两个中心：

#### 5.1.1 规则中心（Rule Center）

**定义**：规则中心是系统内部默认规则的集中管理单元。规则由开发时写死/封装好，不暴露给用户编辑，但确保所有模块的行为一致。

**为什么需要规则中心**：

当前系统的规则散落在各处：
- AI 解析的 prompt 写在 `parse-semantic.ts` 中（复合句拆分规则、字段边界规则）
- 归类逻辑在 `enhance-record.ts` 的 `matchItemSmart` 中（关键词匹配规则）
- 记录类型归一化在 `teto.ts` 的 `normalizeRecordType` 中（情绪/花费/结果 → 发生）
- 模糊输入检测在 `optimize-input/route.ts` 中
- 歧义检测在 `ClarificationNeeded` 类型定义中

这些规则没有统一入口，改一个规则要在多个文件中找。规则中心把它们集中管理。

**规则中心的结构**：

```typescript
// src/lib/rules/index.ts — 规则中心入口

export const RULES = {
  // === 记录类型规则 ===
  record_type: {
    // 4 种主类型的定义
    primary_types: ['发生', '计划', '想法', '总结'],
    // 旧类型归一化映射
    type_normalization: { '情绪': '发生', '花费': '发生', '结果': '发生' },
  },

  // === 解析规则 ===
  parsing: {
    // 复合句拆分规则
    compound_split: {
      split_on: ['不同动作', '不同事项', '不同时间段', '不同统计对象'],
      no_split_on: ['同一行为补充说明', '情绪评价附着', '效果说明附着'],
    },
    // 模糊输入三分法
    fuzzy_classification: {
      A: '无法理解 → 要求澄清',
      B: '可理解但信息不足 → 允许低精度落地或补关键字段',
      C: '结构不合理 → 提示拆分/改写',
    },
    // 时间精度层级
    time_precision: ['exact', 'approx', 'fuzzy', 'unknown'],
    // action_text 长度限制
    action_text_max_length: 4,
  },

  // === 归类规则 ===
  classification: {
    // 事项匹配置信度阈值
    auto_classify_threshold: 0.8,    // 高于此自动归类
    suggest_classify_threshold: 0.5, // 高于此给候选确认
    // 低于此禁止自动归类
    // 子项归属歧义处理
    sub_item_ambiguous_action: 'mark_for_clarification',
  },

  // === 状态流转规则 ===
  lifecycle: {
    // 记录生命周期
    record_statuses: ['active', 'completed', 'postponed', 'cancelled'],
    // 事项状态流转
    item_statuses: ['活跃', '推进中', '放缓', '停滞', '已完成', '已搁置'],
    // 目标状态流转
    goal_statuses: ['草稿', '进行中', '已完成', '暂停', '放弃'],
    // 停滞检测天数
    stagnation_threshold_days: 14,
  },

  // === AI 降级规则 ===
  fallback: {
    // LLM 不可用时的行为
    on_llm_unavailable: 'use_parseNaturalInput',
    // AI 回写策略
    ai_write_policy: 'fill_empty_only',  // 只填空，不覆盖用户值
  },
} as const;
```

**规则中心的使用方式**：

```
1. 所有模块从 RULES 对象读取规则，不硬编码在业务逻辑中
2. 修改规则只改一个地方，所有模块同步生效
3. 规则变更需要记录到 ai_traces（可观测性）
4. 后续版本可开放部分规则给用户自定义（1.6+）
```

**规则中心不是用户功能**：它是一个开发侧的基础设施，让系统行为可预测、可调试、可统一修改。

#### 5.1.2 计算中心（Computation Center）

**定义**：计算中心是系统内部统计口径的统一定义单元。所有模块的统计计算都引用计算中心的口径定义，确保不同页面、不同模块看到的数据一致。

**为什么需要计算中心**：

1.5 项目说明文档中的技术债务明确指出："统计口径：无统一定义层，各处计算可能不一致"。当前的问题：

- `insights.ts` 中的活跃事项计数逻辑和 `items.ts` 中的可能不同
- `goal-engine.ts` 的日均计算和洞察面板的日均计算可能用不同的时间窗口
- duration_minutes 的聚合：有些地方按 record_count，有些按 duration_sum，没有统一

**计算中心的结构**：

```typescript
// src/lib/computation/index.ts — 计算中心入口

export const COMPUTATION = {
  // === 核心指标定义 ===
  metrics: {
    // 活跃度
    activity: {
      definition: '指定时间范围内有记录的事项数量',
      formula: 'COUNT(DISTINCT item_id) WHERE records.occurred_at IN range',
      unit: '个事项',
    },
    // 投入量
    investment: {
      definition: '指定时间范围内某事项的总投入时长',
      formula: 'SUM(duration_minutes) WHERE item_id = X AND occurred_at IN range',
      unit: '分钟',
      note: '不含 inferred 数据（data_nature = fact only）',
    },
    // 记录频率
    record_frequency: {
      definition: '指定时间范围内某事项的记录条数',
      formula: 'COUNT(*) WHERE item_id = X AND occurred_at IN range',
      unit: '条',
      note: '不含 lifecycle_status = cancelled',
    },
    // 目标完成率
    goal_completion_rate: {
      definition: '由 goal-engine 计算，口径随 rule_type 不同',
      formula: '见 goal-engine.ts 按规则类型分别定义',
      unit: '%',
    },
    // 停滞天数
    stagnation_days: {
      definition: '事项最近一条记录距今天的天数',
      formula: 'CURRENT_DATE - MAX(occurred_at::date) WHERE item_id = X',
      threshold: RULES.lifecycle.stagnation_threshold_days, // 引用规则中心
      unit: '天',
    },
    // 支出金额
    expense: {
      definition: '指定时间范围内的总支出',
      formula: 'SUM(cost) WHERE money_direction = "expense" AND occurred_at IN range',
      unit: 'CNY',
      note: 'cost 为 NULL 的记录不参与计算',
    },
    // 收入金额
    income: {
      definition: '指定时间范围内的总收入',
      formula: 'SUM(cost) WHERE money_direction = "income" AND occurred_at IN range',
      unit: 'CNY',
    },
  },

  // === 时间窗口定义 ===
  time_windows: {
    recent_7d: { label: '近 7 天', offset_days: 7 },
    recent_30d: { label: '近 30 天', offset_days: 30 },
    this_week: { label: '本周', start: 'monday_of_current_week' },
    this_month: { label: '本月', start: 'first_of_current_month' },
    last_week: { label: '上周', start: 'monday_of_last_week', end: 'sunday_of_last_week' },
    last_month: { label: '上月', start: 'first_of_last_month', end: 'last_of_last_month' },
  },

  // === 数据过滤口径 ===
  data_scope: {
    // 默认只统计事实数据
    default_nature: 'fact' as const,
    // 哪些统计包含推断数据
    include_inferred: ['activity_heatmap', 'time_distribution'],
    // 哪些统计排除推断数据
    exclude_inferred: ['goal_completion_rate', 'expense', 'income'],
    // lifecycle_status 排除规则
    excluded_lifecycle: ['cancelled'],
  },

  // === 对比口径 ===
  comparison: {
    // 固定对比对
    pairs: [
      { label: '周变化', current: 'this_week', previous: 'last_week', scope: 'week' },
      { label: '月变化', current: 'this_month', previous: 'last_month', scope: 'month' },
    ],
  },
} as const;
```

**计算中心的使用方式**：

```
1. insights.ts、goal-engine.ts、items.ts 等所有统计代码引用 COMPUTATION.metrics
2. 新增维度面板时，口径从计算中心读取，不自行定义
3. 发现口径不一致时，改计算中心一个地方，所有模块同步
4. 对外展示的统计数字必须有对应的 COMPUTATION 定义可追溯
```

**验收标准**：
- 8 条验收链路全部走通（见 1.5 执行清单第 16 节）
- 规则中心 RULES 对象覆盖：记录类型、解析、归类、状态流转、降级 5 个类别

- 排名
- 复盘
- 知识沉淀

#### 5.2.4 可观测性体系（决策 ID 追踪 + 日志系统）

**问题**：AI 解析链路是黑盒。用户说"归类错了"，无法定位是哪个环节出的问题。全项目只有 65 处 console.log/error/warn，无结构化日志，无 trace ID，无决策追踪。

**解决方案**：

##### ai_traces 表

```sql
CREATE TABLE ai_traces (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id) NOT NULL,
  record_id UUID REFERENCES records(id),

  -- 链路标识
  trace_id UUID NOT NULL,           -- 一次完整增强链路的唯一 ID
  step_number INT NOT NULL,         -- 步骤编号
  step_name TEXT NOT NULL,           -- 步骤名称

  -- 输入输出
  input JSONB,                       -- 该步骤的输入
  output JSONB,                      -- 该步骤的输出
  success BOOLEAN DEFAULT true,      -- 是否成功

  -- 决策信息
  decision_id UUID,                  -- 该步骤内关键决策的 ID
  decision_type TEXT,                -- 决策类型（item_match, sub_item_assign, field_fill, ...）
  confidence FLOAT,                  -- 置信度
  was_corrected BOOLEAN DEFAULT false, -- 用户是否修正了此决策

  -- 性能
  duration_ms INT,                   -- 该步骤耗时
  tokens_used INT,                   -- LLM token 消耗

  created_at TIMESTAMPTZ DEFAULT now()
);
```

##### AI 增强链路追踪

```
每次 AI 增强启动时生成 trace_id，每个步骤写入 ai_traces：
  step 1: optimize-input（模糊输入检测）
  step 2: parse-semantic（DeepSeek 解析）
  step 3: compound-split（复合句拆分）
  step 4: item-match（事项匹配）→ 记录 decision_id + confidence
  step 5: sub-item-assign（子项归属）→ 记录 decision_id + confidence
  step 6: field-fill（字段回写）→ 记录每个字段的来源
  step 7: clarification（歧义检测）

用户修正时：查找对应的 decision_id → 标记 was_corrected = true → 写入 user_rules
```

##### 结构化日志

```
替代 console.log，使用 pino 输出 JSON 格式日志：
  { trace_id, user_id, step, duration_ms, success, ... }

出问题时直接查 ai_traces 表定位断点，不用重新扫描全部数据浪费 token。
```

##### 北极星指标：归类修正率

```sql
-- 按 decision_type 统计修正率
SELECT decision_type,
  COUNT(*) AS total,
  SUM(CASE WHEN was_corrected THEN 1 ELSE 0 END) AS corrected,
  ROUND(SUM(CASE WHEN was_corrected THEN 1 ELSE 0 END)::numeric / COUNT(*) * 100, 1) AS correction_rate
FROM ai_traces
WHERE user_id = $1 AND created_at >= NOW() - INTERVAL '30 days'
GROUP BY decision_type
ORDER BY correction_rate DESC;
```

**"你最近为什么迷茫？"**

#### 5.3.7 Note 实体与知识沉淀

```typescript
interface Note {
  id: string;
  user_id: string;

  // 上下文归属（至少一项非空）
  item_id: string | null;       // 附着的事项
  sub_item_id: string | null;   // 附着的子项
  record_id: string | null;     // 附着的记录（记录级笔记）

  // 内容
  title: string;                // 笔记标题
  content: string;              // 内容（markdown）
  note_type: 'knowledge' | 'review' | 'insight'; // 笔记类型

  // 间隔复习（SM-2 算法）
  repetition: {
    ease_factor: number;        // 难度因子（初始 2.5）
    interval_days: number;      // 复习间隔天数
    repetitions: number;        // 已完成复习次数
    next_review_date: string;   // 下次复习日期
    last_review_date: string;   // 上次复习日期
  } | null;                     // review 类型不启用间隔复习

  // 关联
  source_record_ids: string[];  // 从哪些记录衍生
  tags: string[];               // 标签

  // 元信息
  created_at: string;
  updated_at: string;
}
```

**Note 实体设计要点**：
- `record_id` 字段允许笔记直接附着在记录上，适合"这条记录让我想到..."的场景
- `note_type` 区分三种用途：knowledge（学习笔记）、review（复盘结果）、insight（洞察沉淀）
- 间隔复习仅对 `knowledge` 类型启用——复盘和洞察不需要遗忘曲线
- SM-2 算法实现参考 Anki 的简化版，核心逻辑约 50 行

**Note 实体边界**：
- 笔记**必须**附着在事项/子项/记录上，不能创建无归属的笔记
- 笔记之间**不支持**双向链接（不做网状知识图谱）
- 笔记**不支持**模板系统（不做结构化知识框架）
- 笔记的**核心场景**是"我在做这件事时积累的理解"，不是"我发现这个概念和那个概念有关系"
- 如果用户的知识管理需求超出这个边界，应使用专门工具

**核心交互**：
1. 从记录创建笔记：在记录详情页点击"添加笔记"，自动关联该记录和所属事项
2. 从事项页创建笔记：在事项详情页的"笔记"区域直接创建
3. 笔记列表与搜索：事项页内展示所有笔记，支持关键词搜索
4. 笔记与记录的关联：笔记可以引用多条记录作为来源
5. 间隔复习提醒：到期笔记在洞察页/首页显示"待复习"标记

#### 5.3.8 缓存与离线输入队列

**性能缓存**：

复盘和笔记功能显著增加了数据查询复杂度，需要引入查询结果缓存：

```typescript
// src/lib/cache/index.ts
export const CACHE_CONFIG = {
  // 洞察数据缓存（计算后缓存，定期刷新）
  insights: { ttl_minutes: 15, stale_while_revalidate: true },
  // 事项排名缓存
  rankings: { ttl_minutes: 30, stale_while_revalidate: true },
  // 维度面板缓存
  dimension_panels: { ttl_minutes: 10, stale_while_revalidate: true },
  // 投入产出分析缓存
  investment_output: { ttl_minutes: 30, stale_while_revalidate: true },
} as const;
```

实现策略：
- 使用 TanStack Query 的 `staleTime` + `gcTime` 作为客户端缓存层
- 服务端热点查询考虑 Supabase RPC + 缓存中间件
- 缓存失效策略：写入时主动失效（`queryClient.invalidateQueries`）
- `stale_while_revalidate`：先返回缓存数据，后台刷新，用户无感知等待

**离线输入队列**：

```
1. 检测网络状态（navigator.onLine + online/offline 事件）
2. 在线时：正常提交到 /api/v2/records
3. 离线时：
   a. 将记录写入 IndexedDB（待同步队列，使用 Dexie.js）
   b. UI 显示"离线模式"标记
   c. 记录在本地列表中可见（带"待同步"标记）
4. 恢复在线时：
   a. 从 IndexedDB 读取待同步队列
   b. 按时间顺序批量提交到 /api/v2/records/batch
   c. 提交成功后从 IndexedDB 删除
   d. 失败的记录保留在队列中，下次重试
5. 冲突处理：以云端数据为准（Last-Write-Wins）
```

技术依赖：
- **Dexie.js**：IndexedDB 的 Promise 封装，比原生 API 好用得多
- **Serwist**（原 Workbox）：Service Worker 生成，管理离线缓存策略
- 两者在 1.6 版本的 TanStack Query 基础上自然扩展

**验收标准**：
- 事项排名至少支持 2 种排名维度
- 投入产出四象限可生成
- "纯消耗"事项可被识别
- 排名结果可追溯到具体记录
- 周期复盘可生成，内容有事实依据
- 事项复盘可从事项页触发
- 两个诊断型问题可通过数据结构化回答
- Note 实体可创建、编辑、删除，支持 record_id 关联
- 间隔复习对 knowledge 类型笔记可正常计算下次复习日期
- 洞察面板加载时间 < 2s（缓存命中时 < 500ms）
- 离线时可录入记录，恢复在线后自动同步
- 所有复盘输出和笔记可追溯到原始记录

### 5.6 TETO 3.0：多用户协同

**目标**：在个人版完整闭环的基础上，增加团队/家庭协同能力，构建"外挂式共享层"

**前置条件**：2.x 验收通过（多用户独立使用已验证）

**核心原则**：协同不是监控，透明不是侵犯，贡献可见但隐私可控

#### 5.6.1 架构策略

采用"外挂式"共享层，不动现有单用户架构：

```
现有层（不动）：
  records, items, goals, phases, tags, notes
  → user_id 隔离，所有现有功能不受影响

新增层（共享）：
  organizations
    → id, name, type('team'|'family'), created_at

  organization_members
    → organization_id, user_id, role('owner'|'admin'|'member'|'observer')

  shared_items
    → item_id, organization_id, default_visibility(0-3)

  shared_records
    → record_id, organization_id, visibility(0-3)

  organization_goals
    → goal_id, organization_id, shared_by, visibility(0-3)
```

**架构关键约束**：
- 现有单用户功能零改动——3.0 上线后，不加入组织的用户感知不到任何变化
- 共享层是"叠加"而非"修改"——所有共享通过 shared_* 表实现，不修改原表的 RLS 策略
- 查询路径：单用户数据走原路径（records WHERE user_id = ?），共享数据走共享路径（shared_records JOIN records）

#### 5.6.2 四级可见度模型

```
Level 0：私密 → 只有自己可见（默认）
Level 1：统计可见 → 别人只看到聚合数字（如"本周投入 12 小时"）
Level 2：摘要可见 → 别人看到记录摘要，不含维度属性（如"做了英语阅读练习"）
Level 3：全文可见 → 别人看到完整记录，含维度属性（需主动设置）
```

**可见度设置粒度**：
- 事项级默认可见度（shared_items.default_visibility）
- 单条记录可覆盖事项默认值（shared_records.visibility）
- 批量调整：可将某事项下所有记录提升/降低一个级别

**Supabase RLS 实现策略**：

```sql
-- 共享记录的 RLS 示例（核心难点）
CREATE POLICY "view_shared_records" ON records
  FOR SELECT USING (
    -- 基础：自己的记录
    user_id = auth.uid()
    OR
    -- 共享：通过 shared_records 表，且可见度满足条件
    EXISTS (
      SELECT 1 FROM shared_records sr
      JOIN organization_members om ON om.organization_id = sr.organization_id
      WHERE sr.record_id = records.id
        AND om.user_id = auth.uid()
        AND sr.visibility >= 1  -- 根据查询需求过滤级别
    )
  );
```

> **风险提示**：四级可见度在 RLS 层的实现是 3.0 最大的技术风险。每条查询都需要 JOIN 多张表，可能导致性能问题。缓解方案：共享查询走专用 RPC 函数（带缓存），不依赖通用 RLS 策略。

#### 5.6.3 组织管理

```
创建组织：
  - 任何用户可创建组织（team 或 family 类型）
  - 创建者自动成为 owner

邀请成员：
  - owner/admin 可生成邀请链接（有效期 7 天）
  - 被邀请者点击链接 → 加入组织
  - 邀请时可预设默认角色

角色与权限：
  owner:  删除组织、转让 owner、管理成员、所有 admin 权限
  admin:  邀请/移除成员、管理共享事项、调整可见度
  member: 创建共享记录、查看共享数据、参与复盘
  observer: 仅查看共享数据，不可创建记录

退出与解散：
  - 成员可随时退出组织（个人数据不受影响）
  - owner 转让后可退出
  - 组织无成员时自动解散（共享数据保留 30 天后清除）
```

#### 5.6.4 共享事项与协作

```
共享事项：
  - 任何成员可将个人事项"共享"到组织
  - 共享时设置默认可见度（Level 0-3）
  - 共享后，其他成员可在组织视图看到（受可见度控制）
  - 共享事项可"取消共享"，已共享的记录保持已共享状态

任务分配：
  - 事项下的子项可指定负责人（organization_members.user_id）
  - 负责人变更记录在共享事项的活动流中
  - 不做"任务分配"功能——子项归属本身就是任务分配

贡献记录：
  - 每个成员在共享事项上的行动自动汇总
  - 汇总内容受可见度控制（Level 1+ 可看统计，Level 2+ 可看摘要）
  - 不做"工作量统计排名"作为考核依据

阻塞标记：
  - 成员可标记事项/子项为"阻塞"
  - 阻塞标记附带说明和可选的阻塞原因
  - 阻塞事项在团队视图高亮显示
```

#### 5.6.5 团队视图与多人排名

```
团队视图：
  - 组织级别的概览面板
  - 显示共享事项的整体进度
  - 显示成员贡献摘要（受可见度控制）
  - 不做"成员活动监控"功能

多人排名（3.0 核心新增）：
  - 排名范围：仅限共享事项内的数据
  - 排名维度：
    1. 共享事项的时间投入排名
    2. 共享事项的目标完成率排名
    3. 贡献度排名（记录量 + 时间投入加权）
  - 排名可见度：
    - Level 1+ 数据参与排名计算，但排名结果所有人可见
    - 排名是"谁在这个共享事项上投入最多"，不是"谁工作最多"
  - 排名边界：
    - 不做全组织排名（只在共享事项范围内）
    - 不做跨事项对比排名
    - 排名是自我认知工具，不是管理工具

团队复盘：
  - 基于共享数据的集体复盘
  - 复盘结果保存为组织级 Note
  - 复盘内容只包含共享事项的数据
  - 不包含任何成员的私密数据
```

#### 5.6.6 家庭版功能

```
家庭共同事项：家务、采购、维修等
家庭财务：共享支出、预算追踪
家务分担：可视化的家务分配和完成情况
成长记录：孩子的学习和成长（可控可见度）
家庭目标：共同的目标和计划
负担分布：情绪和精力在家庭成员间的分布（需 Level 2+ 可见度）
```

#### 5.6.7 3.0 开发前置

在正式开发 3.0 之前，需要：
1. **架构 POC**：验证共享层与现有架构的兼容性，特别是 RLS 性能
2. **权限模型设计文档**：细化四级可见度的实现细节，特别是边缘 case
3. **数据迁移策略**：现有单用户数据如何纳入组织结构
4. **伦理审查**：确保系统设计不会助长监控行为
5. **隐私影响评估**：3.0 共享功能上线前的强制性评估
6. **性能基线**：共享查询的性能基准，确保 JOIN 不导致查询超时

**验收标准**：
- 组织可创建、成员可邀请加入
- 共享事项在组织视图可见（受可见度控制）
- 四级可见度模型可正常工作
- 多人排名仅基于共享事项数据
- 个人数据在未共享时对组织内其他人完全不可见
- RLS 策略通过安全审计
- 共享查询性能 < 3s（1000 条记录 + 5 成员场景）

# 第三部分：风险与约束

## 6. 关键风险

### 6.1 功能膨胀

**缓解**：每个版本通过"三条验证"才能开始新功能：
1. 这个功能是否直接服务于当前版本的核心目标？
2. 去掉这个功能，版本目标是否仍然成立？
3. 这个功能是否可以在后续版本做而不影响当前版本？

### 6.2 洞察越界

**缓解**：严格区分三种输出类型：
- **观察型**：事实、趋势、对比、关联、异常 → 2.0 之前
- **诊断型**：基于数据的多维分析、异常排序 → 1.7 开始
- **建议型**：你应该、你必须 → **始终不做**

### 6.3 知识沉淀膨胀为第二大脑

**缓解**：
- 笔记必须附着在事项/子项上
- 不做笔记间的双向链接
- 不做独立的知识图谱
- 如果用户需要网状知识管理，推荐使用专门工具

### 6.4 3.0 变成监控系统

**缓解**：
- 默认 Level 0（私密），用户必须主动选择提升可见度
- 不做"成员活动监控"功能，只做"共享事项的进度追踪"
- 不做"实时在线状态"
- 不做"工作量统计排名"作为考核依据
- 排名是自我认知工具，不是管理工具

### 6.5 1.5 长期不闭环

**缓解**：
- 1.5 验收标准是"核心链路可走通"，不是"所有边界情况完美处理"
- 允许遗留已知问题，记录到 debt 列表
- 每条验收链路通过即标记完成，全过即进 1.6

## 8. 总体设计原则

### 8.1 事项-维度二元论

> 事项回答"你在做什么"，维度回答"你做得怎么样"。两者正交，不互相替代。

### 8.2 数据驱动而非建议驱动

> 系统告诉你"是什么"和"怎么样"，不告诉你"你应该"。用户自己判断和决策。

### 8.3 渐进式而非跳跃式

> 每个版本只做一件事，做完验证再进入下一个。2.0 是验证节点，不是开发节点。

### 8.4 协同而非监控

> 3.0 的共享是"我们一起看进度"，不是"我看着你做了什么"。默认私密，主动共享。

### 8.5 知识附着而非知识自由

> 笔记属于事项，不属于虚空。TETO 管的是"你在做事时积累的理解"，不是"你所有的知识"。

# 第四部分：补充评估

## 9. 离线功能与数据同步

### 9.1 需求分析

用户希望在早期版本中加入：
1. 页面缓存
2. 本地记录功能
3. 本地数据与云端（Supabase）的协同同步

### 9.2 当前架构的限制

| 层级 | 现状 | 离线能力 |
|------|------|---------|
| 渲染层 | Next.js App Router (Server Components) | 服务端渲染依赖网络 |
| 数据层 | Supabase (PostgreSQL + RLS) | 纯云端数据库，无本地缓存 |
| API 层 | Server Actions / REST API | 需要服务端运行 |
| 客户端 | React Client Components | 有状态但无持久化 |

当前架构是完全"在线优先"的——断网即不可用。

### 9.3 技术实现方案与复杂度

| 方案 | 描述 | 复杂度 | 价值 |
|------|------|--------|------|
| **A. 简单页面缓存** | Next.js ISR/SSG + Service Worker 缓存静态资源 | 低 | 有限。只能缓存页面外壳，动态数据仍需网络 |
| **B. 离线输入队列** | IndexedDB 缓存待提交记录，联网后批量同步 | 中 | 高。确保断网时也能录入，不丢数据 |
| **C. 只读离线缓存** | Service Worker + IndexedDB 缓存近期记录/事项 | 中高 | 中。离线可看历史数据但不能操作 |
| **D. 本地优先架构** | 本地 IndexedDB 为主库，Supabase 为同步目标 | 极高 | 极高。但需要重写整个数据层 |

### 9.4 推荐方案

**采用方案 B（离线输入队列），在 1.7 版本中实现。**

理由：
- 方案 A 价值太低，不解决核心痛点（断网时无法录入）
- 方案 C 是 B 的自然扩展，可以在 B 之后考虑
- 方案 D 需要重写数据层，成本远超收益，不到万不得已不做

**离线输入队列的实现思路**：

```
1. 检测网络状态（navigator.onLine + online/offline 事件）
2. 在线时：正常提交到 /api/v2/records
3. 离线时：
   a. 将记录写入 IndexedDB（待同步队列）
   b. UI 显示"离线模式"标记
   c. 记录在本地列表中可见（带"待同步"标记）
4. 恢复在线时：
   a. 从 IndexedDB 读取待同步队列
   b. 按时间顺序批量提交到 /api/v2/records/batch
   c. 提交成功后从 IndexedDB 删除
   d. 失败的记录保留在队列中，下次重试
5. 冲突处理：以云端数据为准（Last-Write-Wins）
```

**不建议更早实现的原因**：
- 1.5 的首要目标是让在线链路走通，离线能力是锦上添花
- 离线输入队列依赖稳定的 API 和数据结构，1.5 完成前接口可能频繁变动
- 如果 1.5 的在线体验都不好，离线模式没有意义

### 9.5 版本规划建议

- **1.5**：不做离线功能
- **1.7**：实现离线输入队列（方案 B）
- **2.x**：评估是否需要只读离线缓存（方案 C）
- **3.0+**：如果多人协同需要实时同步，评估方案 D

## 10. 其他未充分覆盖的重要问题

### 10.1 移动端体验

**问题**：当前系统是桌面优先设计。如果目标是"全面贯穿日常生活"，移动端是不可或缺的——你不会随时打开电脑记录"刚跑了3公里"。

**现状**：Next.js 响应式布局部分存在，但未专门优化移动端。QuickInput 112KB 的组件在手机上的体验未知。

**选项**：
- A. 渐进式响应式优化（在现有 Web 上优化移动端适配）
- B. PWA（添加安装能力、推送通知、离线支持）
- C. 原生 App（React Native / Flutter）

**建议**：优先 A，2.x 版本考虑 B。C 在 3.0 之前不考虑。

### 10.2 通知/提醒系统

**问题**：当前系统完全没有通知能力。计划记录到达日无提醒，目标即将超期无预警，间隔复习到时间无提示。

**影响范围**：
- 计划类记录提醒（1.5 执行清单 P5，标记为未实现）
- 目标超期预警
- 间隔复习提醒
- 周期复盘提示

**技术实现**：
- Web Push Notification（需要 Service Worker + Push API）
- 邮件通知（Supabase Edge Functions + 邮件服务）
- 应用内通知（最简单，已有页面即可展示）

**建议**：应用内通知作为 1.6 的一部分实现（最简单），Web Push 作为 PWA 改造的一部分。

### 10.3 数据安全与隐私

**问题**：当前记录包含情绪、财务、位置、社交关系等高度敏感数据。随着维度分析能力增强（1.6），这些数据的聚合分析会产生更敏感的个人信息。

**风险**：
- Supabase RLS 保护单用户隔离，但无法防御服务端漏洞
- 财务数据在传输和存储中无额外加密
- 情绪数据如果泄露，可能被滥用

**建议**：
- 1.5-1.7：依赖 Supabase RLS + HTTPS（当前水平）
- 2.0：评估敏感字段（cost, mood, energy）是否需要应用层加密
- 3.0：共享功能上线前，必须完成隐私影响评估

### 10.4 冷启动问题

**问题**：新用户注册后，系统是空的——没有记录、没有事项、没有洞察。洞察页全是零值，目标引擎无数据可算，维度面板无内容可显示。用户的第一印象是"这个系统没用"。

**解决方案**：
- **引导流程**：新用户注册后引导创建第一批事项和记录
- **示例数据**：提供可选的示例数据填充（用户可选择清除）
- **渐进式展示**：洞察面板在数据不足时显示引导提示，而非空白
- **低门槛触发**：洞察面板在 3 条记录以上即可开始显示初步分析

**建议**：作为 1.6 的一部分实现（维度面板需要考虑空状态）。

### 10.5 性能瓶颈预判

**问题**：随着数据量增长，以下操作可能成为瓶颈：

| 操作 | 风险点 | 触发条件 |
|------|--------|---------|
| 洞察页加载 | insights.ts 聚合 6 种并行查询 | 记录 > 10000 条 |
| goal-engine 计算 | 每个目标需扫描关联记录 | 事项 > 20 个，每个 > 5 个目标 |
| 180 天热力图 | 查询 180 天记录计数 | 正常使用半年后 |
| 维度交叉分析 | 多维度 × 多事项的聚合计算 | 7 维度 × 20 事项 = 140 个交叉格 |

**建议**：
- 1.5-1.7：不过早优化，但记录关键查询的执行时间
- 1.7：引入洞察数据缓存 + 离线输入队列（TanStack Query + Dexie.js）
- 2.x：评估是否需要物化视图或预计算表

### 10.6 国际化/多语言

**问题**：当前系统硬编码中文（记录类型、事项状态、情绪枚举等）。如果 3.0 扩展到国际用户，所有枚举值和 UI 文案需要国际化。

**影响**：teto.ts 中的 OUTCOME_TYPE_LABELS、PLACE_TYPE_LABELS、MONEY_DIRECTION_LABELS 等映射表是中文硬编码。DB 中存储英文枚举，但 AI 解析 prompt 是中文。

**建议**：3.0 之前不处理，但 1.5-1.7 开发中注意：
- DB 存英文枚举，前端通过映射表翻译（当前已是这个模式，保持）
- AI prompt 保持中文（目标用户以中文为主）
- UI 文案集中管理，避免散落在组件中

### 10.7 数据可移植性

**问题**：当前只有 CSV 导出（`/api/v2/export/records`）。用户如果决定迁移到其他工具，或想备份数据，格式支持有限。

**建议**：
- 1.5-1.7：CSV 导出足够
- 1.7：增加 JSON 导出（包含 Note 实体和关联关系）
- 2.x：考虑 Markdown 导出（事项 + 记录 + 笔记的完整文档）
- 3.0：考虑与 Obsidian 的数据互通（至少导出为 Obsidian 可读格式）

# 第五部分：工程品质与外部经验借鉴

## 13. 可观测性体系

### 13.1 问题

TETO 的 AI 增强链路是黑盒。用户说"归类错了"，无法定位是哪个环节出的问题：

```
用户输入 → parseNaturalInput(本地) → POST /api/v2/records → enhanceRecord()
  → parseSemantic(DeepSeek) → matchItemSmart → 子项归属 → 回写字段
  → 歧义检测 → ClarificationNeeded
```

当前全项目只有 65 处 console.log/error/warn，无结构化日志，无 trace ID，无决策追踪。调试 AI 解析问题只能重新跑一遍全部数据，浪费 token 且效率低。

### 13.2 解决方案

#### ai_trace 表

```sql
CREATE TABLE ai_traces (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id) NOT NULL,
  record_id UUID REFERENCES records(id),

  -- 链路标识
  trace_id UUID NOT NULL,           -- 一次完整增强链路的唯一 ID
  step_number INT NOT NULL,         -- 步骤编号（1=NLP, 2=parse, 3=match, ...）
  step_name TEXT NOT NULL,           -- 步骤名称

  -- 输入输出
  input JSONB,                       -- 该步骤的输入
  output JSONB,                      -- 该步骤的输出
  success BOOLEAN DEFAULT true,      -- 是否成功

  -- 决策信息
  decision_id UUID,                  -- 该步骤内关键决策的 ID
  decision_type TEXT,                -- 决策类型（item_match, sub_item_assign, field_fill, ...）
  confidence FLOAT,                  -- 置信度
  was_corrected BOOLEAN DEFAULT false, -- 用户是否修正了此决策

  -- 性能
  duration_ms INT,                   -- 该步骤耗时
  tokens_used INT,                   -- LLM token 消耗（如适用）

  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_ai_traces_record ON ai_traces(record_id);
CREATE INDEX idx_ai_traces_trace ON ai_traces(trace_id);
CREATE INDEX idx_ai_traces_decision ON ai_traces(decision_id);
```

#### AI 增强链路追踪

```
每次 AI 增强启动时：
  1. 生成 trace_id
  2. 每个步骤写入 ai_traces：
     step 1: optimize-input（模糊输入检测）
     step 2: parse-semantic（DeepSeek 解析）
     step 3: compound-split（复合句拆分）
     step 4: item-match（事项匹配）→ 记录 decision_id + confidence
     step 5: sub-item-assign（子项归属）→ 记录 decision_id + confidence
     step 6: field-fill（字段回写）→ 记录每个字段的来源
     step 7: clarification（歧义检测）

  3. 用户修正时：
     - 查找对应的 decision_id
     - 标记 was_corrected = true
     - 写入 user_rules（被动规则学习）
```

#### 决策修正率追踪

```sql
-- 归类修正率（北极星指标）
SELECT
  decision_type,
  COUNT(*) AS total,
  SUM(CASE WHEN was_corrected THEN 1 ELSE 0 END) AS corrected,
  ROUND(SUM(CASE WHEN was_corrected THEN 1 ELSE 0 END)::numeric / COUNT(*) * 100, 1) AS correction_rate
FROM ai_traces
WHERE user_id = $1
  AND created_at >= NOW() - INTERVAL '30 days'
GROUP BY decision_type
ORDER BY correction_rate DESC;
```

### 13.3 引入版本

**1.5**。这是 AI 增强链路可信的工程基础，没有可观测性就无法验证归类准确性。

## 16. 向量搜索与智能匹配

### 16.1 当前状况

TETO 的事项匹配（`matchItemSmart`）依赖关键词匹配：
- DeepSeek 输出 `item_hint`（如"英语"）
- 用 `item_hint` 与活跃事项列表做字符串相似度匹配
- 问题：用户换个说法就匹配不到（"背单词"→ 事项叫"英语学习"）

### 16.2 向量搜索方案

```
流程：
  1. 用户输入 → Embedding API（千问/OpenAI text-embedding-3-small）
  2. 向量搜索 → Supabase pgvector 找 Top-5 相似事项
  3. Reranker → Jina Reranker 或 Cohere Rerank 精排
  4. Top-1 归类 → 如果置信度 > 阈值则自动归类，否则用户确认

技术实现：
  - Supabase 启用 pgvector 扩展
  - items 表增加 embedding 列（vector(1536)）
  - 事项创建/更新时计算 embedding 并存储
  - 匹配时用余弦相似度搜索
```

### 16.3 引入时机

**不立即引入**。当前事项数量少（< 50 个）时，关键词匹配够用。引入条件：

- 事项数量 > 50，或
- 用户归类修正率 > 30%（从 ai_traces 表可查），或
- matchItemSmart 的匹配准确率明显下降

**预估版本**：1.7

## 17. 日历能力：计划类记录的视图补全

### 17.1 问题

TETO 已有计划类记录（`type='计划'`，`time_anchor_date` 指向未来），但缺：
- 日历视图（按天/周展示计划的时间分布）
- 到期提醒（1.5 执行清单 P5，标记为未实现）
- 计划→发生的转化闭环

### 17.2 解决方案

不是"纳入日历功能"，而是**为计划类记录补上视图和提醒能力**：

1. **日历视图**：在洞察页或独立页面，按周/月展示计划记录的时间分布
2. **到期提醒**：计划记录的 `time_anchor_date` 到达时，应用内通知提醒
3. **计划→发生转化**：计划到期后提示"是否已执行"，如果执行则自动转为发生记录（`lifecycle_status: completed` + 生成关联的发生记录）

### 17.3 与王自如日历的区别

| | 王自如的日历 | TETO 的计划视图 |
|---|---|---|
| 核心操作 | 创建时间槽 | 记录计划 + 自动解析时间 |
| 多人共享 | 支持 | 3.0 实现 |
| 冲突检测 | 有 | 不做（TETO 不管时间冲突） |
| 与记录的关系 | 日程是独立的 | 计划是记录的一种类型，可转为发生 |
| AI 自动创建 | 全自动 | AI 解析 + 用户确认 |

### 17.4 引入版本

**1.5-1.6**（1.5 做到期提醒 P5，1.6 做日历视图）
