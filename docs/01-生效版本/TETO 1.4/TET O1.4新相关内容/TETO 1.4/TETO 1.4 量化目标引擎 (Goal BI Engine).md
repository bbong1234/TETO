# TETO 1.4 量化目标引擎 (Goal BI Engine)

# 第一部分：功能说明书（用户视角）

### 整体理念

这套系统的核心思想是​ **"金融账本式"** ：不派发任务、不催你打卡、不发通知骂你。你只管随手记录，系统只管默默算账。当你主动打开面板时，冷血地展示你的"欠债"和"完成率"。

### 功能 1：在事项中配置量化标尺

**使用场景：**  你有一个事项叫"[知识] 英语词汇突破"，想给它设定量化目标。

**操作步骤：**

1. 进入该事项的详情页
2. 在"目标"区域，点击新建目标
3. 填写目标信息：
4. - 标题：英语词汇量突破 40000
   - 度量类型：选择"量化型"（numeric）
   - 计量单位：个
   - 每日期望值：110
   - 起算日期：2024-12-23
   - （可选）总目标量：40150
   - （可选）截止日期：2026-12-31
5. 保存。这就是你的"标尺"，系统会基于它来算账。

**一个事项可以有多个量化目标。**  比如"英语学习"事项下可以同时有"词汇量 110个/天"和"听读时长 60分/天"两个不同维度的目标。

### 功能 2：日常无痛记录（已有功能，无需改动）

**使用场景：**  你今天背了单词，想记一笔。

**操作步骤：**

1. 在今日记录页，新建一条记录
2. 写内容："地铁上过了一遍 list 1"
3. 选择关联事项：英语词汇突破
4. 填写数值(metric\_value)：40
5. 提交

晚上又背了 40 个，再记一条，数值填 40。

**关键点：**  一天可以记很多条，系统会自动把同一事项下的数值加总。你不需要手动汇总，记了就走。

周末太累没背？不打开软件就行，系统不会发通知催你。但你的"欠债"会默默增加。

### 功能 3：量化仪表盘（核心新功能）

**使用场景：**  周一你想看看自己的英语学习到底欠了多少债。

**操作步骤：**

1. 打开 TETO，进入"[知识] 英语词汇突破"事项详情页
2. 页面上会出现一个"量化仪表盘"面板

**你会看到什么（以你的 Excel 为参照）：**

|指标名称|显示内容|含义|对应你Excel的哪一行|
| ----------| -----------| -----------------------------------------------------------------| ---------------------|
|坚持天数|486 天|从起算日(2024/12/23)到今天过了多少天|"坚持 第 486 天"|
|今日进度|80 / 110|今天已记录的数值合计 / 每日目标|--|
|每日平均|55.83 个|历史总数值 / 总天数|"每日平均"|
|近7日均|0.00 个|最近7天的日均数值|"近7日均"|
|近30日均|6.00 个|最近30天的日均数值|"近30日均"|
|**总完成率**|**50.76%**|完成总值 / 合计应当。红色(\<50%) 黄色(50-80%) 绿色(\>80%)|"总完成率"(带颜色)|
|合计应当|53,460 个|486天 x 每日110 \= 你"本该"完成的总量|"合计应当"|
|完成总值|27,135 个|你所有记录的 metric\_value 全部加起来|"完成总值"|
|**合计差值**| **-26,325 个 (红色!)**|完成总值 - 合计应当。负数\=欠债，红色高亮|"合计差值 -26325"|
|动态配速|111 个/天|如果设了总目标和截止日：(总目标-已完成)/剩余天数|隐含在"26剩余"中|

**核心体验：**  那个  **-26,325** 的红色数字会放在最显眼的位置。你不需要自己算，系统一秒算完，直接甩在你脸上。关掉软件，把紧迫感带走，转化为明天的动力。

### 功能 4：多目标并列

**使用场景：**  你的"英语学习"事项下有两个量化目标：词汇量(110个/天) 和 听读时长(60分/天)。

**展示效果：**  仪表盘面板会显示两张独立的卡片，各自有各自的完成率、差额、配速。就像你 Excel 里"英语单词"和"英语听读"是两列独立的数据一样。

### 本期 MVP 不包含的功能（后续扩展）

- 图表视图（折线图/柱状图）-- 对应你 Excel 的"学习统计表"折线图
- 流水态视图（按天展开的每日明细）-- 对应你 Excel 的逐日流水表
- 每日总分/加权评分 -- 对应你 Excel 的"总分(120)"列
- 起床/睡觉等时间型指标的特殊处理
- 年度目标/年度预计等更长周期的投射

## 第二部分：技术现状与缺口

### 已有的基础

- **记录带数值：**  用户创建记录时已经可以填写 metric\_value（数值）、metric\_unit（单位）、metric\_name（对象名），并关联到事项(item\_id)。这是引擎的"数据源"，不需要改动。
- **目标基础结构：**  目标(Goal)表已经有"归属事项"(item\_id)、"度量类型"(boolean/numeric)、"总目标值"(target\_value)。这是引擎的"配置源"的一半。
- **事项详情页：**  已有目标管理区域(ItemGoalSection)，支持目标的增删改查。但只是列表展示，没有任何计算和仪表盘。

### 缺少的部分

|缺什么|为什么需要|
| --------------------------------------------------------------------------------------------------------------| ----------------------------------------------------------------------|
|目标表缺 4 个字段：unit(单位)、daily\_target(日均期望)、start\_date(起算日)、deadline\_date(截止日)|没有这些字段，就无法配置"标尺"，引擎没有计算基准|
|没有聚合计算逻辑|目前只有纯增删改查，没有"把所有记录的数值加起来和目标对比"的计算函数|
|没有数据仪表盘组件|目标区域只显示目标标题和状态，没有任何数值面板|

### 数据流转

```
plaintext你的操作          系统内部                     你看到的结果
─────────────────────────────────────────────────────────
配置目标    -->   goals 表存储标尺配置    -->   "已设定：110个/天"
随手记录    -->   records 表存储流水      -->   "已记录：背了40个"
打开面板    -->   引擎计算：加总所有记录   -->   "-26325 个（红色）"
               　对比目标配置
               　算出差额/完成率/配速
```

## 第三部分：开发任务

### Task 1: 数据库表扩展

给 goals 表加 4 个新字段：unit、daily\_target、start\_date、deadline\_date。写一个 SQL 迁移文件 ﻿sql/010\_goal\_benchmark\_fields.sql﻿，用户需要在 Supabase 中执行。

### Task 2: 类型定义更新

更新 ﻿src/types/teto.ts﻿：

- Goal 接口加上新字段
- 新增 GoalEngineResult 接口（定义引擎输出的所有计算字段）
- 创建/更新目标的请求类型也加上新字段

### Task 3: 目标读写适配

更新 ﻿src/lib/db/goals.ts﻿：让创建目标和更新目标的函数支持读写新加的 4 个字段。

### Task 4: 核心计算引擎

新建 ﻿src/lib/db/goal-engine.ts﻿：这是整个功能的心脏。接收一个目标ID，拉取目标配置和该事项下所有带数值的记录，一次性算出 GoalEngineResult 的所有字段（天数、总和、差额、完成率、均值、配速、投射）。

### Task 5: API 接口

新建两个 API 端点：

- ﻿﻿GET /api/v2/goals/{id}/engine﻿ -- 单个目标的计算结果
- ﻿﻿GET /api/v2/items/{id}/goal-engine﻿ -- 事项下所有量化目标的计算结果（前端面板调用这个）

### Task 6: 前端数据获取

新建 ﻿src/lib/hooks/useGoalEngine.ts﻿：一个简单的数据获取 Hook，调用上面的 API，返回计算结果给面板组件使用。

### Task 7: MVP 仪表盘面板

新建 ﻿GoalEngineDashboard.tsx﻿ 组件，嵌入事项详情页。把 Task 4 计算出的所有指标用网格布局平铺展示，差值为负数时红色高亮。
