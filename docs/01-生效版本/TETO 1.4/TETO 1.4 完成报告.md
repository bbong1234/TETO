## 二、结项结论

TETO 1.4 已完成结项。

本阶段的核心不是在三页骨架上继续缝补功能，而是完成了 **两个质变**：

1. **事项从"平铺容器"升级为"可组织的四维主题空间"**——事项内部有了子项（行动线）、目标（方向线）、阶段（时间线）、记录（事实线），不再是 1.3 那个只有标题和状态的扁平对象。

2. **记录从"文本+标签"升级为"结构化语义断言"**——AI 语义解析引擎让每条记录不再是散乱的文本，而是被拆解为主谓宾、时间锚点、关系人、地点、心情、能量的结构化数据。

当前已确认：

- 事项—阶段—目标—文件夹的完整闭环已跑通
- 历史导入两条路径（具体记录导入 + 历史阶段补录）已落地
- 量化目标引擎（差额/完成率/配速/投射）已运行
- 子项系统（行动线拆分、子项升格为独立事项）已落地
- 重复型目标（每天/每周/每月）已落地
- DeepSeek AI 语义解析引擎已集成，复合句拆分、歧义澄清、置信度分级已实现
- 记录生命周期闭环（完成/推迟/取消）已落地
- 记录微关联（completes/derived_from/postponed_from/related_to）已落地
- 洞察页增强（事项画像、阶段洞察、目标洞察、时间分布、跨事项对比）已落地

因此，TETO 1.4 的真实结论是：

> **事项深化与语义解析引擎建设目标已落地，系统已从"记录/事项/洞察"的三页骨架推进为"结构化记录 + 四维事项 + 增强洞察"的个人现实管理系统；事项内部有了自己的空间结构，记录内部有了自己的语法结构。**

## 三、1.4 的阶段定位

TETO 1.4 的定位，不是 1.3 的功能补充，而是：

> **对 1.3 三页骨架的纵深建设。**

1.3 解决的是"系统围绕什么运转"的问题，1.4 解决的是：

- 事项内部到底怎么组织（子项/目标/阶段/记录）
- 记录内部到底怎么结构化（语义解析/时间锚点/复合句拆分）
- 1.3 搁置的目标系统到底怎么建（量化引擎/重复型/达标型）
- 1.3 缺失的历史能力到底怎么补（历史导入/阶段补录）

所以 1.4 不是把系统铺宽，而是把系统打深。

## 四、1.4 的核心变化：两根纵梁的竖起

### 4.1 旧结构：扁平事项 + 文本记录

1.3 及之前，事项和记录的结构是：

- 事项只有标题、描述、状态，是一个扁平容器
- 记录只有文本、类型、标签，是一个散漫条目
- 目标以 goal_id 附在 items 上，方向是 Goal→Item
- 没有阶段，事项没有内部时间分段
- 没有语义解析，记录全靠手动填写结构化字段

这导致：

- 事项页面信息密度低，只能看标题和关联记录数
- 记录的量化数据（时长、花费、指标）全靠手动补充
- 无法回看"这段时间事项整体怎样"
- 无法自动推断记录应该归属哪个事项

### 4.2 新结构：四维事项 + 语义记录

1.4 将事项和记录同时深化：

**事项四维模型：**

| 维度 | 对象 | 回答的问题 | 例子 |
|------|------|------------|------|
| 行动维度 | 子项(SubItem) | 我具体在做什么？ | 背单词、练听读 |
| 方向维度 | 目标(Goal) | 要做到什么程度？ | 每天110个、通过四级 |
| 时间维度 | 阶段(Phase) | 这段时期整体怎样？ | 基础积累期、冲刺期 |
| 事实维度 | 记录(Record) | 某天具体发生了什么？ | 今天背了50个单词 |

**记录语义模型：**

| 维度 | 字段 | 例子 |
|------|------|------|
| 核心断言 | subject + action + object | 我 + 吃 + 猪脚饭 |
| 时间锚点 | time_anchor(raw/direction/resolved_date) | 明天/future/2026-04-21 |
| 空间 | location | 公司 |
| 关系 | people | ["小明","同事"] |
| 修饰 | mood/energy/manner | 开心/疲惫/匆忙地 |
| 量化 | cost/duration/metric | ¥30/2h/50个单词 |
| 关联意图 | record_link_hint/item_hint | →上周考试/→英语 |

这带来的根本改变：

- 事项不再是一个标题卡片，而是一个有四根柱子的空间
- 记录不再是一段散漫文本，而是一个结构化断言
- AI 成为"精准填表员"，把散漫输入拆解为结构化数据
- 目标有了量化引擎，不再是"是否完成"的简单判断
- 历史可以补录，系统不再是只记录"从现在起"的工具

### 5.9 已完成记录类型收敛

从 1.3 的 7 种并列类型（发生/计划/情绪/想法/花费/总结/结果），收敛为以"发生"为中心的 4 种：

> **记录类型不应是并列的，而应以"发生"为中心，其他为维度。**

## 六、1.4 阶段完成的核心价值

### 6.1 给事项建了内部结构

1.3 的事项是一个扁平卡片。1.4 给它竖了四根柱子：子项、目标、阶段、记录。事项不再是"标题+状态"，而是一个有纵深的空间。

### 6.2 给记录装了语法引擎

1.3 的记录是"文本+标签"。1.4 的记录是"结构化断言"。AI 把散漫输入拆解为主谓宾、时间锚点、关系人、地点、心情、能量，不再是贴标签，而是解构语法。

### 6.3 让历史可以进入系统

1.3 只记录"从今天起"。1.4 可以导入历史记录、补录历史阶段。TETO 不再只是"此刻的日记本"，而是"人生的结构化账本"。

### 6.4 让目标有了量化引擎

1.3 没有独立目标。1.4 的量化目标引擎让"每天110个单词"这类目标有了差额、配速、投射，不再是"设了就忘"的空话。

### 6.5 让计划有了生命周期

1.3 的计划只是一条文本。1.4 的计划有完成、推迟、取消三条路径，每条路径自动生成关联记录。计划不再是孤立的待办，而是有后续动作的活对象。

### 6.6 让洞察从粗到细

1.3 的洞察只有记录数和事项数。1.4 的洞察有事项画像、阶段统计、目标分布、时间分布、跨事项对比。洞察不再是简单的统计大屏，而是有结构的事实回顾。

## 七、1.4 从构思到落地的关键演进

### 7.1 前半阶段：事项深化与阶段闭环

1.4 初期聚焦于 1.3 的直接延伸——给事项加阶段、目标、文件夹，建立历史导入能力。这对应开发清单前部分 P0-P7。

关键认知演进：
- 阶段从"人生大环境"（如高中时期）收敛为"事项内部时间分段"（如四级备考期）——因为"高中"属于洞察层的人生画像，不属于事项的内部结构
- 目标方向从 Goal→Item 反转为 Item→Goal，目标通过 item_id 反向归属事项，而非事项通过 goal_id 正向绑定目标
- 记录类型从 7 种并列收敛为 4 种，心情/能量/花费/结果降为记录维度

### 7.2 后半阶段：语义解析引擎建设

在事项闭环稳定后，1.4 转向记录层的深度建设。核心认知转变：

- 从"贴标签"思维（给记录贴心情、能量标签）升级为"语法解构"思维（把记录拆解为主谓宾定状补）
- AI 的角色不是"聊天伴侣"或"建议机器"，而是"精准填表员"——把散漫输入拆解填入结构化坑位
- 目标与洞察必须搁置，直到记录层语义解析打磨稳定——"单张照片的要素没解析清楚，相册归类和图库洞察都是无源之水"

### 7.3 超出规划的实现

1.4 在规划之外额外落地了：
- 子项(SubItem)系统——原始方案中子项仅在"事项模块功能方案"中讨论，未列入 P0-P8，但已完整实现含升格能力
- 重复型目标——原始目标引擎仅规划 boolean/numeric，repeat 是后加的第三种度量类型
- 记录生命周期——complete/postpone/cancel 三个端点 + record_links 微关联
- 歧义澄清框架——ClarificationNeeded 四种类型，超出原始语义引擎规划

## 八、1.4 的局限

### 8.1 语义解析前端展示未升级

原始规划将 QuickInput 芯片区升级为"语义卡片区"（主体行+上下文行+修饰行+数据行+关联行），目前仍使用原有芯片式 UI。AI 解析结果虽然已持久化和回写，但用户在输入时看不到结构化的语义卡片预览。

### 8.2 规则兜底层未实现

当 LLM 不可用（网络异常/API 限额）时，应 fallback 到增强版 parseNaturalInput。目前无此兜底，AI 不可用时仅保留前端的基础正则解析。

### 8.3 记录卡片展示增强未实现

原规划在 RecordItem 中展示时间锚点标记（→明天/←上周）、关联链接、关系人标签、地点标签。目前这些字段已入库但前端未展示。

### 8.4 旧代码残留

- chains 表未 DROP，仍占用空间
- items.goal_id 标记为 @deprecated 但仍保留，API 仍返回旧 goal 字段
- 部分 SQL 迁移（011/013/014）需确认是否已在 Supabase 执行

### 8.5 计划提醒未实现

Task D（time_anchor.direction=future 时触发提醒）搁置。计划类记录有生命周期闭环，但没有主动提醒。

### 8.6 事项详情页不直接返回关联记录

事项详情 API 返回 aggregation 聚合数据但不返回具体记录列表。查看具体记录需跳转记录页通过 item_id 筛选。

## 十一、1.4 之后不应继续做的事

1.4 完成了事项深化和语义引擎建设，但后续不应：

- 把事项的四个维度（子项/目标/阶段/记录）拆成四个一级页面——它们是事项的内部结构，不是独立模块
- 在语义解析精度未稳定前，让 AI 做出不可撤销的自动操作（如自动修改用户填写的值）
- 在记录层语义数据不够丰富前，过早推进"因果图谱"和"目标自动碰撞"
- 把阶段泛化为"人生大环境"——阶段永远是事项内部的时间分段
- 让目标变成"愿景"或"方向"——目标必须是能被客观验证的里程碑
- 在三页主骨架未成熟前，扩张更多一级模块

---

## 十二、1.4 之后可以开始做的事

1.4 已把纵深打好，后续可以在稳定地基上：

- 升级 QuickInput 语义卡片 UI——让用户在输入时就能看到结构化的语义预览
- 实现规则兜底层——LLM 不可用时 fallback 到增强版正则解析
- 记录卡片展示增强——时间锚点标记、关联链接、关系人/地点标签
- 计划类记录日期提醒——time_anchor.direction=future 时主动提醒
- 事项详情页直接返回关联记录——不再需要跳转记录页
- 清理旧代码——DROP chains 表、移除 items.goal_id
- 确认并执行所有待确认的 SQL 迁移
- 优化移动端体验

## 十三、最终结论

TETO 1.4 的真实结论：

> **TETO 已从"记录/事项/洞察"的三页骨架推进为"结构化记录 + 四维事项 + 增强洞察"的个人现实管理系统。事项内部有了子项、目标、阶段、记录四根柱子，不再是扁平容器；记录内部有了主谓宾、时间锚点、关系人、地点的语法结构，不再是散漫文本；AI 成为精准填表员而非聊天伴侣；历史可以补录而非只能记录"从今天起"。纵深建设目标已落地，数据闭环已跑通，真实可验证。**
