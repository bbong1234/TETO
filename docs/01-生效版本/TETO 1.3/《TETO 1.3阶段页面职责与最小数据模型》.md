# 《TETO 1.3阶段页面职责与最小数据模型》

## 一、文档定位

本文件用于把《TETO 新阶段总规（完整正式版）》落到可执行层。

本文件只解决两件事：

1. **第一阶段页面到底有哪些、各自只负责什么**
2. **第一阶段最小数据模型到底是什么**

本文件不进入：

- UI 视觉设计
- 技术框架细节
- 数据库优化
- AI 扩展
- 多人协作
- 企业级结构
- 复杂自动化

---

## 二、第一阶段唯一目标

第一阶段只追求一个最小闭环：

> **用户可以围绕时间，记录真实发生，补充状态，标记成果，关联目标，并在回看中看见结果。**

只要这个闭环没有成立，其他扩展都不算优先。

---

## 三、第一阶段页面总表

第一阶段页面只保留 6 个：

1. **今日页**
2. **发生页**
3. **成果页**
4. **目标页**
5. **容器页**
6. **复盘页**

这是第一阶段的固定页面集合。  
不额外新增“工作台”“总控台”“超级首页”“AI 首页”等页面。

---

## 四、页面职责定义

---

## 4.1 今日页

### 页面定位

默认首页，系统唯一现实入口页。

### 页面目标

让用户先面对“今天真实发生了什么”，而不是先面对任务列表或统计面板。

### 必须承接的内容

1. 今日时间流
2. 今日发生记录入口
3. 今日状态补充
4. 今日成果标记
5. 今日目标关联
6. 今日基础回看

### 今日页必须能做的动作

1. 新建一条发生项
2. 为发生项补时间
3. 为发生项补状态
4. 为发生项补成果
5. 给发生项关联目标
6. 查看今天已有记录

### 今日页不负责

- 全量历史筛选
- 复杂成果分析
- 项目管理
- 复杂任务管理
- 大型统计面板

### 今日页核心原则

> **今日页是现实入口，不是管理后台。**

---

## 4.2 发生页

### 页面定位

原始现实层总览页。

### 页面目标

让用户从“真实发生”视角看系统，而不是从任务视角看系统。

### 必须承接的内容

1. 全部发生项列表
2. 发生项筛选
3. 按时间查看发生
4. 按类型查看发生
5. 查看发生项详情

### 发生页必须能做的动作

1. 查看全部发生项
2. 按日期筛选
3. 按发生类型筛选
4. 按状态筛选
5. 打开发生详情
6. 编辑发生项

### 发生页不负责

- 目标分析主展示
- 成果统计主展示
- 任务执行主操作

### 发生页核心原则

> **发生页负责看现实，不负责组织全部系统。**

---

## 4.3 成果页

### 页面定位

结果层总览页。

### 页面目标

让用户看见“时间和发生最终形成了什么”。

### 必须承接的内容

1. 全部成果项列表
2. 按成果类型筛选
3. 按目标查看成果
4. 查看某段时间形成的成果

### 成果页必须能做的动作

1. 查看所有成果项
2. 按成果类型筛选
3. 按时间筛选
4. 按目标筛选
5. 查看成果来源发生项

### 成果页不负责

- 原始记录入口
- 任务分配
- 项目管理
- 复杂复盘图表

### 成果页核心原则

> **成果页回答“形成了什么”，不回答“该做什么”。**

---

## 4.4 目标页

### 页面定位

方向层页面。

### 页面目标

让用户看见“自己的现实是否正在靠近目标”。

### 必须承接的内容

1. 目标列表
2. 单个目标详情
3. 目标关联的发生
4. 目标关联的成果
5. 目标接近度的基础判断

### 目标页必须能做的动作

1. 创建目标
2. 查看目标
3. 编辑目标
4. 查看目标相关发生项
5. 查看目标相关成果项

### 目标页不负责

- 代替任务页
- 代替项目页
- 代替今日记录页

### 目标页核心原则

> **目标页是方向解释层，不是执行入口。**

---

## 4.5 容器页

### 页面定位

组织层页面。

### 页面目标

统一承载任务、项目、领域、主题等组织对象。

### 第一阶段允许的容器类型

1. 任务
2. 项目
3. 领域
4. 主题

### 必须承接的内容

1. 容器列表
2. 单个容器详情
3. 容器关联的发生项
4. 容器关联的成果项

### 容器页必须能做的动作

1. 创建容器
2. 查看容器
3. 编辑容器
4. 关联发生项
5. 关联成果项

### 容器页不负责

- 作为默认首页
- 承担现实记录入口
- 替代目标判断
- 替代复盘页

### 容器页核心原则

> **容器是组织工具，不是底层现实。**

---

## 4.6 复盘页

### 页面定位

理解层页面。

### 页面目标

帮助用户看清：

- 时间去哪了
- 发生了什么
- 形成了什么
- 是否接近目标

### 必须承接的内容

1. 时间分布
2. 发生分布
3. 成果分布
4. 目标关联回看
5. 基础趋势判断

### 复盘页必须能做的动作

1. 按日期范围回看
2. 查看时间分布
3. 查看发生类型分布
4. 查看成果类型分布
5. 查看目标相关回看

### 复盘页不负责

- 原始数据录入
- 复杂 BI
- AI 自动决策

### 复盘页核心原则

> **复盘页必须让人直接看懂，不靠炫图。**

---

## 五、页面之间的关系

第一阶段页面关系固定如下：

### 主链路

**今日页 → 发生记录 → 成果标记 → 目标关联 → 复盘页**

### 补充链路

- 今日页 可进入 发生页
- 发生页 可进入 单条发生详情
- 发生详情 可查看对应成果
- 成果页 可回看来源发生
- 目标页 可查看相关发生与成果
- 容器页 可查看关联发生与成果
- 复盘页 只用于理解，不承担录入

---

## 六、页面优先级顺序

开发优先级必须固定：

### 第一优先

1. 今日页
2. 发生记录能力
3. 成果标记能力

### 第二优先

4. 目标页
5. 复盘页基础版

### 第三优先

6. 容器页基础版
7. 发生页/成果页完整筛选

也就是说：

> **先把主链路做通，再补充查看页。**

---

## 七、最小数据模型总表

第一阶段只保留 6 个核心数据对象：

1. **time\_block**
2. **event**
3. **outcome**
4. **goal**
5. **container**
6. **state\_tag**

这是最小闭环，不再新增更多核心表。

---

## 八、数据对象正式定义

---

## 8.1 time\_block

### 作用

承接一段现实时间。

### 最小字段

- ​`id`
- ​`date`
- ​`start_time`
- ​`end_time`
- ​`duration_minutes`
- ​`time_precision`

  - 精确
  - 模糊
- ​`created_at`
- ​`updated_at`

### 说明

time\_block 是时间定位对象。  
它不表达“发生了什么”，只表达“这段时间存在”。

---

## 8.2 event

### 作用

承接某段时间里真实发生的内容。

### 最小字段

- ​`id`
- ​`time_block_id`
- ​`title`
- ​`description`
- ​`event_type`
- ​`initiative_type`

  - 主动
  - 被动
- ​`plan_type`

  - 计划内
  - 计划外
- ​`state_tag_id`（可空）
- ​`goal_id`（可空）
- ​`container_id`（可空）
- ​`created_at`
- ​`updated_at`

### 说明

event 是第一核心对象。  
没有 event，系统就退化成空时间记录或空成果统计。

### event\_type 第一阶段建议值

- 行动
- 事务处理
- 沟通
- 思考
- 陪伴
- 休息
- 恢复
- 中断
- 应急
- 其他

---

## 8.3 outcome

### 作用

表达某次发生最终形成的现实结果。

### 最小字段

- ​`id`
- ​`event_id`
- ​`outcome_type`
- ​`summary`
- ​`goal_closeness`
- ​`created_at`
- ​`updated_at`

### 说明

一条发生必须允许没有明显成果。  
所以 outcome 可以存在“无明显成果”类型，而不是强迫每条都产出。

### outcome\_type 第一阶段建议值

- 完成
- 推进
- 产出
- 恢复
- 维持
- 修复
- 消耗
- 偏离
- 无明显成果

### goal\_closeness 第一阶段建议值

- 更接近
- 无明显变化
- 更偏离

---

## 8.4 goal

### 作用

表达方向，而不是待办。

### 最小字段

- ​`id`
- ​`title`
- ​`description`
- ​`goal_status`
- ​`created_at`
- ​`updated_at`

### goal\_status 第一阶段建议值

- 进行中
- 暂停
- 已完成
- 已放弃

### 说明

goal 不需要复杂层级。  
第一阶段只要能承接方向即可。

---

## 8.5 container

### 作用

承接组织对象。

### 最小字段

- ​`id`
- ​`container_type`
- ​`title`
- ​`description`
- ​`status`
- ​`created_at`
- ​`updated_at`

### container\_type 第一阶段建议值

- 任务
- 项目
- 领域
- 主题

### status 第一阶段建议值

- 激活
- 暂停
- 完成
- 废弃

### 说明

container 是组织层，不是底层现实层。

---

## 8.6 state\_tag

### 作用

表达发生时的状态。

### 最小字段

- ​`id`
- ​`name`
- ​`description`
- ​`created_at`
- ​`updated_at`

### 第一阶段建议状态值

- 清醒
- 专注
- 疲惫
- 混乱
- 抵触
- 平稳
- 恢复中
- 被打断
- 低效
- 高效

### 说明

第一阶段不做复杂情绪量化。  
只做基础状态标签。

---

## 九、对象关系定义

第一阶段关系固定如下：

### 1）time\_block 与 event

- 一个 time\_block 可对应一条或多条 event
- 一条 event 必须属于一个 time\_block

### 2）event 与 outcome

- 一条 event 可有 0 或 1 条主要 outcome
- 第一阶段不做多成果拆分

### 3）event 与 goal

- 一条 event 可关联 0 或 1 个 goal
- 第一阶段不做多目标关联

### 4）event 与 container

- 一条 event 可关联 0 或 1 个 container
- 第一阶段不做多容器并挂

### 5）event 与 state\_tag

- 一条 event 可关联 0 或 1 个 state\_tag
- 第一阶段不做多状态复合

---

## 十、为什么先限制为单关联

第一阶段故意限制为：

- 一条发生只挂一个目标
- 一条发生只挂一个容器
- 一条发生只挂一个主要状态
- 一条发生只挂一个主要成果

原因只有一个：

> **先保证系统轻、顺、可记录。**

如果一开始就多对多乱开，第一阶段会立刻变重。

---

## 十一、第一阶段核心录入字段

如果只看用户录入，第一阶段每次最少只需要填这些：

### 新建发生时最少字段

1. 时间
2. 发生标题
3. 发生类型

### 可选补充字段

4. 描述
5. 主动/被动
6. 计划内/计划外
7. 状态
8. 成果
9. 目标
10. 容器

也就是说：

> **第一阶段必须允许快速记录。**

不能让用户每记录一次都填十几个字段。

---

## 十二、第一阶段最小交互规则

### 规则 1：先记录，再补充

允许用户先只写：

- 时间
- 标题
- 类型

然后稍后补：

- 状态
- 成果
- 目标
- 容器

---

### 规则 2：成果可后补

成果不要求同步填写。  
否则记录压力会太大。

---

### 规则 3：目标可不强绑

不是每条发生都必须有目标。  
否则会扭曲现实。

---

### 规则 4：容器必须降级

任务/项目不是必填归属。  
否则旧骨架会借尸还魂。

---

## 十三、第一阶段最小页面-数据映射

### 今日页

需要读写：

- time\_block
- event
- state\_tag
- outcome
- goal
- container（可选）

### 发生页

主要读取：

- event
- time\_block
- state\_tag
- goal
- container

### 成果页

主要读取：

- outcome
- event
- goal

### 目标页

主要读取：

- goal
- event
- outcome

### 容器页

主要读取：

- container
- event
- outcome

### 复盘页

聚合读取：

- time\_block
- event
- outcome
- goal

---

## 十四、第一阶段必须先做出的 4 个核心能力

### 能力 1：记录发生

用户能在时间上新增发生项。

### 能力 2：补状态

用户能给发生项补一个状态。

### 能力 3：补成果

用户能给发生项标一个成果。

### 能力 4：回看结果

用户能在页面中看见：

- 今天发生了什么
- 最终形成了什么
- 是否关联某目标

这四个能力没完成，第一阶段不算成立。

---

## 十五、第一阶段不做的数据复杂化

明确不做：

1. 多层级目标树
2. 多层级项目树
3. 事件多成果拆分
4. 多状态并行标记
5. 自动评分引擎
6. AI 自动生成结果
7. 复杂权重算法
8. 复杂关系图谱

---

## 十六、第一阶段页面验收标准

### 今日页验收

必须做到：

- 能新增发生
- 能看到今天记录
- 能补状态
- 能补成果
- 能关联目标

### 发生页验收

必须做到：

- 能查看全部发生
- 能按时间筛选
- 能打开发生详情

### 成果页验收

必须做到：

- 能查看成果列表
- 能看到成果来源发生

### 目标页验收

必须做到：

- 能创建目标
- 能查看目标关联发生和成果

### 容器页验收

必须做到：

- 能创建容器
- 能查看容器关联发生

### 复盘页验收

必须做到：

- 能按时间范围查看基础分布
- 能看见成果分布
- 能看见目标关联基础情况

---

## 十七、第一阶段数据验收标准

数据层至少必须验证以下事实：

1. time\_block 能创建
2. event 能挂到 time\_block
3. outcome 能挂到 event
4. goal 能被 event 关联
5. container 能被 event 关联
6. state\_tag 能被 event 使用
7. 页面读取结果与保存数据一致

---

## 十八、第一阶段真实闭环验收

必须至少手动跑通这条链：

> **创建今天 09:00-10:00 时间段 → 记录“处理客户问题” → 标记为“被动 / 计划外” → 标记状态“疲惫” → 标记成果“推进” → 关联目标“维持业务稳定” → 在今日页看到它 → 在成果页看到结果 → 在目标页看到关联 → 在复盘页看到该记录被统计进去**
