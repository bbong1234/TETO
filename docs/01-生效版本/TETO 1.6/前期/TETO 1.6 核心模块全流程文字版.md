## 0. 总原则

TETO 1.6 的主流程是：

```
用户自然语言输入
↓
生成编号
↓
规则中心 AI 入库前清分
↓
能确认的直接入库
↓
不能确认的返回确认卡片
↓
入库后进入事项、计算、目标、洞察等正式业务模块
↓
用户纠错后生成 corrections、测试用例条目、错误趋势
↓
等待后续规则更新
```

核心原则：

```
未确认数据不入库
入库数据视为正式可信数据
计算中心只算正式入库数据
目标不直接消费原始记录
洞察不直接计算数据
AI 不自动修改长期规则
每一次判断都要能回放
```

---

# 1. 编号与可观测性总流程

每次用户输入和系统处理都必须可追踪。

## 1.1 编号层级

至少需要以下编号：

```
trace_id    = 一次用户操作的完整链路编号
input_id    = 一次原始输入编号
unit_id     = 从原始输入拆分出来的单句编号
decision_id = 每个关键判断编号
record_id   = 正式入库后的记录编号
module_id   = 系统模块编号
error_code  = 错误编号
test_case_id = 测试用例条目编号
```

---

## 1.2 复合句编号规则

复合句和拆分后的单句不能共用同一个编号。

例如用户输入：

```
今天跑步5公里，晚上烧烤花了80，明天背英语30分钟
```

应该生成：

```
trace_id = T-20260507-xxxxxx
input_id = INP-20260507-xxxxxx
```

拆分后：

```
unit_id = UNIT-INP-xxxxxx-01  今天跑步5公里
unit_id = UNIT-INP-xxxxxx-02  晚上烧烤花了80
unit_id = UNIT-INP-xxxxxx-03  明天背英语30分钟
```

每个单句还会产生多个 `decision_id`：

```
DEC-SPLIT-xxxxxx       为什么这样拆分
DEC-TYPE-xxxxxx        为什么判断为发生/计划/想法/总结
DEC-ITEM-xxxxxx        为什么归到某事项
DEC-TIME-xxxxxx        为什么识别为某时间
DEC-AMOUNT-xxxxxx      为什么识别为某金额
DEC-ADMISSION-xxxxxx   为什么允许或不允许入库
```

---

## 1.3 可观测性要回答的问题

系统必须能通过 `trace_id` 回放：

```
这句话怎么拆的？
为什么归到这个事项？
为什么没有入库？
为什么进入确认卡片？
用户后来改了什么？
有没有生成测试用例？
是规则中心错了，还是计算中心错了？
是事项模块错了，还是目标模块错了？
是前端展示错了，还是后端返回错了？
```

一句话：

> **每一次输入、每一次判断、每一次模块处理都要能回放。**

---

# 2. 录入模块流程

录入模块是所有自然语言进入系统的入口。

## 2.1 正确主流程

```
1. 用户输入 content + date
↓
2. 生成 trace_id / input_id
↓
3. 基础请求校验
   - content 是否为空
   - date 是否合法
   - 用户身份是否合法
   - 请求格式是否合法
↓
4. 规则中心 AI 入库前一步处理
   - 拆分复合句
   - 识别类型
   - 识别事项/子项
   - 识别时间
   - 识别金额
   - 识别时长
   - 识别情绪/备注等扩展字段
   - 判断是否可入库
   - 判断是否需要确认
↓
5. 分流
   A. 高置信度 + 规则通过 → 正式入库
   B. 拆分不确定 → 返回拆分确认卡片
   C. 归属不确定 → 返回归属确认卡片
   D. 字段不确定 / 无法识别 → 返回澄清确认卡片
↓
6. 用户确认后，再进入规则中心或规则校验
↓
7. 通过后正式入库
↓
8. 返回入库结果 / 确认卡片 / 错误信息
```

---

## 2.2 录入模块不能做什么

录入模块禁止：

```
先入库，再 AI 增强
先创建主记录，再拆复合句
有歧义也先写入正式 records
让 AI 异步回写核心字段
让前端判断是否可入库
```

正确原则：

> **AI 判断必须发生在正式入库前。**

---

# 3. 规则中心 AI 流程

规则中心 AI 是自然语言清分入口，不是长期规则自动更新器。

## 3.1 规则中心 AI 负责什么

规则中心 AI 负责：

```
拆分复合句
识别记录类型
识别事项
识别子事项
识别时间
识别金额
识别时长
识别情绪/备注/地点等字段
判断置信度
判断是否需要用户确认
生成 decision_id
输出解释
```

---

## 3.2 规则中心 AI 不负责什么

规则中心 AI 不负责：

```
不直接修改长期规则
不直接更新目标进度
不直接生成洞察结论
不直接写计算结果
不绕过 Domain Service 写库
不把一次纠错变成永久规则
```

一句话：

> **规则中心 AI 负责入库前清分，不负责入库后的业务计算。**

---

# 4. 确认卡片流程

确认卡片是主流程的重要分支。

## 4.1 触发条件

当规则中心 AI 无法稳定确认时，返回确认卡片。

触发情况：

```
拆分不确定
类型不确定
事项不确定
子事项不确定
时间不确定
金额不确定
时长不确定
无法识别
```

注意：
**确认卡片不处理目标影响不确定。**

因为目标不在录入阶段直接判断，目标基于事项聚合结果计算。

---

## 4.2 确认卡片流程

```
规则中心 AI 返回确认卡片
↓
用户选择 / 修改
↓
再次进入规则中心 AI 或规则校验
↓
通过后正式入库
```

---

## 4.3 确认卡片形态

确认卡片不是普通表单。

它应该是：

```
少选项
少字段
AI 预填
用户只确认关键点
尽量不让用户重新填写完整表单
```

一句话：

> **确认卡片是纠偏入口，不是让用户重新填表。**

---

# 5. 正式入库流程

只有规则中心 AI 能确认，或者用户确认后的数据，才能正式入库。

## 5.1 入库前条件

一条记录要正式入库，需要满足：

```
已完成基础请求校验
已完成规则中心 AI 清分
必要字段明确
高风险字段不存在歧义
通过规则校验
通过关系校验
通过 Domain Service 安全写入检查
```

---

## 5.2 入库流程

```
规则中心 AI 输出结构化结果
↓
Domain Service 校验
↓
关系校验
↓
安全写入 records
↓
生成 record_id
↓
写入 trace / decision 日志
↓
返回正式记录
```

---

## 5.3 入库后数据状态

入库后的数据默认进入正式业务系统，可以被：

```
事项模块引用
计算中心读取
目标模块间接使用
洞察模块间接引用
前端展示
纠错模块绑定
```

一句话：

> **入库是数据从“待判断”进入“正式系统”的边界。**

---

# 6. 事项模块流程

事项模块不解析自然语言，只承接规则中心 AI 的归属结果。

## 6.1 事项模块负责什么

事项模块负责：

```
事项列表
子事项列表
事项状态
事项下记录展示
事项聚合入口
事项与目标的关联关系
```

---

## 6.2 事项归属流程

```
规则中心 AI 判断记录属于哪个事项/子事项
↓
高置信度 + 规则通过
↓
正式入库时写入 item_id / sub_item_id
↓
事项模块展示该记录
```

如果事项不确定：

```
规则中心 AI 返回事项确认卡片
↓
用户选择事项/子事项
↓
规则校验通过
↓
正式入库
```

---

## 6.3 事项模块不做什么

事项模块不负责：

```
不解析用户原始输入
不自己判断记录归属
不直接调用 AI 改归属
不绕过规则中心修改核心归属
不直接计算目标进度
```

一句话：

> **事项模块是归属结果的承接者，不是归属判断者。**

---

# 7. 计划 / 发生 / 想法 / 总结流程

记录类型由规则中心 AI 在入库前判断。

## 7.1 类型判断流程

```
用户输入
↓
规则中心 AI 判断类型
↓
输出：
- 发生
- 计划
- 想法
- 总结
↓
高置信度则入库
低置信度则返回确认卡片
```

---

## 7.2 示例

```
今天跑步5公里
→ 发生
```

```
明天跑步30分钟
→ 计划
```

```
以后可以试试晨跑
→ 想法
```

```
这周跑步坚持得不错
→ 总结
```

一句话：

> **类型必须在入库前确定，不能靠入库后异步补。**

---

# 8. 时间模块流程

时间是高风险字段，不能乱猜。

## 8.1 时间识别流程

```
规则中心 AI 识别时间表达
↓
规则中心校验时间是否合法
↓
高置信度 → 写入时间字段
↓
低置信度 → 返回时间确认卡片
```

---

## 8.2 时间不确定示例

```
下周找时间跑步
```

这种不能硬写成某一天，应返回确认卡片。

---

## 8.3 时间模块不做什么

时间模块不负责：

```
不自己解析原始输入
不自动把模糊时间写成确定时间
不让前端推断时间
```

一句话：

> **时间字段明确才入库，不明确就确认。**

---

# 9. 金额模块流程

1.6 阶段金额可以先由 records 字段承载，不必拆成独立财务域。

## 9.1 金额识别流程

```
规则中心 AI 识别金额
↓
规则校验金额格式
↓
高置信度 → 写入 cost / currency 等字段
↓
低置信度 → 返回确认卡片
```

---

## 9.2 示例

```
晚上烧烤花了80
→ cost = 80
```

但要注意：

```
金额明确 ≠ 事项归属一定明确
```

如果金额明确但事项不明确，可以返回归属确认卡片。

一句话：

> **金额可以结构化，但不能因为有金额就自动归错事项。**

---

# 10. 计算中心流程

计算中心只计算正式入库且可统计的数据。

## 10.1 计算中心读取范围

计算中心只读取：

```
已正式入库
已通过规则校验
符合统计资格
未取消
未软删除
```

计算中心不读取：

```
未入库数据
待确认数据
确认卡片草稿
AI 临时判断结果
失败解析结果
```

---

## 10.2 计算中心流程

```
记录入库 / 更新 / 纠错
↓
标记计算中心 dirty
↓
计算中心读取正式数据
↓
按请求计算
↓
返回 ViewModel / explain
```

---

## 10.3 计算中心负责什么

计算中心负责：

```
次数统计
时长统计
金额统计
事项近 N 日数据
事项趋势
目标完成值
对比值
平均值
进度值
```

---

## 10.4 计算中心不负责什么

计算中心不负责：

```
不判断自然语言
不判断记录归属
不接受 AI 直接写指标
不让前端自算
不自己生成洞察文案
```

一句话：

> **计算中心只算正式入库且可统计的数据。**

---

# 11. 目标模块流程

目标模块不能被 AI 直接推动。
目标也不直接消费原始记录。

## 11.1 正确关系

目标进度的来源应该是：

```
记录
↓
事项
↓
计算中心聚合事项数据
↓
目标模块提供目标定义
↓
服务端计算目标进度
↓
前端展示
```

---

## 11.2 目标模块流程

```
事项模块面板需要展示目标进度或完成值
↓
目标模块提供目标定义 / 目标值
↓
计算中心提供事项完成值 / 聚合值
↓
服务端完成目标进度计算
↓
返回事项模块前端展示目标进度
```

---

## 11.3 目标模块不做什么

目标模块不负责：

```
不直接接收用户原始输入
不直接读取未清分记录
不让 AI 直接增加目标进度
不让前端自己算目标进度
不把单条记录直接记入目标
```

一句话：

> **目标进度来自服务端计算，核心数据来源于事项聚合和计算中心。**

---

# 12. 洞察模块流程

洞察模块不直接计算数据，只引用计算中心结果。

## 12.1 洞察模块流程

```
入库记录
↓
计算中心生成统计结果
↓
洞察模块提出数据需求
↓
计算中心处理需求
   例如：
   - 某事项近7天平均值
   - 本周进度
   - 上周进度
   - 本月金额
   - 连续完成天数
↓
洞察模块引用结果
↓
生成洞察卡片
```

---

## 12.2 AI 在洞察中的角色

AI 可以负责表达，但不能负责算数据。

正确方式：

```
前端提出需求
↓
洞察模块请求数据
↓
计算中心返回本周进度、上周进度等数字
↓
AI 收到数字
↓
AI 解释成人话：
“你最近运动比上周少了30%”
```

错误方式：

```
AI 直接扫描记录并说：
“你最近运动少了30%”
```

---

## 12.3 洞察解释要求

洞察卡片要能解释：

```
这个洞察基于哪些记录
计算口径是什么
使用了哪个时间范围
对比对象是什么
```

一句话：

> **洞察可以 AI 表达，但数据必须来自计算中心。**

---

# 13. 用户纠错流程

纠错是入库后的安全出口，也是后续规则改进的证据来源。

## 13.1 纠错流程

```
用户发现记录错误
↓
点击纠错
↓
提交修正
↓
规则中心 AI 或规则校验重新处理
↓
通过后更新正式记录
↓
写 corrections
↓
绑定原 decision_id / unit_id / record_id
↓
生成测试用例条目
↓
记录错误趋势
↓
等待后续规则更新
```

---

## 13.2 纠错不等于自动学习

不能：

```
用户纠错一次
↓
系统自动修改永久规则
```

应该是：

```
用户纠错一次
↓
生成证据
↓
沉淀测试用例条目
↓
后续统一整理
```

一句话：

> **纠错是系统学习的证据，不是自动学习本身。**

---

# 14. 测试用例流程

测试用例由用户纠错或系统错误沉淀而来。

## 14.1 测试用例生成流程

```
用户纠错 / 系统错误
↓
生成测试用例条目
↓
记录：
- 原始输入
- 原始判断
- 错误结果
- 用户修正结果
- 关联 trace_id
- 关联 decision_id
- 关联 error_code
↓
进入待处理测试池
↓
后续由软件管理流程处理
```

---

## 14.2 测试用例的作用

测试用例用于：

```
复现错误
验证规则修改
防止回归
形成错误趋势
辅助后续规则中心完善
```

测试用例不等于：

```
自动更新规则
自动创建用户偏好
自动改 prompt
自动修改代码
```

一句话：

> **每次纠错都应该沉淀成可复现的测试证据。**

---

# 15. 规则更新流程

规则更新必须谨慎。

## 15.1 规则更新流程

```
多个相似纠错
↓
错误聚类
↓
形成规则建议
↓
用户 / 开发者确认
↓
更新规则中心
↓
跑测试用例
↓
通过后生效
```

---

## 15.2 禁止行为

不能：

```
一次纠错 → 自动生成永久规则
```

应该是：

```
一次纠错 → 生成证据
多次纠错 → 生成建议
人工确认 → 更新规则
```

一句话：

> **规则更新必须有证据、有确认、有测试。**

---

# 16. 前端流程

前端必须保持薄客户端。

## 16.1 前端负责什么

前端负责：

```
提交用户输入
展示确认卡片
展示入库结果
展示事项列表
展示目标进度
展示洞察卡片
展示纠错入口
展示错误提示
展示 trace/debug 信息
```

---

## 16.2 前端不负责什么

前端不负责：

```
不拆分复合句
不判断事项归属
不判断记录类型
不计算目标进度
不计算洞察数据
不判断置信度是否达标
不决定是否入库
不直接写业务状态
```

一句话：

> **前端是传菜员，不是厨师。**

---

# 17. 系统模块可观测性流程

除了用户输入要编号，系统模块本身也要编号。

## 17.1 模块编号

至少需要：

```
MOD-INPUT        录入模块
MOD-RULES        规则中心
MOD-RECORD       记录模块
MOD-ITEM         事项模块
MOD-TIME         时间处理
MOD-AMOUNT       金额处理
MOD-COMPUTE      计算中心
MOD-GOAL         目标模块
MOD-INSIGHT      洞察模块
MOD-CORRECTION   纠错模块
MOD-EVAL         测试用例模块
MOD-FRONTEND     前端展示模块
```

---

## 17.2 模块错误定位

如果出现问题，要能判断：

```
是规则中心清分错
还是记录入库错
还是事项归属错
还是计算中心算错
还是目标进度组合错
还是洞察引用错
还是前端展示错
```

例如：

```
用户输入没问题
↓
规则中心也清分正确
↓
记录入库正确
↓
但目标进度显示错
```

此时系统应能定位到：

```
MOD-COMPUTE 或 MOD-GOAL 或 MOD-FRONTEND
```

而不是重新扫描全部代码。

一句话：

> **可观测性不仅追踪用户输入，也要追踪系统模块处理。**

# 18. 全系统总流程

最终完整流程可以压缩为：

```
用户输入
↓
生成 trace_id / input_id
↓
规则中心 AI 入库前清分
↓
如果是复合句，生成 unit_id[]
↓
每个 unit 生成 decision_id[]
↓
分流：
  A. 可确认 → Domain 校验 → 正式入库
  B. 不确定 → 返回确认卡片
  C. 无法识别 → 返回澄清卡片
↓
用户确认后再次校验
↓
正式入库
↓
事项模块承接归属结果
↓
计算中心只读取正式入库且可统计的数据
↓
目标模块基于事项聚合和目标定义计算进度
↓
洞察模块向计算中心要数据并组织表达
↓
前端只负责展示
↓
用户纠错
↓
写 corrections
↓
生成测试用例条目
↓
记录错误趋势
↓
多个相似纠错形成规则建议
↓
用户 / 开发者确认
↓
更新规则中心
↓
跑测试用例
↓
通过后生效
```

---

# 19. 当前禁止事项

当前 TETO 1.6 不应该做：

```
不先入库再 AI 增强
不把待确认数据写入正式库
不让目标直接消费记录
不让洞察直接扫描 records 计算
不让前端自算目标和统计
不让 AI 自动修改长期规则
不让一次纠错变成永久规则
不把复合句和单句共用一个编号
不只追踪用户输入而忽略系统模块
```

# 最终一句话定稿

> **TETO 1.6 的核心流程是：用户输入先编号，再由规则中心 AI 在入库前完成拆分、清分、识别和准入判断；明确的数据正式入库，不明确的数据返回确认卡片；入库后由事项承接、计算中心统一计算、目标读取事项聚合、洞察引用计算结果；用户纠错沉淀为 corrections 和测试用例条目，后续通过错误聚类与人工确认再更新规则中心；全过程通过输入编号、判断编号和模块编号实现可回放、可诊断、可验证。**
