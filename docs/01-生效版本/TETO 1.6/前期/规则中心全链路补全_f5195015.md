# TETO 1.6 主链路接线修复计划

## 阶段定位

- **主阶段**: TETO 1.6
- **任务类型**: 主链路接线修复 / 生产级底座补线 / 自查纠错闭环接入
- **目标**: 不新增复杂未来功能，只修复审计报告中已确认的主链路断点

## 严格禁止

1. 不重写整个 QuickInput / Pipeline Runner
2. 不新增复杂 Multi-Agent
3. 不修改 docs / sql 目录已有文件（只能新增）
4. 不删除已有 migration / 不物理删除用户数据
5. 不让前端新增业务判断 / 不让 AI/Agent/Tool 直接写 DB
6. 不绕过 Domain Service / 不把 console.log 当生产日志
7. 不一次改所有未来规划模块
8. 不改变现有 API 响应基本兼容性
9. 不在未确认字段存在前臆造字段

## 最终验收标准

```
用户输入 → 写入：走规则中心
AI 增强 → 回写：走 AI 写边界
用户纠错 → 更新：走规则中心
用户删除 → 删除：走规则中心
decision → 可持久化
trace → 可持久化
diagnose → 可查真实数据
correction → 能标记 trust/goal/insight 需要重算
```

## Task 1: correct API 走 Domain Service（优先级最高）

**文件**: `src/app/api/v2/records/[id]/correct/route.ts`

- 将直接 `supabase.from('records').update()` 替换为 `updateRecordSafely()`
- 保留 corrections 表写入 + regression test case 生成
- 纠错后设置 `review_status='corrected'`
- 若传入 decision_id 则绑定，否则生成 `DEC-CORR-xxx` fallback
- 写入 decision log（类型 `CORRECTION_APPLIED`）

**验收**: correct API 不再裸 update records，走 updateRecordSafely，correction 正常写入

---

## Task 2: DELETE 走 Domain Service

**文件**: `src/lib/domain/record-service.ts`、`src/app/api/v2/records/[id]/route.ts`

- 在 record-service.ts 新增 `deleteRecordSafely()`:
  1. 查询 existingRecord（带 user_id）
  2. 不存在 → blocking error
  3. 检查 lifecycle_status（completed/postponed/cancelled 返回 warning）
  4. 检查 record_links 关联（有则返回 warning）
  5. 调用底层 deleteRecord
  6. 返回 DomainResult
- 修改 DELETE route：`deleteRecord()` → `deleteRecordSafely()`

**验收**: DELETE 不走裸 deleteRecord，经过 Domain Service，有关联记录时返回 warning

---

## Task 3: decision_logs 持久化

**文件**: `src/lib/observability/decision-logger.ts`

- `logDecision()` 新增 DB 写入（同时保留 console log）
- 新增 `persistDecisionLog()` 函数，写入 `decision_logs` 表
- 必须接入节点: parseSemantic、matchItemSmart、enhanceRecord、applyAiEnhancementSafely、correct API、deleteRecordSafely

**验收**: decision_logs 表有真实记录，trends API 不再空返回

---

## Task 4: trace_summaries 最小持久化

**文件**: `src/lib/observability/trace.ts`、`src/app/api/v2/diagnose/route.ts`

- withTrace() 中写入 trace_summaries（trace_id, user_id, path, started_at）
- 请求结束时更新 ended_at + status
- enhanceRecord 继承 POST 的 trace_id
- diagnose API 优先查持久化表，fallback 到内存

**验收**: POST /records 后 trace_summaries 有记录，diagnose API 可查到

---

## Task 5: AI enhance 全部走 applyAiEnhancementSafely

**文件**: `src/lib/ai/enhance-record.ts`

- 确认 enhance-record.ts 中无裸 supabase.from('records').update()
- 歧义分支和普通增强分支都走 applyAiEnhancementSafely
- AI 不覆盖 content/type/date/lifecycle_status
- 设置 review_status='unchecked'（若当前非 confirmed/corrected）
- 客户端 enhanceWithAi 保留但标注 P1 风险

**验收**: enhance-record.ts 无裸 update，AI 回写返回 AiWriteResult

## Task 6: correction 后 mark derived dirty

**文件**: `src/lib/domain/record-service.ts`（或新文件）

- 若无 dirty 表，写 decision_log（metadata 标记 requires_recompute）
- 接入位置: correct API 后、AI enhancement 后、deleteRecordSafely 后

**验收**: correct 后 decision_logs 有 requires_recompute 标记

## Task 7: 链路检查结论输出

代码修改完成后输出简短结论：

1. 用户首次输入是否仍先走 localParse？
2. parseSemantic 是否仍只在异步增强阶段？
3. 用户初始写入是否经过 createRecordSafely？
4. AI 增强回写是否全部经过 applyAiEnhancementSafely？
5. correct 是否经过 updateRecordSafely？
6. DELETE 是否经过 deleteRecordSafely？
7. trace/decision 是否已持久化？
8. diagnose 是否能查到真实数据？
9. 还剩哪些 P1/P2 后续任务？

1. correct API 走 Domain Service
2. DELETE 走 Domain Service
3. decision_logs 持久化
