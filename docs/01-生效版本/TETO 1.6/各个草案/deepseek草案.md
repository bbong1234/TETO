## TETO 1.6 蓝图：工程底座版

### 一、1.6 一句话定位

> **把 1.5 建好的“散装规则与计算中心”升级为“全系统统一引用的、可解释、可追踪、可诊断、可自愈的工程底座”。**

1.5 已经声明了规则中心和计算中心，但它们是**孤岛**——散落的硬编码仍在旁路它们，调用链路不可见，决策过程无法追溯。1.6 就是要让这两个中心真正“活”起来，并加上可观测性神经系统。

---

### 二、1.6 要解决的四大核心痛点

| 你当前的真实困扰                               | 1.6 的回应                                                                 |
| ------------------------------------------------ | ---------------------------------------------------------------------------- |
|  **“很多接口都是错的”**  — 指标在不同地方算出来不一样，不知道信哪个   | **计算中心闭环**：所有统计口径强制走统一入口，值自带解释和来源追溯                         |

### 三、1.6 的五大模块

text

```
Block A：规则中心闭环
Block B：计算中心闭环
Block C：故障可观测性与链路追踪
Block D：决策 ID 追踪与结构化日志
Block E：令牌化设计体系
```

执行顺序严格按 **A → B → C → D → E**，前一个是后一个的基础。

---

### 四、各模块概要

#### Block A：规则中心闭环

**目标**：让规则中心成为全系统唯一的规则入口，消灭散落硬编码。

- **规则编号化**：每条规则有唯一 id（如 `R-CL-001`），方便日志和 AI 引用
- **规则解释化**：每条规则可输出人类可读的“它是什么、为什么这么设、被哪里调用”
- **规则调用统一化**：检查并迁移所有仍硬编码的判断（如 `0.85` 阈值），强制走 `RULES.xxx`
- **规则版本化**：规则中心带版本号，决策日志可记录“当时用的是哪个版本”

**定性验收**：任何一个硬编码阈值都无法躲过规则中心；核心规则可被展开查看解释。

#### Block B：计算中心闭环

**目标**：让所有统计数字都有统一的口径来源和解释能力。

- **指标编号化**：每个指标有唯一 computation_id（如 `C-ACT-001` 活跃度）
- **口径可解释**：点击一个数字，能看到它怎么算的、包含了哪些记录、排除了哪些记录
- **来源可追溯**：后端返回 `included_record_ids` / `excluded_record_ids`，前端可按需展示
- **时间窗口统一**：今天、本周、近7天、近30天的定义全系统一致

**定性验收**：洞察页、目标页、事项页的同一个指标显示完全一致，且能展开查看计算过程。

#### Block C：故障可观测性与链路追踪

**目标**：解决“击鼓传花”问题，让每一次用户操作的完整链路可被追踪、断点可立即定位。

- **链路步骤标准化**：定义从输入到解析、归类、入库、统计刷新等关键步骤的固定枚举
- **trace_id 与 step_id**：每次录入生成唯一 trace_id，每个处理环节追加 step_id
- **断点定位面板**（开发用）：可查看一条 trace 的完整步骤列表，卡在哪里一目了然
- **组件行为编号体系**：组件、典型行为、异常模式都有固定编号，AI 可据此快速诊断

**定性验收**：通过 trace_id 能在 debug 页面看到一条记录的完整旅途；AI 诊断时只需提供 trace_id 和错误 step，无需全量扫描项目。

#### Block D：决策 ID 追踪与结构化日志

**目标**：让系统的每一次关键判断（为什么归到英语？为什么标记低置信度？）都留下决策快照。

- **decision_id 体系**：事项匹配、子项归属、字段回写、目标计入等均生成决策 id
- **决策记录内容**：包含输入快照、候选结果、选中结果、使用规则 id、置信度、版本号
- **用户纠错绑定**：当你手动修正归类后，系统能关联到当初的 decision_id，知道“AI 错在哪”

**定性验收**：可以从一条记录反查到当时的归类决策详情，并看到它后来是否被用户修正。

#### Block E：令牌化设计体系

**目标**：为 2.0 的大量面板（情绪维度、钱包、复盘等）准备统一的设计底座，避免 UI 碎片化。

- **设计令牌（tokens.json）** ：统一颜色、字体、间距、状态色、维度色
- **基础组件库**：Card、Panel、StatBlock、Badge、ExplanationBlock、TraceStep、DecisionBadge
- **面板模板**：DebugPanelTemplate、ExplanationPanelTemplate（2.0 可复用）

**定性验收**：新增一个调试面板或解释面板时，不需重新定义基础样式和布局。

### 五、1.6 的最小可用闭环（MVP）

为了不陷入完美主义，1.6 的最小可用目标定为：

> **任意一条记录的完整录入链路能产生 trace_id，关键决策能产生 decision_id，核心统计数字能展开查看口径解释，debug 页面能定位断点。**  整个系统开始“能诊断自己”了。

**不在 1.6 做**：

- 不新增任何业务功能（情绪维度、钱包、复盘等）
- 不做复杂Agent
- 不重写已有模块，只在现有基础上做闭环化和可观测化

---

### 六、1.6 与后续版本的关系

1.6 是 **纯工程底座版本**，本身不直接产出用户可见的新功能，但它直接决定：

- **2.0 量化维度是否可信**：钱包金额、情绪趋势、投入产出比的统计相同吗？错了能立即被发现吗？
- **3.0 团队协同是否可行**：成员贡献值计算的口径是统一的吗？能追溯吗？
- **AI 协作效率能否质变**：未来开发者或 AI 修 bug 时，是一小时 vs 一分钟的差别。

**没有 1.6，2.0 和 3.0 的地基是湿沙。**
