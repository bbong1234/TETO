直接结论：**如果你的真实目标是 1.6 重构整个项目，那么这个方向是合理的，而且比“只做补丁式可观测性”更符合你现在的问题。** 
你现在遇到的不是单点 bug，而是 **架构脆弱、数据引用不可信、接口边界不稳、AI 无法低成本诊断** 的系统性问题。

所以 1.6 不应该被压缩成“小修小补版本”。
它可以正式定义为：

> **TETO 1.6：工程底座重构版。目标是把 TETO 从“主逻辑能跑的原型系统”，重构为“数据可信、规则统一、计算可解释、链路可追踪、AI 可诊断、后续可产品化扩展”的工程系统。**

你对 TETO 的“巨大 Excel 表格”理解也非常准确。
更客观地说，TETO 的本质是：

> **一个由 AI 辅助写入的结构化人生数据库。用户每天输入自然语言，系统把它转成结构化记录，再通过规则、筛选、统计、引用、标记、归类、计算，生成不同视图。**

现在最大的问题不是“有没有表格”，而是：

> **这个表格里的数据被谁写入、按什么规则写入、被哪个视图引用、引用时有没有过滤错、统计时有没有算错、AI 改动时会不会牵连别处——这些都不够清楚。**

所以 1.6 的重点应该从“加功能”转成：

> **重构数据写入、数据引用、规则调用、计算输出、错误定位这五条底层链路。**

# 二、你现在的问题本质是什么

你说了几个关键点：

1. 主逻辑已经通了；
2. 架构太脆弱；
3. 牵一发动全身；
4. 错误排查耗时；
5. 改完一个错误，又出来第二个；
6. TETO 本质像巨大 Excel；
7. AI 每天帮你写表格；
8. 但很多时候引用的数据是错误的。

这说明 TETO 当前的问题不是“缺少功能”，而是下面这几个系统性问题。

---

## 1. 写入链路不够可信

自然语言输入后，系统可能会做很多事：

```
用户输入
→ 本地解析
→ AI 解析
→ fallback
→ 复合句拆分
→ 字段填充
→ 类型判断
→ 事项归类
→ 子项归属
→ 目标计入
→ 入库
→ 统计刷新
→ 前端展示
```

任何一个环节错，最后表格里的数据都会错。

而现在的问题是：

> 数据错了以后，你很难知道是“写入时错了”，还是“读取时错了”，还是“统计时错了”，还是“展示时错了”。

这就是为什么你感觉改一个 bug，又冒出第二个。

---

## 2. 读取链路不够统一

TETO 像 Excel，核心不是只有“写单元格”，还包括：

- 哪些行被筛选；
- 哪些列被引用；
- 哪些字段参与统计；
- 哪些记录被排除；
- 哪些记录只是计划，不是事实；
- 哪些记录是 AI 推断，不是用户确认；
- 哪些记录已经取消；
- 哪些目标应该计入；
- 哪些指标应该按周算，哪些按月算。

如果不同页面各自写查询逻辑，就会出现：

```
洞察页：本周投入 = 6 小时
目标页：本周投入 = 5.5 小时
事项页：本周投入 = 7 小时
```

这就是“引用的数据错误”。

所以 1.6 必须解决：

> 所有读取和统计必须经过统一的数据访问层 / 查询口径层 / 计算中心。

不能让页面直接乱查数据库。

---

## 3. 接口契约不够硬

你提到“很多接口都是错的”。

这通常不是简单的 API bug，而是接口契约不清：

- 这个接口到底接受什么字段？
- 哪些字段是必填？
- 哪些字段由 AI 写？
- 哪些字段只能用户写？
- 哪些字段不能覆盖？
- 返回值应该包含什么？
- 错误时返回什么？
- 是否携带 trace\_id？
- 是否携带 decision\_id？
- 是否携带 rule\_version？
- 是否携带 computation\_version？

如果没有稳定契约，AI 编程工具每次改代码都可能“看起来能跑”，但悄悄破坏其他页面。

所以 1.6 必须补：

> **接口契约层。**

每个核心 API 都应该有明确的输入、输出、错误码、追踪字段。

---

## 4. AI 写入权限不够显性

TETO 的特殊点是：

> AI 不是旁观者，它会参与写表格。

这非常危险，也非常强大。

所以必须明确：

- AI 可以写哪些字段；
- AI 只能建议哪些字段；
- AI 不能覆盖哪些字段；
- 用户确认后哪些字段才算事实；
- fallback 写入的字段置信度是多少；
- 用户纠错后是否锁定该字段；
- AI 再次增强时能否覆盖用户纠错。

1.5 里已经有一些 `ai-write-policy`、字段所有权策略，但如果你要 1.6 重构，就必须把它提升为核心架构：

> **AI 写入治理层。**

否则 TETO 的“巨大 Excel”会被 AI 越写越乱。

---

DeepSeek 的五大模块是合理的：

```
Block B：计算中心闭环  
Block C：故障可观测性与链路追踪  
Block D：决策 ID 追踪与结构化日志  
Block E：令牌化设计体系
```

如果你确认 1.6 是重构版，而不是小补丁版，那这五个模块都可以保留。

但我建议调整它们的地位。

# 四、我建议补成七大模块，而不是五大模块

DeepSeek 的五大模块缺了两个更底层的东西：

1. **数据可信层**
2. **接口契约层**

所以 TETO 1.6 重构版建议改成七大模块：

```
Block 0：数据可信层重构
Block 1：接口契约层重构
Block 2：规则中心闭环
Block 3：计算中心闭环
Block 4：链路追踪与故障可观测性

```

注意：我加了 Block 0 和 Block 1。

原因是：

> 如果数据写入不可信、接口契约不稳定，那么后面的规则解释、计算解释、trace、decision 都只是在给混乱系统加监控，不能真正解决牵一发动全身的问题。

# 五、TETO 1.6 重构版应该解决的核心问题

我建议把 1.6 的核心问题定义成 6 个。

---

## 问题 1：数据从哪里来，是否可信？

对应模块：**数据可信层**

要解决：

- 记录是用户原文、AI 推断、用户确认、系统计算，还是 fallback 生成？
- 哪些字段是事实？
- 哪些字段是推断？
- 哪些字段被用户纠正过？
- 哪些字段参与统计？
- 哪些字段只是展示？
- 计划记录和事实记录如何区分？
- 取消记录是否参与统计？
- 低置信度记录是否参与自动计算？

你说 TETO 像 Excel，那这里就是：

> 每个单元格都要知道自己是手填的、公式算的、AI 推断的，还是后来被人改过的。

如果这个不做，统计永远会不可信。

---

## 问题 2：接口之间如何不互相污染？

对应模块：**接口契约层**

要解决：

- API 输入输出统一；
- 错误码统一；
- 返回结构统一；
- 每个接口携带 trace\_id；
- 核心写入接口携带 decision\_id；
- 业务错误和系统错误分离；
- AI 编程工具改接口时有边界；
- 前端不能绕过 domain 层直接乱写。

这解决你说的：

> “很多接口都是错的。”

接口不是靠补 bug 变稳的，而是靠契约变稳。

---

## 问题 3：规则是否唯一、可查、可解释？

对应模块：**规则中心闭环**

要解决：

- 所有规则统一从规则中心走；
- 所有关键规则有 rule\_id；
- 所有规则有解释；
- 所有规则有版本；
- 决策日志记录使用过哪些规则；
- 禁止业务模块私自写硬编码阈值。

这解决：

> 同一个判断在不同地方不一致。

---

## 问题 4：计算是否唯一、可追溯、可复现？

对应模块：**计算中心闭环**

要解决：

- 所有统计走计算中心；
- 每个指标有 computation\_id；
- 每个指标说明公式；
- 每个指标说明数据范围；
- 每个指标返回 included/excluded records；
- 时间窗口统一；
- fact / inferred / planned 的计入口径统一。

这解决：

> 引用的数据经常是错的。

---

## 问题 5：出错时能否定位断点？

对应模块：**链路追踪与可观测性**

要解决：

- 每次用户操作有 trace\_id；
- 每个处理环节有 step\_id；
- 每个组件有 component\_id；
- 每个异常有 error\_code；
- Debug 页面能看到链路；
- AI 可以根据 trace 定位问题，不需要扫描全项目。

这解决：

> 改错成本高、token 浪费、排查慢。

---

## 问题 6：系统为什么这么判断？

对应模块：**决策 ID 与结构化日志**

要解决：

- AI 为什么这么归类；
- 为什么写入这个字段；
- 为什么触发 fallback；
- 为什么目标计入；
- 为什么统计值变化；
- 用户纠错后如何记录；
- 未来如何沉淀成规则。

这解决：

> 系统黑箱问题。

# 九、DeepSeek 方案最大的缺口

我认为主要有 6 个缺口。

---

## 缺口 1：没有明确“数据可信分级”

你现在最核心的问题是“引用的数据是错误的”。

这不是光有 computation\_id 能解决的。

必须定义数据可信等级，例如：

```
source_type:
- user_input       用户原文
- ai_inferred      AI 推断
- fallback_parsed  降级解析
- user_confirmed   用户确认
- user_corrected   用户纠错
- system_computed   系统计算
```

以及：

```
confidence_level:
- high
- medium
- low
- unknown
```

还有：

```
stat_eligible:
- true
- false
```

也就是这条记录 / 字段是否允许参与统计。

否则一个低置信度 AI 推断字段被拿去统计，最后就会出现错误数据。

---

## 缺口 2：没有明确“字段级来源”

TETO 像 Excel，但比 Excel 更复杂。

一条 record 里面可能：

- 时间是用户明确说的；
- 金额是 AI 解析的；
- 事项归类是系统猜的；
- 情绪是 fallback 推断的；
- 结果是用户后面改的；
- 目标计入是系统计算的。

所以只记录 record 的来源不够，应该至少考虑：

> 关键字段的来源和决策。

尤其是这些字段：

- `occurred_at`
- `item_id`
- `sub_item_id`
- `type`
- `duration_minutes`
- `cost`
- `metric_value`
- `mood`
- `outcome_type`
- `status`
- `goal_id / progress_source`

不一定 1.6 全部做字段级溯源，但蓝图里必须预留。

---

## 缺口 3：没有明确“读模型 / 写模型分离”

你的问题是“写进去”和“引用出来”都可能错。

所以 1.6 应该考虑：

```
写入模型：负责把自然语言变成结构化记录
读取模型：负责把结构化记录变成页面和统计
```

如果继续让页面直接读 records 表，然后自己 filter / map / reduce，就会继续乱。

需要统一：

- RecordQueryService；
- StatsQueryService；
- GoalProgressQueryService；
- InsightDataService。

不一定名字是这些，但思想是：

> 页面不直接拼口径，页面只请求已定义好的读模型。

---

## 缺口 4：没有明确“接口契约测试”

如果你要交给专业 IDE / AI 编程工具重构，那么必须用测试保护。

否则 AI 重构越多，越容易新 bug。

1.6 至少应该补：

- API contract tests；
- rule tests；
- computation tests；
- trace tests；
- decision log tests；
- record write policy tests；
- regression tests。

王自如说“报错自动写成测试用例”，这个可以先不自动化，但 1.6 应该建立：

> 每修一个核心 bug，就沉淀一个回归测试。

否则“改完一个错误，又出来第二个”的问题不会根治。

---

## 缺口 5：没有明确“AI 编程可维护文档”

你说 AI 不方便修改，这不只是 trace 问题，也是项目结构暴露问题。

需要给 AI 编程工具准备：

- 架构地图；
- 核心链路图；
- 文件职责表；
- 禁止修改区域；
- 改动前检查清单；
- 改动后验证清单；
- 常见错误编号表；
- 接口契约表。

这类文档不一定是用户文档，而是：

> 给 AI IDE 的工程导航文档。

这会直接减少 token 浪费。

---

## 缺口 6：没有定义“重构边界”

如果 1.6 是重构整个项目，必须明确：

- 哪些可以重构；
- 哪些不能动；
- 哪些只能包一层；
- 哪些必须保留兼容；
- 哪些旧接口废弃；
- 哪些数据迁移需要新增 SQL；
- 哪些页面可以暂时坏；
- 哪些主链不能中断。

否则重构容易失控。

## 七大模块

### Block 0：数据可信层

目标：

> 确认每条数据、每个关键字段，来自哪里，是否可信，是否可参与统计。

核心内容：

- 数据来源标记；
- 字段来源标记；
- AI 推断 / 用户确认 / 用户纠错区分；
- 低置信度数据统计策略；
- 计划 / 事实 / 推断 / 取消记录的计入口径；
- stat eligibility，也就是统计资格。

验收：

- 能判断一条记录哪些字段可信；
- 能判断哪些字段可以参与统计；
- 低置信度数据不会无声进入关键统计；
- 用户纠错能提升数据可信状态。

---

### Block 1：接口契约层

目标：

> 所有核心 API 的输入、输出、错误、追踪字段统一，避免 AI 改代码时破坏链路。

核心内容：

- API request schema；
- API response schema；
- error response schema；
- trace\_id 传播；
- decision\_id 返回；
- 类型定义统一；
- 前后端契约测试。

验收：

- 核心接口输入输出有明确 schema；
- 错误结构统一；
- AI 编程工具能根据契约改代码；
- 改接口不会悄悄破坏页面。

---

### Block 2：规则中心闭环

目标：

> 所有规则统一入口、统一编号、统一解释、统一版本。

核心内容：

- RULES\_VERSION；
- rule\_id；
- rule explain；
- 硬编码扫描；
- 规则调用点统一；
- 决策记录 rule\_ids。

验收：

- 关键规则不再散落；
- 至少 10 条关键规则可解释；
- 关键决策能记录用过哪些规则。

---

### Block 3：计算中心闭环

目标：

> 所有统计数字统一口径、可解释、可追溯、可复现。

核心内容：

- COMPUTATION\_VERSION；
- computation\_id；
- metric explain；
- 时间窗口统一；
- 数据过滤统一；
- included/excluded records；
- 页面禁止自算核心指标。

验收：

- 洞察页 / 目标页 / 事项页同一指标一致；
- 关键数字能解释；
- 能反查计入和排除的数据。

---

### Block 4：链路追踪与可观测性

目标：

> 出错时知道球传到哪里断了。

核心内容：

- trace\_id；
- step\_id；
- component\_id；
- operation trace；
- computation trace；
- duration；
- status；
- error\_code；
- debug trace view。

验收：

- 一次录入可看到完整链路；
- 一次计算可看到数据来源和计算过程；

- AI 诊断不需要全量扫描。

---

### Block 5：决策 ID 与结构化日志

# 十一、我建议的执行顺序

DeepSeek 建议：

```
A → B → C → D → E
```

如果你要重构整个项目，我建议改成：

```
0. 项目结构审计
1. 数据可信层
2. 接口契约层
3. 规则中心闭环
4. 计算中心闭环
5. 链路追踪
6. 决策日志
7. 调试 UI + 设计 tokens
8. 回归测试与验收
```

也就是：

## 第一步不是写规则中心，而是做项目审计

因为你已经判断“很多接口都是错的”。
那最先要做的是：

> 找出到底哪些接口错、哪些页面绕过中心、哪些统计重复计算、哪些数据引用不可信。

否则直接重构会变成盲改。

# 十二、1.6 最小可用闭环应该改成这样

DeepSeek 的 MVP 是：

> 任意一条记录的完整录入链路能产生 trace\_id，关键决策能产生 decision\_id，核心统计数字能展开查看口径解释，debug 页面能定位断点。

这个对，但还少了“数据可信”和“接口契约”。

我建议改成：

> **用户输入一条记录后，系统能追踪它从自然语言到结构化入库的完整链路；每个关键字段知道来源、置信度和是否可参与统计；每个关键判断有 decision\_id；每个核心统计数字能解释口径并反查计入 / 排除数据；接口返回结构稳定统一；当出错时能通过 trace\_id、step\_id、error\_code 直接定位断点。**

这个才是真正解决你现在的问题。

# 十五、1.6 的核心验收标准

我建议最终验收不要只看“代码写了什么”，而看这些结果：

## 数据可信验收

1. 一条记录能说明哪些字段来自用户、AI、fallback、系统计算或用户纠错。
2. 低置信度字段不会无声参与关键统计。
3. 用户纠错后能记录纠错来源和原始 decision。

## 接口契约验收

4. 核心 API 有稳定 request / response / error schema。
5. API 错误返回统一携带 error\_code 和 trace\_id。
6. AI 编程工具修改接口时有契约约束。

## 规则中心验收

7. 关键规则有 rule\_id、版本、解释。
8. 核心业务判断不再绕过规则中心。

## 计算中心验收

9. 关键指标有 computation\_id、公式、时间窗口、数据范围。
10. 同一指标在不同页面显示一致。
11. 关键统计能反查 included / excluded records。

## 可观测性验收

12. 一次完整录入有 trace\_id 和 step 列表。
13. 一次核心计算有 computation trace。
14. 出错能定位到 step\_id 和 component\_id。
15. AI 诊断时可以只提供 trace\_id + error\_code，而不是全项目扫描。

## 决策日志验收

16. 事项归类、字段回写、fallback、目标计入有 decision\_id。
17. 决策记录包含输入、候选、输出、置信度、使用规则。
18. 用户纠错能绑定原 decision。

## 调试 UI 验收

19. 有 debug trace 页面或等价入口。
20. 能查看 trace、step、decision、error、rule、computation 关系。

## 工程质量验收

21. 核心链路有回归测试。
22. 每修复一个核心 bug，至少沉淀一个测试用例。
23. 重构后主录入链路、目标计算链路、洞察统计链路可验证通过。
