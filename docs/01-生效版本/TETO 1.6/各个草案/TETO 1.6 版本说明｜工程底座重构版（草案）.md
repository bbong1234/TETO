# 一、TETO 1.6 的阶段定位

TETO 1.6 定位为：

> **工程底座重构版。**

它不是一个新增业务功能版本，也不是继续堆叠洞察、复盘、钱包、情绪、多人协作、Agent 等上层能力的版本。

TETO 1.6 的核心任务是：

> **围绕 TETO 作为“AI 辅助写入的巨大结构化表格”这一系统本质，重构数据写入、数据读取、规则调用、计算输出、接口契约、链路追踪、决策日志和调试展示，使系统从“主逻辑能跑”，升级为“数据可信、接口稳定、规则统一、计算一致、错误可定位、AI 可协作维护”的产品级工程底座。**

---

# 二、为什么需要 TETO 1.6

TETO 1.5 已经完成了重要基础：

- 三层九组记录模型；
- 规则中心；
- 计算中心；
- 目标规则引擎；
- AI 降级模式；
- Domain 不变量体系；
- 数据完整性诊断；
- 基础统计框架。

这些成果说明：

> TETO 的主逻辑已经基本通了。

但 1.5 之后暴露出新的核心问题：

1. **系统架构较脆弱**

   - 改一个地方容易影响多个链路；
   - 一个 bug 修完后，另一个相关 bug 又出现；
   - 说明模块边界、接口契约、数据流向还不够稳定。
2. **错误排查成本高**

   - 很多问题需要反复扫描项目、扫描数据、扫描调用链；
   - AI 编程工具很难快速知道问题发生在哪个环节；
   - token 消耗高，定位效率低。
3. **数据引用不够可信**

   - TETO 本质上像一个巨大的结构化 Excel；
   - 每天通过 AI 把自然语言写入这个表格；
   - 然后系统再筛选、引用、统计、展示这些数据；
   - 但当前可能出现“引用的数据不对”“不同页面口径不一致”“统计结果来源不清”的问题。
4. **规则中心和计算中心还没有真正闭环**

   - 1.5 已经建立了规则中心和计算中心；
   - 但部分业务逻辑可能仍然绕过它们；
   - 规则和计算虽然存在，但还不一定做到全系统唯一入口、可解释、可追踪、可诊断。
5. **AI 协作开发缺少工程导航**

   - 当前 AI 修改项目时，容易需要全量理解；
   - 缺少 trace、decision、error\_code、component\_id 等定位工具；
   - AI 不知道“球传到哪一步断了”，只能大范围猜测。

因此，TETO 1.6 的必要性不是“加新功能”，而是：

> **在继续产品化之前，先修复系统底座的脆弱性。**

> **TETO 1.6 是工程底座重构版：把 1.5 已经跑通的主逻辑，升级为数据可信、接口稳定、规则统一、计算一致、链路可追踪、决策可解释、错误可定位、AI 可维护的产品级底座。**

# 四、TETO 的系统本质

在 1.6 视角下，TETO 不应被简单理解为“记录软件”“目标软件”或“洞察软件”。

TETO 更准确的本质是：

> **一个由 AI 辅助写入的结构化人生数据库。**

也可以更直观地理解为：

> **一个巨大的智能 Excel 表格。**

用户每天通过自然语言输入：

- 做了什么；
- 花了多少钱；
- 花了多少时间；
- 和谁有关；
- 在哪里发生；
- 情绪如何；
- 目标是否推进；
- 结果如何；
- 哪些事情被延迟、完成、取消或变更。

AI 和系统将这些输入写入结构化数据表。
之后，TETO 再根据不同规则与计算口径，把这些数据：

- 筛选出来；
- 标记出来；
- 引用出来；
- 统计出来；
- 对比出来；
- 展示出来；
- 解释出来。

所以 TETO 的核心可靠性不只取决于“能不能录入”，而取决于：

> **写进去的数据是否可信，读出来的数据是否正确，算出来的数字是否一致，展示出来的结果是否可追溯。**

这正是 1.6 要解决的问题。

# 五、TETO 1.6 要解决的核心问题

## 1. 数据可信问题

当前需要回答：

- 一条记录是用户明确输入的，还是 AI 推断的？
- 一个字段是用户确认的，还是 fallback 解析的？
- 一个归类结果是高置信度，还是低置信度猜测？
- 一个字段后来是否被用户纠正过？
- 低置信度字段是否可以参与统计？
- 计划记录、事实记录、取消记录、推断记录是否应该进入同一个计算口径？

1.6 要建立数据可信机制，让系统知道：

> **每条数据从哪里来，是否可信，是否允许参与统计。**

---

## 2. 接口契约问题

当前需要回答：

- 每个核心 API 接受什么输入？
- 返回什么输出？
- 错误时返回什么结构？
- 是否携带 trace\_id？
- 是否携带 decision\_id？
- 是否携带 rule\_version？
- 是否携带 computation\_version？
- 哪些字段可以被 AI 写？
- 哪些字段只能由用户确认？
- 哪些字段不允许被覆盖？

1.6 要建立接口契约层，让 AI 编程工具和系统自身都有稳定边界。

目标是：

> **接口不能靠猜，前后端不能各说各话，AI 修改代码不能破坏隐性契约。**

---

## 3. 规则中心闭环问题

1.5 已经建立规则中心，但 1.6 要进一步确认：

- 是否所有规则都真的从规则中心读取；
- 是否还有散落硬编码；
- 规则是否有编号；
- 规则是否有解释；
- 规则是否有版本；
- 某次判断到底用了哪些规则；
- 规则修改后影响哪些链路。

1.6 的目标是：

> **规则中心从“存在一个 RULES 对象”，升级为“全系统唯一规则入口”。**

---

## 4. 计算中心闭环问题

1.5 已经建立计算中心和统计框架，但 1.6 要进一步确认：

- 洞察页、目标页、事项页是否使用同一计算口径；
- 同一个指标在不同页面是否一致；
- 每个数字是否能解释公式；
- 每个数字是否能说明时间窗口；
- 每个数字是否能反查计入记录和排除记录；
- 前端是否还有自行计算核心指标的情况。

1.6 的目标是：

> **每个关键统计数字都可解释、可追溯、可复现。**

---

## 5. 链路追踪问题

当前最大痛点之一是：

> 出错时不知道断在哪一步。

1.6 要建立“击鼓传花式”的链路追踪机制。

也就是每一次用户操作都可以形成：

```
用户输入
→ 本地解析
→ AI 解析
→ fallback 判断
→ 字段回写
→ 事项归类
→ Domain 校验
→ 数据库写入
→ 目标计入
→ 统计刷新
→ 前端展示
```

每一步都要知道：

- 接到了什么；
- 处理了什么；
- 输出了什么；
- 传给了谁；
- 是否失败；
- 失败在哪里；
- 用了什么规则；
- 生成了什么决策；
- 耗时多久。

1.6 的目标是：

> **系统出错时，不再全项目扫描，而是通过 trace\_id、step\_id、error\_code 直接定位断点。**

---

## 6. 决策追踪问题

TETO 中很多结果来自判断：

- 为什么归类到这个事项？
- 为什么字段被写成这个值？
- 为什么触发 fallback？
- 为什么目标进度发生变化？
- 为什么统计结果排除了某条记录？
- 为什么系统认为这条记录置信度低？

1.6 要建立 decision\_id 体系。

每次关键判断都应该留下决策快照：

- 输入是什么；
- 候选项有哪些；
- 选中了什么；
- 为什么选它；
- 置信度是多少；
- 用了哪些 rule\_id；
- 用了哪些 computation\_id；
- 是否后来被用户纠正。

目标是：

> **系统不只是给结果，还要留下“为什么这样判断”的证据。**

---

## 7. AI 可维护问题

1.6 的重要目标之一是让 AI 编程工具更容易维护 TETO。

不是让 AI 更聪明，而是让项目更适合 AI 理解：

- 有架构地图；
- 有接口契约；
- 有规则编号；
- 有计算编号；
- 有错误编号；
- 有 trace；
- 有 decision；
- 有测试；
- 有文件职责边界；
- 有禁止修改范围。

目标是：

> **AI 不再需要每次全量扫描项目，而是能根据编号、链路和契约快速定位问题。**

# 七、TETO 1.6 的最小可用闭环

# 十、TETO 1.6 与 1.5 的关系

TETO 1.5 是：

> **录入清分与统计口径稳定化阶段。**

它解决的是：

- 记录结构如何分层；
- 规则中心和计算中心如何建立；
- 目标引擎如何规则化；
- AI 不可用时如何降级；
- 数据完整性如何初步诊断。

TETO 1.6 是：

> **工程底座重构阶段。**

它要解决的是：

- 规则中心是否真的被全系统引用；
- 计算中心是否真的统一所有统计；
- 数据写入是否可信；
- 数据读取是否一致；
- 接口契约是否稳定；
- 错误是否可定位；
- 决策是否可追踪；
- AI 是否能高效协作维护。

所以 1.6 不是推翻 1.5，而是：

> **对 1.5 已有主逻辑进行工程级重构、闭环化和产品化前加固。**

# 十一、TETO 1.6 与后续版本的关系

TETO 1.6 是后续产品级开发的地基。

如果没有 1.6：

- 2.0 做情绪、钱包、效率、社交、空间等维度时，统计数据可能不可信；
- 后续做投入产出、复盘、趋势分析时，口径可能不一致；
- 后续做 AI 辅助建议时，AI 可能基于错误数据给出错误判断；
- 后续做多人协同时，数据权限、计算口径、操作追踪会更复杂；
- 后续继续让 AI IDE 改代码时，仍然会高 token、低定位效率。

所以 1.6 的价值不是用户马上看到更多功能，而是：

> **让后续所有功能建立在可信数据、稳定接口、统一规则、统一计算和可追踪链路之上。**

---

# 十二、TETO 1.6 的核心验收方向

TETO 1.6 的完成标准不应是“写了多少代码”，而应是下面这些结果真实成立。

## 1. 数据可信验收

- 一条记录能说明关键字段来源；
- 能区分用户输入、AI 推断、fallback、系统计算、用户纠错；
- 低置信度数据不会无声进入关键统计；
- 用户纠错能提升或改变数据可信状态。

## 2. 接口契约验收

- 核心 API 有稳定 request / response / error schema；
- API 错误返回统一携带 error\_code 和 trace\_id；
- 前后端类型对齐；
- AI IDE 修改接口时有契约约束。

## 3. 规则中心验收

- 关键规则有 rule\_id；
- 关键规则有解释；
- 规则中心有版本号；
- 核心业务判断不再绕过规则中心。

## 4. 计算中心验收

- 关键指标有 computation\_id；
- 关键指标有公式和口径说明；
- 同一指标在不同页面显示一致；
- 关键统计能反查 included / excluded records。

## 5. 可观测性验收

- 一次完整录入有 trace\_id 和 step 列表；
- 一次核心计算有 computation trace；
- 出错能定位到 step\_id 和 component\_id；
- AI 诊断时可以只提供 trace\_id + error\_code，而不是全项目扫描。

## 6. 决策日志验收

- 事项归类、字段回写、fallback、目标计入有 decision\_id；
- 决策记录包含输入、候选、输出、置信度、使用规则；
- 用户纠错能绑定原 decision。

## 7. 调试 UI 验收

- 有 debug trace 页面或等价入口；
- 能查看 trace、step、decision、error、rule、computation 的关系；
- 调试 UI 使用统一基础组件和 tokens。

## 8. 工程质量验收

- 核心链路有回归测试；
- 每修复一个核心 bug，至少沉淀一个测试用例；
- 主录入链路、目标计算链路、洞察统计链路可验证通过。
