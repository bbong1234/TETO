# TETO 1.5 蓝图方案

## 文档使用原则

1. **本文档内容分三层**
   - **已定边界**：当前可作为1.5讨论依据，除非重大理由不改动
   - **当前倾向方案**：方向明确，但实现未锁死，实施时可调整
   - **蓝图方向**：仅保留思路，不进入当前交付范围

2. **1.5默认优先手动明确，而非自动脑补**
   在语义不稳定、统计影响大、归类不确定时，优先允许用户手动明确，而不是强求系统自动推断。

3. **统计与洞察优先依赖结构化数据**
   1.5的统计与洞察底座应优先基于结构化字段、明确归类和可追溯记录，不以自由文本推断直接作为核心统计依据。

4. **AI可以增强，但不能成为单点依赖**
   AI不可用时，系统可以变笨，但不能瘫。

## 一、1.5 总定位【已定边界】

> **1.5 先解决“怎么把真实内容正确录进来并能稳定统计”，不先追求复杂历史建模，也不先追求会思考的高级洞察。**

### 1.5 与 1.4 的关系

1.4 完成了纵深建设（四维事项 + 语义引擎 + 量化目标），但真实使用中暴露出：输入后经常还要手动改、AI清分不够稳、事项归类不够准、历史导入与时间段表达模糊、洞察能展示但不够像"看懂自己"。

1.5的本质不是重构骨架，而是围绕真实使用问题，对1.4的录入、归类、统计与洞察链路做稳定化深化。

### 三条主线

1.5 先正式定为 3 条主线：

1. **录入主线** — 让输入低阻力、能接住、清分稳定
2. **归类主线** — 让记录落到正确的事项结构里
3. **统计总结主线** — 让数据可算、结果可看懂

历史记录降为录入主线下的一个特殊子问题，不单独升为主线。

### 六模块流水线

```
模块1: 输入模块 — 负责“把现实接进来”
  ↓
模块2: 解析与清分模块 — 负责“把原话拆明白”
  ↓
模块3: 事项归类模块 — 负责“把内容放对位置”
  ↓
模块4: 确认与纠错模块 — 负责“拦住错误、承接修正”
  ↓
模块5: 统计量化模块 — 负责“把沉淀数据算出来”
  ↓
模块6: 洞察输出模块 — 负责“把结果讲给用户听”
```

任何一环不稳，后面都会失真。

### 优先级排序

**第一优先**（不顺手用户根本不会用）：
1. 输入模块 — 低阻力、能接住
2. 解析与清分模块 — 别乱分、别乱拆
3. 事项归类模块 — 别乱归类

**第二优先**（决定系统是否可信）：
4. 确认与纠错模块 — 少错、可控、可信

**第三优先**（数据进来后得能算、能看懂）：
5. 统计量化模块
6. 洞察输出模块

### 历史记录降级原则

历史记录不应升成1.5主轴，应降级为：
- 普通录入 = 主入口（包括补录昨天、前天——只要用户能逐条明确表达，都算普通输入）
- 历史导入 = 子入口（是否属于历史导入，不看是不是过去，而看是不是“批量/概括/超出普通逐条录入范围”）
- 历史规律/模糊历史 = 历史导入里的受控特殊情况

1.5的主任务不是研究"历史哲学"，而是让当前输入顺手、清分稳定、事项归类正确、统计和洞察可信。

## 三、解析清分与确认纠错（模块2+模块4）【已定边界+倾向方案】

### 3.1 问题

1.4 的AI解析虽然落地，但用户看不到系统为什么这样判断：
- AI归到某个事项/子项，用户不知道为什么
- 模糊句分流不稳定，用户无法预测系统行为
- 规则不透明、原因不可见、纠错沉淀不足
- 复合句拆分标准不明确
- 模糊输入一律当“模糊”处理，没区分是看不懂、信息不足、还是结构不合理

### 3.2 复合句拆分原则

> **按可独立统计的行为单元拆，而不是单纯按语法拆。**

**要拆开的情况**（一句话里同时出现以下任一种）：
- 不同动作：学了英语，还健身了
- 不同事项：处理工作，又复习英语
- 不同时间段：上午开会，晚上跑步
- 不同统计对象：学了2小时英语，花了100元买资料

**可以不拆的情况**：
- 只是同一行为的补充说明：背了30个单词，感觉状态不错
- 情绪/评价/效果：开了个会，挺烦 / 学了英语1小时，效率一般
- 后面部分属于主记录的附加信息，不单独统计

若一句话虽包含多个片段，但最终只形成同一统计对象的补充说明，则不强制拆分。

### 3.3 模糊输入的3类区分

不是所有“模糊”都该同样处理，必须分成3类：

| 类型 | 例子 | 处理方式 |
|------|------|----------|
| **A. 无法理解** | 表达太碎、缩写太多、缺主语缺动作、语义冲突 | 要求澄清，不直接落地 |
| **B. 可理解但信息不足** | 今天弄了一下 / 搞了会儿那个 / 处理了一些工作 | 优先补关键信息；若不影响后续统计主链，可先收为低精度记录 |
| **C. 不合理** | 一条里塞了太多不相关内容 / 时间明显冲突 / 计划和结果混成一条 | 提示拆分或改写 |

1.5的原则：不是所有问题都叫“模糊”。要区分“看不懂”“信息不够”“结构不合理”。

### 3.4 确认哲学

> **录入优先正确，必要时宁可确认，也不要乱收。但方式应以低打扰、分级确认为主，而不是一律强规范。**

| 风险级别 | 例子 | 处理策略 |
|---------|------|----------|
| **低风险** | "今天学了英语" — 没挂到具体事项也不严重 | 先落地，后续可改 |
| **中风险** | "最近状态不太好" — 落哪类/归哪事项不一定准 | 候选确认，给出选项 |
| **高风险** | "我去年基本都是8:30上班" — 自动展开成大量记录错了代价极大 | 禁止自动处理，必须用户确认 |

如果不分级，系统会走向两个极端：要么什么都追问（输入很烦），要么什么都自动猜（数据很脏）。

高风险不等于复杂，而是错误代价高。

目标：提高录入正确率，但方式应以低打扰、分级确认为主，不能把录入门槛抬得太高。

### 3.5 AI降级韧性

> **AI不可用时，系统可以变笨，但不能瘫。**

1.5很依赖AI解析/清分/归类，但这是外部依赖，必须定义AI不可用时的最低可用模式：

**降级后最低可用标准**：
- 手动录入不断
- 基础字段可手动填写
- 简单时间识别仍可用
- 不复杂的复合句不自动拆也没关系
- 事项允许用户手选

已落地的本地归类规则应尽量继续可用。
- 即使AI挂了，已学习的常见事项映射、词汇归类偏好、常用子项偏好仍应生效
- 这恰好是“韧性”的一部分

**用户感知**：
- 轻提示“当前智能解析不可用，已切换基础模式”
- 不要完全静默（否则用户会觉得“怎么突然变傻了”）
- 也不要惊扰（不影响继续录入）

这是1.5必须项，不该后拖太久。

### 3.6 当前倾向方案

#### A. 规则回显层【倾向方案】

每条AI解析结果附带"判断理由"：

```
AI判断：归到【英语】→ 单词
理由：内容含“背单词”，匹配事项"英语"的子项"单词"
置信度：0.87（高）
```

实现方式：在 `enhance-record.ts` 的AI回写结果中增加 `reasoning` 字段，前端在记录详情中展示。

#### B. 规则透明面板【倾向方案】

在设置页增加"规则总览"面板，展示：
- 当前事项→关键词映射规则（AI学习到的 + 用户手动修正的）
- 子项→关键词映射
- 计划/发生分流规则
- 常见模糊表达→默认解释

#### C. 用户可修正规则【倾向方案】

当AI判断有误时，用户可以：
1. 直接修正归属（已在1.4支持）
2. 修正后系统记住这个映射（新增：写入个性化规则表）
3. 下次类似输入自动应用修正后的规则

### 3.7 数据模型【倾向方案】

可考虑新增 user_rules 表，承接用户修正后沉淀的归类规则。字段设计待结合现有数据结构细化。

方向参考：
```sql
CREATE TABLE user_rules (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  rule_type TEXT NOT NULL,        -- 'item_mapping' | 'sub_item_mapping' | 'type_routing' | 'fuzzy_resolution'
  trigger_pattern TEXT NOT NULL,  -- 触发模式（关键词/表达式）
  target_id UUID,                 -- 目标对象ID（事项/子项）
  target_type TEXT,               -- 'item' | 'sub_item'
  confidence TEXT DEFAULT 'high', -- 'high' | 'medium' | 'low'
  source TEXT DEFAULT 'ai_learned', -- 'ai_learned' | 'user_set' | 'system_default'
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ,
  updated_at TIMESTAMPTZ
);
```

## 四、历史记录承接（录入主线的子问题）【已定边界+倾向方案】

### 4.1 核心判断标准

是否属于历史导入，不看是不是过去，而看是不是“批量/概括/超出普通逐条录入范围”：

| 输入类型 | 判断标准 | 例子 | 处理方式 |
|---------|---------|------|----------|
| **普通输入（含补录）** | 用户能逐条明确表达 | 昨天晚上跑了步 / 前天开了个会 / 上周三去办了签证 | 按普通记录处理，不算历史导入 |
| **清晰历史导入** | 批量明确数据，超出逐条录入的自然范围 | Excel/CSV明细 / 一大批已有数据 | 批量导入，与普通记录同级 |
| **模糊历史导入** | 概括性表达，无法精确定位到每天 | 那段时间基本每天7:40起床 / 去年大多8:30上班 | 受控特殊处理（见4.2） |

### 4.2 底层原则：模糊记忆不能伪装成精确事实

这是1.5必须先锁死的原则：

> **概括性历史输入可以进入系统，也可以在需要时展开为日级条目，但展开后的条目必须被视为由概括性历史输入生成的推断数据，而不是与原始逐日记录完全等价的事实记录。**

"那段时间一直7:40起床"表达的是回忆和概括，不是90条"某日7:40起床"的事实证据。如果展开后和真实逐日记录混在同一个语义层，统计会出现伪精确。

**真正要讨论的不是能不能展开，而是展开后的语义、标记、展示和统计边界。**

四条基础原则：

1. **概括性历史可以进入系统** — 不能因为它不精确就完全不收
2. **概括性历史不能伪装成原始事实** — 这是最重要的原则
3. **如果需要日级展示，可以生成推断条目** — 但必须明确它不是原始记录
4. **统计时要区分是否纳入推断数据** — 严格统计只算原始事实，宽松统计允许纳入

### 4.3 三层历史数据语义（简化版）

| 层级 | 身份 | 1.5处理方式 |
|------|------|-------------|
| **A. 明确历史记录** | 精确事实记录 | 直接导入，与普通记录同级 |
| **B. 概括性规律历史** | 模式描述 | 先收入系统，带"规律"标记，先别做太复杂自动化。1.5先只把它当参考性历史，不当核心事实统计底座 |
| **C. 人生背景类历史** | 身份/环境/阶段 | 1.5先不强求进入记录主链，可优先挂到阶段或背景信息层 |

B类内容如果需要展开，展开后必须标记为推断数据（data_nature='inferred'），不得与原始事实无差别混用。

**为什么这样简化**：1.5的主任务不是研究"历史哲学"，而是让当前输入顺手、清分稳定、归类正确。历史记录不应拖偏主线。

### 4.4 规律型输入的处理【倾向方案】

当前倾向：规律型输入先识别其为概括性历史，再由用户选择是作为规律描述保存，还是进一步细化。不在这章里把交互流程写太死。

规律型输入是1.5需要承接的一类特殊输入，其最终是否作为独立记录类型，还是作为记录的特殊形态/附加语义，当前暂不完全锁死。

当前实现可优先按独立类型试行，但这不是1.5当前必须锁死的产品真理。

### 4.5 数据模型【倾向方案】

若采用规律记录/推断条目方案，则 records 或相关扩展结构需支持时间范围、来源关联和数据性质标记。

方向参考：
```sql
-- records 表可能新增字段（方向参考，非最终结构）
ALTER TABLE records ADD COLUMN IF NOT EXISTS period_start_date DATE;      -- 规律起始日
ALTER TABLE records ADD COLUMN IF NOT EXISTS period_end_date DATE;        -- 规律结束日
ALTER TABLE records ADD COLUMN IF NOT EXISTS period_frequency TEXT;       -- 'daily' | 'weekly' | 'monthly' | 'irregular'
ALTER TABLE records ADD COLUMN IF NOT EXISTS is_period_rule BOOLEAN DEFAULT false; -- 是否为规律记录
ALTER TABLE records ADD COLUMN IF NOT EXISTS period_expanded BOOLEAN DEFAULT false; -- 是否已展开
ALTER TABLE records ADD COLUMN IF NOT EXISTS period_source_id UUID;       -- 推断条目的来源规律记录ID
ALTER TABLE records ADD COLUMN IF NOT EXISTS data_nature TEXT DEFAULT 'fact';
  -- 'fact' = 原始事实记录
  -- 'inferred' = 由概括性规律推断生成的条目
```

规律型输入当前可作为特殊记录形态承接；RECORD_TYPES 是否正式扩展'规律'为第5类，暂不锁死。当前实现可优先按独立类型试行。

规律记录的两种状态：
- **未展开**：一条规律记录，带时间范围
- **已展开**：规律记录标记为 period_expanded=true，展开的逐日条目 data_nature='inferred'，通过 period_source_id 关联回原规律记录

**data_nature 字段的核心作用**：
- 区分原始事实记录和推断生成的条目
- 统计和洞察 API 可以通过此字段区分数据性质

#### 阶段（承接历史事项阶段/身份背景输入）

现有 phases 表已支持（is_historical=true），无需新增字段。

历史事项阶段/身份背景输入直接创建为阶段，不经过记录体系。

### 4.6 展示与统计中的数据性质处理【倾向方案】

若后续进入规律展开方案，需要在展示层明确区分规律描述、原始事实与推断条目。

统计口径需支持区分原始事实与推断数据，默认优先事实口径。是否暴露严格/宽松双模式，可在实施时再定。

## 五、事项归类与工作台化（归类主线）【已定边界+倾向方案】

### 5.1 当前问题

1.4 的事项页偏档案展示，用户看完就走了，没有"在这里做事"的感觉。

### 5.2 事项/子项/记录边界定义

> **事项是主题，子项是路径，记录是落点。**

| 层级 | 定义 | 例子 |
|------|------|------|
| **事项** | 有独立持续周期、可长期推进、可单独统计的主题 | 英语、健身、工作 |
| **子项/行动线** | 事项内部相对稳定的推进方向 | 英语→单词/听读/语法 |
| **记录** | 一次具体发生、计划、想法或总结 | 背了30个单词、学了1小时英语 |

**什么只是记录**：“打印一张纸”太小，不配叫事项，只是一条记录。

**什么算事项**：长期持续投入的主题。如果事项太碎（打印纸、发消息、去楼下买水都变事项），统计没意义；如果事项太粗（全塞进“工作”“学习”“生活”），统计也没意义。

### 5.3 阶段 vs 执行视图

> **阶段 ≠ 执行视图。阶段是“描述一段时间”，执行视图是“帮助现在推进”。**

| 概念 | 本质 | 例子 |
|------|------|------|
| **阶段** | 时间切片 | 备考冲刺期 / 求职期 / 停滞期 |
| **执行视图** | 当前推进面板 | 现在要做什么 / 卡在哪 / 哪条行动线最近没动 |

不要混成一个页面逻辑。

### 5.4 非事项数据

> **允许长期存在，不强制归入事项。**

现实里很多东西本来就不是事项：买东西、路上琐事、心情波动、零散支出、临时社交。这些可以存在，统计时单独看“非事项区”。

如果某类非事项内容长期高频出现，以后再考虑是否升格成事项。

### 5.5 事项页方向：执行+回看并存【倾向方案】

事项页应从"偏档案展示"升级为"执行+回看并存"的结构：

- **执行区**：帮助当前推进（具体呈现形态待定）
- **档案区**：帮助回看积累

暂不锁死：是否一定为行动线看板形式、是否一定有今日焦点、是否一期就做计划流转。

### 5.6 事项模板【蓝图方向】

模板系统作为后续可能方向保留观察，1.5前期不作为重点。

理由：模板不是当前最卡的问题，它会增加新的对象维护成本，且建立在对事项结构足够稳定的前提上。

若后续进入，方向参考：
```sql
CREATE TABLE item_templates (
  id UUID PRIMARY KEY,
  user_id UUID,
  name TEXT NOT NULL,
  description TEXT,
  default_sub_items JSONB,
  default_goals JSONB,
  default_phases JSONB,
  category TEXT,
  is_system BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ,
  updated_at TIMESTAMPTZ
);
```

## 六、统计量化与洞察输出（统计总结主线）【已定边界+倾向方案】

### 6.1 当前问题

1.4 洞察页有8个图表组件，但只展示统计数据，用户需要自己解读。

### 6.2 统计4条主轴

> 1.5的量化不要铺太开，先只围绕这4条：**做了什么、离目标多远、时间花在哪、有没有效果。**

| 统计主轴 | 看什么 | 例子 |
|---------|--------|------|
| **行动 vs 目标** | 我做了什么 / 和目标是否一致 / 离目标还有多远 | 英语背了30个单词，目标50个，还差20个 |
| **时间 vs 计划** | 我在指定时间内有没有推进 / 是否按预期节奏完成 / 是否延期 | 本周计划每天学2小时，实际平均1.5小时 |
| **投入 vs 效果** | 我花了多少时间精力 / 最后有没有产生结果 / 这个结果是否有效 | 英语投入10小时/周，雅思分数从5.5到6.0 |
| **近期时间分布** | 最近花最多时间在哪里 / 哪些投入高频但低效 / 哪些事项长期占用注意力 | 最近7天50%时间在工作，20%在英语 |

### 6.3 洞察原则

> **1.5洞察先做“事实性总结”，不做“建议型教练”。**

可以说：
- 最近7天你主要时间花在英语和工作
- 你对雅思投入增加了，但阶段目标尚未完成
- 本周高频出现的行动线是背单词与阅读
- 计划中的某事项连续5天没有推进

先不要说：
- 你应该这样做
- 你这个方法不对
- 你下一步必须干嘛

### 6.4 洞察必须有对比视角

> **洞察不是看数据，是看变化。没有对比的洞察是平的。**

1.5必须有“可对比”的时间洞察，分两层：

**第一层：固定对比（1.5必做）**
- 本周 vs 上周
- 本月 vs 上月
- 最近30天 vs 前30天

**第二层：锚点型复盘（1.5可先保留入口，不追求自动化）**
- 四级备考前后
- 换工作前后
- 某目标启动前后

### 6.5 洞察必须可解释、可追溯

> **任何重要洞察，都应该能追溯到时间范围、统计口径和原始记录来源。如果答不上来，洞察就不可信。**

这是硬原则：
- 每条洞察必须说明：时间范围、对比对象、核心统计依据
- 自然语言洞察应自带依据，例如不要只说“你最近英语投入下降了30%”，而应说“最近30天英语相关记录累计12小时，上一个30天为17小时，因此投入下降约30%”
- 每条洞察有数据来源链接（可点击查看原始记录）
- 这样一句话里就自带范围、指标、对比基线，可信度会高很多

### 6.5a 洞察默认展示层级【已定边界】

> **先全局总览，再事项下钻，最后进入目标/阶段/锚点等特定视角。1.5不追求一开始就做多入口并列首页。**

三层：

**第一层：全局总览**
- 最近时间主要花在哪
- 和上期相比变化在哪
- 哪些事项最活跃
- 哪些计划明显滞后

**第二层：事项视角**
- 某个事项最近投入如何
- 子项/行动线是否推进
- 和目标关系如何

**第三层：目标/阶段/锚点视角**
- 这个目标推进得怎样
- 这个阶段前后有什么变化
- 这个锚点附近发生了什么

为什么这样定：1.5不是BI工具，不是让用户一上来自己挑无数维度。它首先要提供一个默认就有价值的入口，而“先全局，再下钻”是最稳的。

### 6.5b 统计口径必须统一定义【已定边界】

> **1.5的统计与洞察必须建立统一的“指标口径定义层”，所有关键指标都应有明确公式、时间范围和纳入条件。**

若统计口径没有统一定义层，后续洞察会越来越飘——这个页面这么算、那个图表那么算、自然语言洞察又是另一套，然后“可解释性”就崩了。

当前口径定义层不需要复杂系统，只要先做到：
- 指标名称
- 计算方式
- 时间范围
- 数据纳入范围
- 是否包含推断数据
- 是否按记录数/时长/结果值计算

1.5应优先统一的5个核心口径：

| 指标 | 需要明确什么 |
|------|-------------|
| **活跃度** | 按记录数？投入时长？最近更新时间？ |
| **投入** | 按时长？按次数？按记录条数？ |
| **停滞** | 多久没记录算停滞？是否区分事项优先级？ |
| **计划达成率** | 按完成比例？按按时完成比例？ |
| **效果** | 按结果记录衡量？按目标结果字段衡量？ |

### 6.6 洞察必须分层：结构化事实 + LLM表达【已定边界+倾向方案】

> **LLM只负责表达层，不直接充当统计结论生成底座。先有结构化洞察结论，再让LLM负责表达。**

**第一层：规则化事实总结（不依赖LLM即可成立）**
- 最近30天英语12小时，上期17小时，下降29%
- 本周工作占比48%，高于上周12%

**第二层：自然语言润色（LLM负责把事实总结写得更顺）**

如果文档里不写清这个边界，后面很容易做成“让AI直接总结”，风险较高。

#### A. 现有图表保留增强

- 事项画像增加偏差指标（目标vs实际）
- 时间分布增加趋势对比
- 跨事项对比增加投入-结果散点图

#### B. 自然语言润色模块【倾向方案】

新增 `src/lib/ai/generate-insights.ts`：

调用 LLM，基于第一层结构化事实总结做自然语言润色，不直接从原始数据自由发挥：

```typescript
interface NLInsight {
  period: string;              // '最近7天' | '最近30天'
  summary: string;             // 总体概要
  highlights: string[];        // 亮点
  warnings: string[];          // 警示
  item_insights: {             // 事项级洞察
    item_id: string;
    item_title: string;
    observation: string;       // “英语近7天日均投入下降30%”
    comparison?: string;       // “与上月同期相比投入减半”
  }[];
  generated_at: string;
}
```

#### C. 展示风格

- 像事实报告，不像聊天
- 每条洞察有数据来源链接（可点击查看原始记录）
- 不做建议，只做总结和观察
- 低频生成（每次打开洞察页时按需生成，或每日缓存一次）

#### D. 外部基准对接【蓝图方向】

保留为蓝图方向，当前不占正文实现权重。

#### E. 计划建议化【蓝图方向】

后续增量方向，1.5当前不作为主交付。

#### F. 数据导出

> **1.5最低要求是手动导出可用。**

这类系统的数据价值很高，不是普通临时数据，而是长期人生记录。

1.5先保证：
- 支持手动导出
- 导出格式优先可读、可迁移、可还原（至少别做成完全封闭黑箱）
- 定期提醒导出

## 七、个性化规则学习（确认纠错模块的延伸）【已定边界+倾向方案+蓝图方向】

### 7.1 学习什么——1.5先只做被动学习

> **1.5先只锁死：被动学习（用户修正时记住）。主动学习后放。**

| 学习类型 | 例子 | 1.5处理 |
|---------|------|----------|
| **事项归属偏好** | “背单词”→英语 | 被动学习，必做 |
| **子项/行动线归属偏好** | “听力”→英语.听读 | 被动学习，必做 |
| **常见词汇与归属的映射** | “跑步”→健身 | 被动学习，必做 |

先不要学作息基线、默认计划类型等——这些后面再加。

### 7.2 学习机制

1. **被动学习**：用户修正AI判断时，自动写入user_rules（1.5必做）
2. **主动学习**：系统定期分析记录并提议规则（后移，1.5不做）
3. **不学习**：不自动修改已确认规则，只建议

### 7.3 规则管理——轻量但不是黑箱

> **1.5要有轻量规则管理，但不要重。**

最轻量的就够：
- 看系统记住了哪些常见归类偏好
- 能删除明显学错的偏好
- 能重置个性化学习

不用复杂页面，但不能完全黑箱。

### 7.4 user_preferences【蓝图方向】

user_preferences 表当前没有明确闭环，先不要急着立表。避免为未来猜想提前建结构。

若后续进入，方向参考：

## 八、记录类型与承接能力【已定边界+倾向方案】

### 8.1 核心策略：少类型、强字段、可挂维度

> **不靠无限加"第5、第6、第7种记录类型"，而是靠记录类型 + 字段 + 维度。**

**第一层：核心记录类型保持少**

先仍然保留这4类为主：
- 发生
- 计划
- 想法
- 总结

这4类先不要轻易再扩。因为类型一多，用户反而更难录。

新增"规律"类型当前为倾向方案，暂不锁死为正式第5类。当前实现可优先按独立类型试行。

**第二层：用通用字段承接真实差异**

很多真实内容不是“类型不同”，而是字段不同：

| 通用字段 | 承接什么 |
|---------|----------|
| 时间 | 发生时间 |
| 时长 | 投入时长 |
| 事项 | 归属事项 |
| 子项/行动线 | 事项内方向 |
| 行为 | 做了什么 |
| 对象 | 作用于什么 |
| 结果 | 产出/成果 |
| 状态 | 进行中/已完成等 |
| 情绪 | 感受/状态 |
| 地点 | 在哪 |
| 金额 | 花费/收入 |
| 数量 | 多少个/次 |
| 来源 | 从哪来 |
| 备注 | 补充说明 |

例如：买咖啡、跑步、开会、焦虑、学英语、交房租、拿到雅思分数——本质上不一定要新类型，很多只是字段组合不同。

**第三层：用“附加维度”承接特殊内容**

某些内容不是靠记录类型区分，而是靠附加维度：
- 是否和事项有关
- 是否和目标有关
- 是否是结果类
- 是否是外部评估值
- 是否是状态类记录
- 是否是消费类记录

### 8.2 想法/笔记增强【倾向方案】

不做"笔记系统"，增强"想法"类型记录的承接能力：
- 想法记录可后续转计划（一键升级为计划类记录）
- 想法记录可后续转事项（一键创建新事项）
- 想法记录应支持后续转计划、转事项，并可在事项语境下被回看。具体UI形态待定。

### 未确定的问题

按优先级分为A/B/C三类：

#### A类：

**9.1 纠错行为是否结构化记录**

“用户修改过”和“系统为什么错”是两件事。如果要让规则真正进化，不仅要看最终结果，还要看：
- AI原来怎么判的
- 用户改成了什么
- 改的是类型、事项、子项、时间，还是文本本身
- 同类错误重复了几次
- 哪些修改只是临时修补，哪些才值得沉淀成规则

这直接影响规则学习、错误分析和后续优化。

#### B类：

**9.4 外部评估体系在系统中的归属**

多邻国等级、雅思分数、某考试分数——这些在TETO里属于什么？
- 一条记录？某事项的结果？某目标的外部验证值？某阶段的成果节点？洞察中的对照基准？
- 当前不急着拍板，但在1.5实施中观察

**9.5 “下一步行动”由谁给出**

事项工作台中的"下一步"，有两种可能来源：
- 用户显式标记（准确可控但需多一步维护）
- 系统根据记录/计划/停滞自动推断（省事但容易误判）

1.5倾向方案：混合模式。用户可标记，系统也给建议，两者并列展示。

**9.6 事项活跃度怎么定义**

“哪个事项最近在推进 / 哪个事项停了”——活跃到底怎么算？记录次数？投入时长？最近一次更新时间？是否接近目标？
没定的话，后面很多洞察会飘。

**9.7 目标本身如何定义“有效果”**

“完成”不等于“有效”：学了20小时不代表进步明显，背了很多单词不代表成绩提升。
效果是按结果记录衡量，还是按目标结果字段衡量？这块没定，投入vs效果会很空。

**9.8 非事项时间消耗如何算“有用没用”**

休息可能有用、社交可能有用、发呆也未必没用。系统不能太武断。
更稳的表达：是否直接服务目标 / 是否产出明确结果 / 是否属于恢复性时间 / 是否属于消耗性时间。

#### C类：

- 系统自动识别阶段变化
- 复杂锚点智能识别
- 全自动历史概括展开
- 高级建议型洞察

## 十二、验收标准

1.5 的完成标准不是"功能写了"，而是以下链路真实可走通。分A/B两层：

### A. 必验收主链

**链路1：录入顺手且正确**

- 输入"背了30个单词" → 正确清分为发生类 → 归到英语→单词 → 显示判断理由 → 用户可修正
- 输入"学了英语还健身了" → 拆成2条记录 → 分别归类
- 概括性历史输入能够被识别并以非事实方式承接

**链路2：归类稳定可修正**

- AI归到某事项 → 显示判断理由 → 用户可修正 → 修正后系统记住
- 非事项数据存在但不强制归类

**链路3：统计可信**

- 统计默认基于原始事实记录
- 若统计中包含推断数据，应明确标注“含N条推断数据”
- 4条统计主轴可查询

**链路4：洞察可信**

- 打开洞察页 → 看到图表+对比视角 → 下方有事实性总结 → 每条洞察自带时间范围+对比对象+统计依据 → 可追溯数据来源
- 不做“你应该...”式建议

**链路5：AI降级有韧性**

- AI不可用时 → 系统切换基础模式 → 录入不断 → 本地规则仍生效 → 轻提示用户当前模式

### B. 增量验收

**链路6：1.4遗留全清**

- 语义卡片UI、规则兜底、记录卡片增强、计划提醒全部可用
- chains表已删除、items.goal_id已移除

**链路7：规则可管理**

- 看到系统学到的归类偏好 → 能删除明显学错的 → 能重置个性化学习

**链路8：数据可导出**

- 支持手动导出 → 导出格式可读、可迁移
