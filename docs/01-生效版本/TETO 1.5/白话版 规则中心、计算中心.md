# 白话版 规则中心、计算中心

**你可以把 TETO 现在要补的这套东西，理解成“给系统加一个总检查站 + 一个统一算法中心”。**

以前系统像这样：

```
每个门都能进货
每个员工都能自己记账
每个部门都自己算报表
AI 也能偷偷改账本
所以一改一个地方，别的地方就炸
```

现在我们要改成：

```
所有数据进来，先过总检查站
所有修改数据，也先过总检查站
AI 想改数据，也要先过总检查站
所有统计报表，都去统一算法中心算
```

这样才不会一修就炸，也不会每个页面数字不一样。

---

# 一、TETO 现在到底在干什么？

TETO 的核心其实很简单：

```
用户输入一条记录
系统把它变成结构化数据
然后挂到事项、子项、阶段、目标上
最后再拿这些数据做统计和洞察
```

比如你输入：

```
今天背了英语单词 30 分钟
```

系统要判断：

```
这是“发生”还是“计划”？
属于哪个事项？英语？
属于哪个子项？单词？
属于哪个阶段？备考？
是哪一天发生的？
有没有时长？
要不要算进目标进度？
要不要算进最近 7 天投入？
```

所以看起来只是记一条记录，其实背后牵扯很多东西。

---

# 二、为什么现在会“修一个 bug 引出另一个 bug”？

因为现在系统很可能是这种状态：

```
创建记录的地方，有一套规则
编辑记录的地方，有一套规则
完成记录的地方，有一套规则
推迟记录的地方，有一套规则
取消记录的地方，有一套规则
批量创建的地方，有一套规则
AI 自动增强的地方，又有一套规则
洞察统计的地方，又自己猜一套规则
```

也就是说：

```
每个入口都在自己判断数据对不对
```

结果就是：

你修了 A 入口：

```
创建记录不再出错
```

但是 B 入口还在乱写：

```
AI 回写又把字段改错
```

或者 C 入口还在乱写：

```
批量创建又塞了一批脏数据
```

所以你会感觉：

```
怎么修不完？
怎么修一个又冒一个？
```

根因不是你修得不好，而是：

> **系统没有一个统一的规则中心。**

---

# 三、什么是“规则中心”？用大白话说

规则中心就是：

> **系统的数据保安。**

所有数据进数据库前，它先检查：

```
这条记录合不合法？
这个子项是不是属于这个事项？
这个阶段是不是属于这个事项？
这个目标有没有挂错？
这个 AI 推断有没有来源？
这个取消的记录是不是不该进统计？
这个未确认的记录是不是不该进洞察？
```

如果明显错：

```
不让写进去
```

如果有点可疑：

```
允许写，但提醒
```

如果能写但不该统计：

```
允许存在，但标记“别算进严格统计”
```

---

# 四、举个最简单的例子

假设你的事项是：

```
英语
```

英语下面有子项：

```
背单词
听力
口语
```

另一个事项是：

```
健身
```

健身下面有子项：

```
跑步
力量训练
```

现在一条记录写成：

```
item_id = 英语
sub_item_id = 跑步
```

这就是错的。

因为：

```
跑步属于健身，不属于英语
```

以前如果某个入口没检查，就可能写进去。  
以后规则中心会拦住：

```
不允许：sub_item_id 不属于 item_id
```

这就避免后面统计炸掉。

---

# 五、为什么简单筛选会变复杂？

你可能觉得：

```
不就是筛一下英语最近 7 天记录吗？
```

但系统实际要判断很多东西。

比如“英语最近 7 天投入”要不要包含这些？

```
取消的计划？
AI 猜出来但你没确认的记录？
周期规律？
历史推断？
没有时长的记录？
只是一个想法？
只是一个总结？
计划了但没做？
今天录入但其实是上周发生的？
```

这些都不一样。

比如：

```
今天补录：上周三背单词 30 分钟
```

它的：

```
录入时间 = 今天
发生时间 = 上周三
```

那统计最近 7 天投入时，应该按哪个时间？

应该按：

```
发生时间
```

不是录入时间。

所以“简单筛选”复杂，是因为 TETO 不是普通备忘录，它要做真实生活数据统计。

---

# 六、什么是“统计口径层”？用大白话说

统计口径层就是：

> **系统的统一算法中心。**

它规定：

```
什么叫活跃？
什么叫投入？
什么叫完成？
什么叫有效行动？
什么叫停滞？
最近 7 天按哪个日期算？
取消的算不算？
AI 未确认的算不算？
规律记录算不算？
```

以后所有页面都不能自己乱算。

以前可能是：

```
事项页：英语本周投入 3 小时
洞察页：英语本周投入 4 小时
目标页：英语本周投入 2.5 小时
AI 总结：你本周英语学习 5 小时
```

因为它们各算各的。

以后要变成：

```
所有地方都问统一算法中心
```

所以数字会一致。

---

# 七、现在这套方案分几层？

你可以不用记 P1/P2/P3。  
我用大白话说，分成 5 件事。

---

## 第 1 件事：先守住最主要的入口

也就是：

```
新建记录
编辑记录
```

这两个最常用。

现在 Qoder 的方案就是先做这个。

它会做到：

```
新建记录前检查
编辑记录前检查
发现明显错误就不让保存
发现可疑问题就提醒
旧数据可以体检
```

这一步能先止血。

---

## 第 2 件事：把其他入口也接进来

除了新建和编辑，还有很多入口也会改记录：

```
完成
推迟
取消
批量创建
批量删除
关联记录
```

这些以后也必须过规则中心。

否则会出现：

```
新建记录很安全
但批量创建又绕过规则中心
```

那问题还会回来。

---

## 第 3 件事：管住 AI

AI 很有用，但 AI 不能直接改账本。

以后逻辑应该是：

```
AI 先提出建议
规则中心检查
不违法才写入
用户确认过的字段 AI 不能乱改
AI 猜的东西要标记“未确认”
AI 推断出来的东西要有来源
```

比如你手动把一条记录归到“英语/单词”，AI 不能过一会儿又给你改成“英语/听力”。

这就是 AI 回写边界。

---

## 第 4 件事：统一统计算法

这一步才是真正解决“为什么筛选复杂”。

以后系统要统一规定：

```
时间投入怎么算
事项活跃度怎么算
目标进度怎么算
计划完成率怎么算
停滞怎么算
最近 7 天趋势怎么算
```

比如：

```
取消的记录不算
AI 未确认的不算核心洞察
规律记录不直接当事实
推断记录默认不算严格统计
按发生时间算，不按录入时间算
```

这样所有页面的数字才会一致。

---

## 第 5 件事：处理多步操作的半成功问题

有些操作不是改一张表，而是改很多东西。

比如删除一个事项，可能要同时处理：

```
事项
子项
阶段
记录
目标
关联关系
```

如果做到一半失败，就会留下脏数据。

这就像：

```
你拆房子，墙拆了，电线没断，水管还连着
```

以后这些操作最好做成：

```
要么全成功
要么全失败
```

这就是事务化。

但这个需要更谨慎，可能以后要配合 SQL/RPC，不是现在第一步就做。

---

# 八、现在 Qoder 的方案处在哪一步？

现在 Qoder 的方案只是在做：

```
第 1 件事：守住新建记录和编辑记录
再加一个旧数据体检
```

它可以解决：

```
新建记录乱挂子项
编辑记录乱挂阶段
非法枚举写入
旧数据哪里脏看不见
```

但它还不能完全解决：

```
AI 偷偷改错
批量创建绕过规则
完成/取消/推迟状态乱
洞察统计各算各的
删除事项半成功
```

所以这一步是对的，但只是第一步。

---

# 九、那它到底能不能解决问题 1？

**能解决一大半，但要继续接其他入口。**

因为问题 1 是“修一个 bug 引出另一个 bug”。

只守住新建/编辑，会让最常见的问题减少很多。

但如果这些入口还没管：

```
完成
推迟
取消
批量
AI 回写
删除事项
```

那它们还可能制造新 bug。

所以完整解决问题 1 的路径是：

```
先管新建/编辑
再管完成/推迟/取消/批量
再管 AI 回写
再管删除/移动这类多步操作
```

---

# 十、那它到底能不能解决问题 3？

**现在这一步还不能。**

问题 3 是统计和筛选复杂。

这个要靠：

```
统一统计算法中心
```

也就是以后所有页面都去同一个地方算：

```
英语投入
最近 7 天
目标进度
事项活跃度
停滞天数
计划完成率
```

现在 Qoder 方案只是提前标记：

```
这条取消记录不该进统计
这条未确认记录不该进核心洞察
这条周期规律不该直接当事实
```

但如果页面还没用统一算法，这些标记不会自动生效。

所以：

```
当前方案 = 给统计可信化打地基
统计口径层 = 真正解决问题 3
```

---

# 十一、用一个超简单比喻总结

TETO 现在像一个公司：

```
销售自己记账
仓库自己记账
财务自己记账
老板还让 AI 自动改账
每个人算法不一样
所以报表对不上
```

现在要做的是：

## 第一步：设立入库质检

```
所有新数据进来先检查
错的不能入库
可疑的贴标签
```

对应现在 Qoder 方案。

---

## 第二步：所有部门都必须走质检

```
不只是新建记录
完成、取消、批量、AI 都要走质检
```

对应解决问题 1。

---

## 第三步：设立统一财务报表规则

```
所有人算收入、成本、利润，都按同一套算法
```

对应解决问题 3。

---

## 第四步：大操作必须有事务

```
不能钱扣了，货没发
要么都成功，要么都失败
```

对应多步操作稳定化。

---

# 十二、你现在应该怎么理解整体系统？

最终 TETO 应该这样运行：

```
用户输入一句话
  ↓
系统/AI 解析成记录
  ↓
规则中心检查这条记录是否合法
  ↓
合法才写入数据库
  ↓
AI 如果要补充信息，也必须再过规则中心
  ↓
用户修改/完成/取消/批量操作，也必须过规则中心
  ↓
旧数据可以用体检工具查问题
  ↓
所有统计和洞察都去统一算法中心拿结果
  ↓
页面展示一致、AI 总结可信、目标进度可信
```

---

# 十三、最重要的一句话

你只需要记住这句：

> **规则中心负责“不让数据乱”，统计口径层负责“不让数字乱”。**

所以：

```
解决问题 1：靠规则中心管住所有写入入口
解决问题 3：靠统计口径层统一所有计算方法
```

当前 Qoder 的方案只是：

```
规则中心的第一版
```

还需要继续补：

```
其他写入入口
AI 回写
统计口径层
多步事务
```

才算真正解决。
