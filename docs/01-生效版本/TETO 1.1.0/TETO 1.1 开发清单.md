## 三、当前阶段总目标

TETO 1.1 当前要做的是：

> **从固定结构记录工具，升级为可配置任务 + 可关联项目 + 可计算目标完成度的个人管理系统。**

本阶段重点包括：

1. 建立 DIY 任务系统
2. 建立目标值系统
3. 建立基于目标值的完成度系统
4. 建立任务与项目的关联关系
5. 支持项目创建子项目
6. 增加 Excel 导入能力
7. 增加页面区块拖拽排序能力
8. 完成正式域名上线

## 五、当前开发总原则

1. 一次只推进一个明确任务块
2. 先做底层结构，再做界面增强
3. 先做核心数据模型，再做导入、拖拽、上线
4. 不并行开发多个大系统
5. 所有完成标准以“真实可验证”为准
6. 涉及数据库结构变更，必须生成 SQL 文件
7. 涉及 SQL 时，必须提醒去 Supabase SQL Editor 执行
8. 未完成 SQL 执行与真实验证前，不算任务完成
9. 已完成任务不得整体重做，只能补缺口
10. 所有核心设计优先满足单用户稳定可用，不提前扩张到多人系统
11. 若某项看似需要重做，必须先判断是否只是缺文档、缺验证、缺补充，而不是整项重来

## 六、P0：当前最优先任务

### 1. 统一数据结构设计

当前问题：

- 1.1 新功能都依赖新的统一数据结构
- 任务、目标、完成度、项目、子项目、导入之间尚无统一模型
- 如果不先定数据结构，后面页面、导入、拖拽都会反复返工

本轮目标：

- 明确 1.1 的核心实体和关系
- 明确哪些表需要新增、哪些表需要扩展
- 明确任务、项目、目标值、完成度之间的关系
- 明确子项目层级结构
- 明确 Excel 导入的落点
- 明确哪些地方需要 SQL 变更

至少要输出：

1. 任务实体设计
2. 项目实体设计
3. 子项目层级设计
4. 目标值实体设计
5. 完成度计算来源设计
6. Excel 导入目标表设计
7. 用户布局偏好存储设计（只定义，不先实现）
8. 任务记录规则设计
9. 统计周期与时间边界设计
10. 停用 / 归档 / 删除策略设计
11. 项目完成度汇总规则设计
12. 导入模板与导入冲突处理设计
13. 用户数据隔离设计

### 数据结构设计中必须明确的规则

#### 1）任务记录规则

- 任务按类型区分，不混为一类
- 至少支持：

  - 布尔完成型（完成 / 未完成）
  - 次数型
  - 数值累计型
  - 时长型
  - 金额型（如后续确有需要）
- 同一用户、同一任务、同一日期，只保留一条日汇总记录
- 用户可当天多次完成，但系统最终保存该日最终结果
- 允许快捷输入
- 允许手动输入
- 允许补录历史日期
- 不采用同任务同日多条碎片记录模式，避免识别与汇总混乱

#### 2）统计周期与时间边界

- 支持：日 / 周 / 月 / 年 / 自定义
- 每周从周一开始
- 默认时区为中国时区
- 一条记录归属哪一天，以用户手动指定日期为准

#### 3）停用 / 归档 / 删除策略

- 任务支持停用
- 项目支持归档
- 停用 / 归档后默认不再进入常用创建与选择流程
- 历史数据必须保留
- 默认不鼓励物理删除
- 有历史记录的数据不应直接硬删除

#### 4）项目完成度汇总规则

- 第一版中，一个任务只允许归属一个项目
- 项目完成度默认按纳入统计任务的平均完成度计算
- 无目标值任务默认不参与项目完成度
- 暂不引入复杂权重机制

#### 5）子项目层级限制

- 第一版只支持一级子项目
- 即：主项目 + 一级子项目
- 暂不支持无限层级树结构

#### 6）导入规则

- 导入第一版以固定模板为主
- 字段映射只作为有限补充，不做万能自由映射
- 导入前必须预览
- 遇到重复 / 冲突数据时，先提示，不自动覆盖，不自动智能合并
- 用户确认后再写入
- 必须提供导入失败回执

#### 7）数据归属与用户隔离

- 所有核心表都必须明确归属于某个用户
- 必须能在 Supabase 中清晰区分不同用户的数据
- 核心表默认应具备 user_id 归属逻辑

### 验收标准

- 输出一份《TETO 1.1 数据结构设计》
- 明确每个核心实体的字段建议
- 明确实体之间关系
- 明确第一批数据库改动范围
- 明确哪些内容先做、哪些后做
- 明确哪些内容本轮只定义、不立即实现

## 七、P1：核心系统开发顺序

### 1. DIY 任务系统

目标：

- 支持用户自定义任务
- 任务可定义名称、类型、单位、记录方式
- 任务可决定是否参与统计、项目、完成度计算
- 支持两种任务视图：

  - 表单视图
  - 表格视图（类似 TET 的表格录入方式）

说明：

- 表单视图适合单条编辑、字段较完整的录入
- 表格视图适合高频、紧凑、批量式录入
- 两种视图应服务于同一套底层数据结构
- 不允许为表格视图单独做一套脱离主系统的数据模型
- 任务应支持基础搜索、筛选、排序
- 搜索 / 筛选 / 排序属于基础可用性能力，但优先级低于核心结构闭环

验收标准：

- 可以创建任务
- 可以编辑任务
- 可以读取任务
- 页面可正常回显
- Supabase 中可看到真实记录
- 任务页面支持表单视图与表格视图切换
- 两种视图下的数据读写结果一致
- 表格视图至少支持基础输入、编辑、保存、回显
- 支持基础搜索 / 筛选 / 排序
- 任务停用后仍保留历史数据

---

### 2. 目标值系统

目标：

- 为任务设置目标值
- 支持不同目标周期或目标类型
- 目标值与任务绑定

说明：

- 目标值至少支持日 / 周 / 月 / 年 / 自定义周期
- 无目标值任务应明确显示为“无目标”
- 目标值修改后，必须保证后续展示逻辑可解释、可验证

验收标准：

- 可创建目标值
- 可关联到任务
- 可读取与修改
- 页面能正确展示
- Supabase 中可看到真实记录
- 无目标值状态展示正常

---

### 3. 基于目标值的完成度系统

目标：

- 基于实际记录与目标值计算完成度
- 支持任务级完成度展示
- 为项目级完成度汇总打基础

说明：

- 完成度默认不封顶 100%
- 应支持展示：

  - 单日完成度
  - 周完成度
  - 月完成度
  - 年完成度
  - 总完成度
- 无目标值时显示“无目标”
- 无记录默认按 0 处理
- 第一版统一按 0 处理，不区分“未填”和“主动填 0”
- 0 是合法输入
- 项目汇总时，无目标值任务不参与分母计算

验收标准：

- 页面可展示完成比例
- 数据变化后完成度会更新
- 空值 / 0 值 / 无目标值时处理正常
- 页面展示与数据库、计算逻辑一致
- 超额完成可正确显示大于 100% 的结果
- 多周期完成度展示正常

---

### 4. 项目与任务关联

目标：

- 任务可以挂到项目
- 项目可以汇总关联任务贡献
- 为后续项目完成度做基础

说明：

- 第一版中，一个任务只允许归属一个项目
- 项目完成度默认按纳入统计任务的平均完成度计算
- 无目标值任务默认不参与项目完成度

验收标准：

- 可以建立任务与项目关系
- 项目页面可看到关联任务
- 删除 / 变更关联关系后可正确更新
- Supabase 中关系数据真实存在
- 项目完成度规则可验证

---

### 5. 子项目系统

目标：

- 项目支持子项目
- 子项目可归属主项目
- 主项目可汇总子项目进度

说明：

- 第一版只支持一级子项目
- 暂不支持无限层级树

验收标准：

- 能创建主项目 / 子项目
- 能建立层级关系
- 页面能正确展示层级
- 数据真实存储
- 汇总逻辑可验证

---

## 八、P2：增强能力

### 1. Excel 导入功能

目标：

- 支持 Excel / 表格数据导入
- 第一版优先采用固定模板导入
- 字段映射只作为有限补充，不做万能导入器
- 先限定导入范围，不做任意格式自动兼容
- 先支持任务数据 / 项目日志 / Daily Record 历史数据中的一部分

验收标准：

- 可以上传文件
- 可以做基础解析
- 可以预览导入结果
- 可以识别模板问题或字段问题
- 可以确认写入
- 导入冲突时会提示
- 不自动覆盖、不自动智能合并
- Supabase 中可看到真实导入记录
- 导入失败时有错误提示
- 能显示失败行、失败原因、成功条数、失败条数

---

### 2. 页面区块拖拽排序

目标：

- 允许部分页面卡片 / 区块拖拽排序
- 支持基础布局保存
- 与现有折叠能力兼容

说明：

- 第一版只支持固定页面中的区块顺序调整
- 不做自由网格布局系统
- 不做跨页面拖放
- 不做低代码式自由排版

验收标准：

- 页面区块可拖拽调整顺序
- 刷新后布局保留
- 不影响页面主流程
- 布局数据真实保存或可靠持久化

---

### 3. 正式域名上线

目标：

- 完成正式环境部署
- 绑定正式域名
- 验证线上访问链路

上线前至少检查：

- 环境变量是否正确
- Supabase URL / key 是否正确
- 登录回调域名是否正确
- HTTPS 是否正常
- 线上数据库是否连接正确环境
- 主要页面是否可正常访问
- 线上数据读写是否正常

验收标准：

- 域名可访问
- HTTPS 正常
- 登录与主要数据读写正常
- Supabase 线上环境连接无误
- 关键页面可正常使用

## 十二、执行要求

### 1. 单任务执行

不要并行推进多个大块。

### 2. 数据库变更规则

凡涉及数据库结构新增或修改，必须同时满足：

1. 生成 SQL 文件
2. SQL 落到项目根目录 `sql/` 文件夹
3. 文件命名清晰、递增、可读
4. 明确提醒去 Supabase SQL Editor 执行
5. SQL 未执行前不算完成
6. 执行后必须做真实验证
7. 未在 Supabase 中看到真实结构变化，不算完成

### 3. 重复劳动拦截

若某项任务已完成，只能补缺失环节，不能整项重做。

### 4. 验收优先于推进

每完成一项，先验收，再进入下一项。

### 5. 用户数据隔离要求

- 所有核心数据必须能明确区分属于哪个用户
- 不允许出现用户数据归属混乱
- 数据结构与页面读写逻辑都必须服从单用户隔离原则

### 6. 存储与展示单位分离

- 时长类数据可展示为小时，底层优先存分钟
- 金额类数据可展示为元，底层优先存分
- 展示层再做格式化处理
- 底层优先采用稳定、可计算的单位存储

### 7. 历史数据保护

一旦任务已有历史记录，其核心类型、统计方式、单位等关键字段不得随意修改；若确需修改，必须受限制并保证历史统计不失真。

---

## 十三、验收原则

开发完成不以“代码已提交”为准，而以“真实可验证”为准：

- 页面能打开
- 数据能真实保存
- 数据能读取
- 数据能编辑
- 页面能正确回显
- Supabase 中能看到对应记录
- 页面展示结果与数据库一致
- 空值 / 0 值 / 无数据情况正常
- 补录数据后统计结果正确
- 不同周期完成度结果可验证

---

## 十四、当前拍板结论

TETO 1.1 当前只做：

> **任务系统 + 目标值系统 + 完成度系统 + 项目关联/子项目 + Excel 导入 + 页面区块拖拽排序 + 正式域名上线**

不做：

> **多人协作 / 企业权限 / 第二大脑 / 财务模块 / AI 自动解析 / 通用平台化扩展 / 万能导入器 / 移动端 App**
