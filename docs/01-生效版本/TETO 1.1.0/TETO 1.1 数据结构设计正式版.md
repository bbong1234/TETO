# 《TETO 1.1 数据结构设计》正式版

# TETO 1.1 数据结构设计

## 一、文档定位

本文件用于定义 TETO 1.1 阶段的核心数据结构。

本文件负责：

- 明确 1.1 的核心实体
- 明确实体之间的关系
- 明确各实体关键字段建议
- 明确第一批数据库改动范围
- 明确哪些结构先做，哪些后做
- 为后续 SQL、页面设计、接口联调提供统一依据

本文件不负责：

- 替代《TETO AI 协作总则（总控版）》
- 替代《TETO 1.1 开发规则》
- 替代《TETO 1.1 开发清单》
- 直接替代页面结构设计
- 直接替代 SQL 文件

---

## 二、设计目标

TETO 1.1 的数据结构需要支持以下能力：

1. DIY 任务系统
2. 目标值系统
3. 基于目标值的完成度系统
4. 项目与任务关联
5. 一级子项目
6. Excel 导入
7. 页面区块拖拽排序
8. 单用户数据隔离
9. 后续可验证的 Supabase 真实落地

---

## 三、设计总原则

1. 所有核心数据必须归属于某个用户
2. 同一用户、同一任务、同一日期，只保留一条日汇总记录
3. 先保证结构稳定，再考虑界面灵活性
4. 第一版优先满足单用户可用，不扩展到多人协作
5. 尽量避免无限层级、无限映射、无限通用化
6. 所有统计以真实可验证为准
7. 数据结构优先支持“任务 / 记录 / 目标 / 完成度 / 项目”主闭环
8. 底层存储优先使用稳定、易计算的字段设计

---

## 四、核心实体总览

TETO 1.1 第一批核心实体建议为：

1. users（用户，沿用认证系统）
2. task_definitions（任务定义表）
3. task_daily_records（任务日记录表）
4. task_targets（任务目标表）
5. projects（项目表）
6. task_project_links（任务与项目关联表，可简化为 task 上 project_id）
7. import_batches（导入批次表）
8. import_rows（导入明细表，可后做）
9. user_layout_preferences（用户布局偏好表）

说明：

- 若第一版要尽量简单，`task_project_links`​ 可以先不单独建表，直接在 `task_definitions`​ 中保留 `project_id`
- 若后续要支持更复杂关系，再升级为独立关联表
- 由于当前已拍板“一个任务只归属一个项目”，**第一版建议直接在任务表中存** **​`project_id`​**

## 5.1 task_definitions（任务定义表）

### 用途

用于存储用户自定义任务本身。

一个任务定义的是：

- 这是什么任务
- 用什么方式记录
- 单位是什么
- 是否参与统计
- 是否参与完成度计算
- 是否挂到某个项目
- 当前是否启用

### 建议字段

- ​`id`
- ​`user_id`
- ​`name`
- ​`code`（可选，内部标识）
- ​`description`（可选）
- ​`task_type`
- ​`record_mode`
- ​`unit_name`
- ​`display_unit`
- ​`storage_unit`
- ​`value_type`
- ​`allow_quick_input`
- ​`allow_manual_input`
- ​`allow_backfill`
- ​`include_in_stats`
- ​`include_in_completion`
- ​`project_id`（可为空）
- ​`sort_order`
- ​`status`
- ​`archived_at`（可为空）
- ​`created_at`
- ​`updated_at`

### 字段解释建议

#### `task_type`

任务类型，建议枚举：

- ​`boolean`
- ​`count`
- ​`number`
- ​`duration`
- ​`amount`

说明：

- ​`boolean`：完成 / 未完成
- ​`count`：次数型，如运动 3 次
- ​`number`：普通数值型，如背单词 15 个
- ​`duration`：时长型，展示小时，底层可存分钟
- ​`amount`：金额型，展示元，底层可存分

#### `record_mode`

记录模式，建议先保留但不要过度复杂：

- ​`form`
- ​`table`
- ​`both`

说明：

- 页面层可以决定主视图
- 底层数据结构不因视图切换而变化

#### `unit_name`

任务单位名称，如：

- 次
- 个
- 分钟
- 元
- 公里

#### `display_unit`

展示单位，如：

- 小时
- 元
- 次
- 个

#### `storage_unit`

底层存储单位，如：

- minute
- cent
- count
- number

#### `value_type`

值类型，建议保留：

- ​`integer`
- ​`decimal`
- ​`boolean`

#### `status`

任务状态：

- ​`active`
- ​`inactive`

说明：

- 第一版支持停用
- 不建议直接硬删除已有历史记录的任务

### 约束建议

- 同一用户下任务名称可重复与否，建议第一版先允许重复，但前端提示尽量避免
- ​`user_id + id` 必须明确归属
- ​`project_id` 若有值，必须归属同一用户的项目

### 验收意义

该表建成后，应支持：

- 创建任务
- 编辑任务
- 停用任务
- 读取任务
- 挂到项目
- 支持任务列表展示、搜索、筛选、排序

## 5.2 task_daily_records（任务日记录表）

### 用途

用于存储任务的“每日最终汇总值”。

### 核心规则

> 同一用户、同一任务、同一日期，只保留一条记录。

这张表不是碎片操作日志表，而是：

> **按日聚合后的最终记录表。**

用户可以一天内多次完成某任务，但系统最终保存的是这一天的最终结果。

### 建议字段

- ​`id`
- ​`user_id`
- ​`task_id`
- ​`record_date`
- ​`value_number`（可为空）
- ​`value_boolean`（可为空）
- ​`note`（可为空）
- ​`source_type`
- ​`created_at`
- ​`updated_at`

### 字段解释建议

#### `record_date`

归属日期。

规则：

- 以用户手动指定日期为准
- 默认按中国时区理解与展示

#### `value_number`

数值类任务最终值，适用于：

- count
- number
- duration
- amount

#### `value_boolean`

布尔类任务最终值，适用于：

- boolean

#### `source_type`

记录来源，建议枚举：

- ​`manual`
- ​`quick_input`
- ​`import`

说明：

- 便于后续排查记录来源
- 不属于当前必须页面功能，但建议结构先留

### 唯一约束建议

- ​`user_id + task_id + record_date` 唯一

这样能保证：

- 同一天同任务不会出现多条最终记录
- 后续 Excel 导入或页面保存时可做覆盖前检查 / 冲突判断

### 关于 0 值规则

第一版正式规则：

- 无记录按 0 处理
- 0 是合法输入
- 第一版不区分“未填”和“主动填 0”

说明：

- 计算时统一按 0 处理
- 页面层如后续需要再扩展“未填写”状态，不影响第一版结构

## 5.3 task_targets（任务目标表）

### 用途

用于定义任务在不同周期下的目标值。

### 需要支持的目标周期

- 日
- 周
- 月
- 年
- 自定义

### 建议字段

- ​`id`
- ​`user_id`
- ​`task_id`
- ​`target_period_type`
- ​`target_value`
- ​`start_date`（自定义周期时必需，可为空）
- ​`end_date`（自定义周期时必需，可为空）
- ​`is_active`
- ​`created_at`
- ​`updated_at`

### 字段解释建议

#### `target_period_type`

建议枚举：

- ​`daily`
- ​`weekly`
- ​`monthly`
- ​`yearly`
- ​`custom`

#### `target_value`

目标值。

例如：

- 背单词：20
- 运动次数：3
- 学习时长：120（底层可存分钟）
- boolean 任务可约定为 1

#### `is_active`

是否启用。

说明：

- 后续如有目标历史版本再扩展
- 第一版先用“当前生效目标”思路即可

### 规则建议

- 同一任务在同一周期类型下，第一版只保留一条当前生效目标
- 若要改目标值，优先更新当前目标，而不是保留复杂版本历史
- 自定义周期必须有 `start_date`​ 和 `end_date`

### 无目标规则

- 若任务没有目标值，则页面显示“无目标”
- 无目标任务不参与项目完成度汇总

## 5.4 projects（项目表）

### 用途

用于存储项目与一级子项目。

### 第一版层级规则

- 只支持一级子项目
- 即：

  - 主项目
  - 一级子项目
- 不支持无限树结构

### 建议字段

- ​`id`
- ​`user_id`
- ​`name`
- ​`description`（可为空）
- ​`parent_project_id`（可为空）
- ​`status`
- ​`sort_order`
- ​`created_at`
- ​`updated_at`
- ​`archived_at`（可为空）

### 字段解释建议

#### `parent_project_id`

- 为空：主项目
- 有值：子项目

规则：

- 子项目的 `parent_project_id` 指向同一用户下的主项目
- 第一版限制：子项目不能再挂子项目

#### `status`

建议枚举：

- ​`active`
- ​`archived`

### 规则建议

- 任务只允许归属一个项目
- 项目可归档
- 归档后不默认参与新建流程，但历史数据保留

## 六、实体关系设计

### 1）用户 与 任务

- 一个用户有多个任务
- 一个任务只属于一个用户

### 2）用户 与 任务日记录

- 一个用户有多条任务日记录
- 一条任务日记录只属于一个用户

### 3）任务 与 任务日记录

- 一个任务有多条每日记录
- 一条每日记录只属于一个任务

### 4）任务 与 目标

- 一个任务可有多个周期目标
- 同一周期类型下，第一版建议只保留一条当前生效目标

### 5）用户 与 项目

- 一个用户有多个项目
- 一个项目只属于一个用户

### 6）项目 与 子项目

- 一个主项目有多个一级子项目
- 一个子项目只属于一个主项目
- 第一版不支持更深层级

### 7）项目 与 任务

- 一个项目可关联多个任务
- 一个任务第一版只归属一个项目

### 8）用户 与 布局偏好

- 一个用户可有多个页面布局偏好
- 每个页面建议只保留一条当前布局记录

### 9）用户 与 导入批次

- 一个用户可有多个导入批次
- 每个导入批次只属于一个用户

---

## 七、完成度计算来源设计

### 1. 任务完成度计算基础

完成度来自：

- 实际记录值：来自 `task_daily_records`
- 目标值：来自 `task_targets`

### 2. 完成度基础公式

#### 数值类任务

完成度 = 实际值 / 目标值

例如：

- 目标 10
- 实际 15
- 完成度 = 150%

#### 布尔型任务

可约定：

- 完成 = 1 / 1 = 100%
- 未完成 = 0 / 1 = 0%

### 3. 封顶规则

第一版规则：

> **完成度不封顶 100%。**

### 4. 展示维度

需支持：

- 单日完成度
- 周完成度
- 月完成度
- 年完成度
- 总完成度

### 5. 无目标规则

- 无目标值时显示“无目标”
- 无目标任务不参与项目完成度汇总

### 6. 无记录规则

- 无记录默认按 0 处理
- 第一版统一按 0 处理，不区分“未填”和“主动填 0”

---

## 八、项目完成度汇总规则设计

### 第一版拍板规则

1. 一个任务只允许归属一个项目
2. 项目完成度默认按纳入统计任务的平均完成度计算
3. 无目标值任务不参与项目完成度
4. 暂不引入复杂权重机制
5. 子项目完成度汇总先与项目完成度采用同类口径
6. 若项目下没有可参与计算的任务，则显示“无可统计任务”或“无目标”

### 为什么先不用权重

因为 1.1 当前重点是先跑通主闭环：

- 任务
- 记录
- 目标
- 完成度
- 项目关联

复杂权重会明显增加：

- 字段复杂度
- 页面输入复杂度
- 汇总理解成本

所以暂缓。

---

## 九、记录规则补充设计

### 1. 同日单记录原则

同一用户、同一任务、同一日期，只保留一条日汇总记录。

### 2. 支持补录

用户可以手动指定历史日期录入。

### 3. 支持快捷输入与手动输入

底层都写入同一张 `task_daily_records` 表。

### 4. 不保留碎片化多次记录

第一版不做“上午一条、下午一条、晚上又一条”的明细日志模式。  
若用户当天分多次完成，应由前端或输入逻辑合并成该日最终值。

---

## 十、导入设计补充

### 1. 导入优先采用固定模板

原因：

- 更稳定
- 更容易验证
- 更适合第一版上线

### 2. 有限字段映射

允许少量字段映射修正，但不做任意格式万能兼容。

### 3. 冲突处理

若导入数据命中：

- 同一用户
- 同一任务
- 同一日期

应先提示冲突，而不是自动覆盖。

### 4. 失败回执

至少要能看到：

- 哪一行失败
- 为什么失败
- 成功多少条
- 失败多少条

---

## 十一、单位存储设计

### 原则

> 展示单位与底层存储单位分离。

### 建议

#### 时长类

- 展示：小时
- 存储：分钟

#### 金额类

- 展示：元
- 存储：分

#### 普通数值类

- 按实际最稳定格式存储
- 若涉及小数，明确精度规则

### 意义

- 避免浮点误差
- 便于计算
- 便于统计和排序

---

## 十二、状态与删除策略

### 任务

- 支持 `active / inactive`
- 不建议对有历史记录的任务做硬删除

### 项目

- 支持 `active / archived`
- 归档后历史数据保留

### 目标

- 通过 `is_active` 管理当前是否生效

### 删除原则

- 第一版以停用 / 归档替代硬删除为主
- 有历史记录的数据默认不硬删

---

## 十三、第一批数据库改动范围

第一批建议新增或确认以下结构：

1. 任务定义表 `task_definitions`
2. 任务日记录表 `task_daily_records`
3. 任务目标表 `task_targets`
4. 项目表 `projects`
5. 用户布局偏好表 `user_layout_preferences`
6. 导入批次表 `import_batches`

可后做：

7. 导入明细表 `import_rows`

说明：

- 若现有系统已有部分表，应优先判断是“扩展旧表”还是“新增新表”
- 不应在未对照现有表结构前直接粗暴重建

---

## 十四、建议索引与约束方向

### 建议唯一约束

1. ​`task_daily_records(user_id, task_id, record_date)` 唯一
2. ​`user_layout_preferences(user_id, page_key)` 唯一
3. 任务、项目、目标都必须归属于 `user_id`

### 建议索引方向

1. ​`task_definitions.user_id`
2. ​`task_definitions.project_id`
3. ​`task_daily_records.user_id`
4. ​`task_daily_records.task_id`
5. ​`task_daily_records.record_date`
6. ​`task_targets.user_id`
7. ​`task_targets.task_id`
8. ​`projects.user_id`
9. ​`projects.parent_project_id`

---

## 十五、当前不在本轮展开的结构

本轮先不展开以下复杂结构：

1. 多人协作权限结构
2. 企业组织结构
3. 无限层级项目树
4. 高级任务权重系统
5. 自动化规则引擎
6. AI 自动解析数据表
7. 万能导入映射引擎
8. 移动端专用同步结构
9. 碎片化操作日志系统

---

## 十六、当前拍板结论

TETO 1.1 的统一数据结构，第一版应围绕以下主闭环建立：

> **任务定义 → 每日记录 → 目标值 → 完成度 → 项目归属 → 子项目汇总 → 导入补充 → 布局偏好**

其中最关键的结构原则是：

1. 所有数据按 `user_id` 隔离
2. 同任务同日期只保留一条日汇总记录
3. 目标支持日 / 周 / 月 / 年 / 自定义
4. 完成度不封顶 100%
5. 无目标显示“无目标”
6. 无记录按 0 处理
7. 一个任务只归属一个项目
8. 子项目只支持一级
9. 导入以固定模板为主
10. 布局只做固定页面区块排序
